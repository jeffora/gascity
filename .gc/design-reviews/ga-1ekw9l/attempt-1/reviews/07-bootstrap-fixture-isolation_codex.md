# Hiroshi Tanabe - Codex

**Verdict:** block

**Top strengths:**
- The plan correctly identifies the current bootstrap hazards: `internal/hooks/hooks.go` imports `internal/bootstrap/packs/core`, `internal/bootstrap/bootstrap.go` still embeds `packs/**`, and tests override `BootstrapPacks` with `AssetDir: "packs/core"` (`implementation-plan.md` lines 46-50).
- The desired end state is much sharper than the current system: no production import of `internal/bootstrap/packs/core`, no production `//go:embed packs/**` carrying Core, deletion or `bootstrap.EmptyFS` replacement for `bootstrapAssets`/`embeddedBootstrapPacks`/`BootstrapPacks`, package deletion as compile-time proof, and path-literal scanners for `packs/core`, `AssetDir`, `Subpath`, constructed paths, docs, examples, fixtures, and generated files (`implementation-plan.md` lines 467-474).
- The fixture requirements are concrete enough to prevent common drift: production bootstrap becomes a non-nil empty FS, tests assert `Stat`, `WalkDir`, and `ReadFile`, and fixture guards reject copied production Core directories such as `formulas/`, `orders/`, `overlay/`, `skills/`, and `assets/prompts/` (`implementation-plan.md` lines 476-487).

**Critical risks:**
- [Blocker] The implementation plan contradicts the requirements artifact on Core's canonical source. Requirements still say `internal/bootstrap/packs/core` is the sole canonical Core authority and AC2 verification target (`requirements.md` lines 30, 55, 76, 85, and 103), while the implementation plan moves Core to `internal/packs/core` and deletes/removes `internal/bootstrap/packs/core` as an asset source (`implementation-plan.md` lines 17-18, 82-87, and 467-471). This is directly in the bootstrap lane: a downstream AC2 validator can either enforce the old bootstrap path and fail the design, or implementers can keep bootstrap Core reachable to satisfy stale requirements. Reconcile the requirements before approval or explicitly make the implementation plan's first prerequisite a requirements update that changes the canonical source and verification rows.
- [Blocker] `GC_BOOTSTRAP=skip` containment is described, but not structurally proven. The current production bootstrap path returns early when the environment variable is set, and test defaults set it broadly in `cmd/gc/main_test.go`. The plan says that, "if retained," skip may only skip empty bootstrap fixture materialization and must not skip `internal/systempacks` materialization or validation (`implementation-plan.md` lines 489-493), but it does not say how a production command with `GC_BOOTSTRAP=skip` is prevented from bypassing required-system-pack setup. This needs a concrete design: remove the production env bypass, make it test-only by construction, or add a production negative test proving `GC_BOOTSTRAP=skip` cannot suppress Core materialization, file-set integrity, participation validation, retired-source classification, or doctor cleanup.
- [Major] The hidden dependency inventory is broad but not anchored to a named closure artifact in the bootstrap section. The plan says to audit doctor checks, command tests, testscript defaults, prompt tests, config bundled-import tests, precompact hook tests, packlint, generated docs, hook references, and helper paths (`implementation-plan.md` lines 462-465), but the implementation path does not state which checked artifact owns those rows, how it is generated, or what fails before package deletion. Current targeted search shows dependencies in hooks docs, doctor cache checks, materialization tests, compose bootstrap-managed names, prompt tests, and testscript defaults. Bind this inventory to `source-consumer-closure.yaml` or a dedicated bootstrap-dependency inventory with owner, replacement, allowed-test-fixture classification, and a validator that runs before deleting the old package.

**Missing evidence:**
- No support artifacts are currently checked under `plans/core-gastown-pack-migration/support/`; only `requirements.md` and `implementation-plan.md` exist.
- No sample `bootstrap.EmptyFS` contract, fixture guard, old-bootstrap-path scanner allowlist, or `GC_BOOTSTRAP=skip` negative production command test exists yet.
- No generated closure row demonstrates how current references such as `internal/doctor/implicit_import_cache_check.go`, `internal/materialize/skills_test.go`, `internal/config/compose.go`, `cmd/gc/prompt_test.go`, and `cmd/gc/main_test.go` will be migrated or classified.

**Required changes:**
- Update the requirements artifact or the implementation plan so both agree on the post-migration Core source authority. If `internal/packs/core` is the chosen end state, AC2 and its examples must stop requiring `internal/bootstrap/packs/core` as canonical runtime authority.
- Specify the structural `GC_BOOTSTRAP=skip` containment mechanism and add at least one production command test with that environment variable set, proving required system-pack materialization and validation still run.
- Name the hidden bootstrap dependency inventory artifact, define its schema, and make Slice 3 depend on a passing inventory validator before old package deletion, old path scanner enforcement, or fixture rewrites can merge.

**Questions:**
- Is `GC_BOOTSTRAP=skip` still needed outside tests after `internal/systempacks` exists, or can it be deleted from production bootstrap handling entirely?
- Which artifact owns the authoritative list of current bootstrap consumers: `source-consumer-closure.yaml`, `asset-migration-ledger.yaml`, or a separate bootstrap fixture inventory?
