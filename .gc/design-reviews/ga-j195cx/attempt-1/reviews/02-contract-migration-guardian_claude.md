# Elias Vega - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- Existing `contract = "graph.v2"` formulas keep compiling end-to-end. The compatibility matrix (§Compatibility With `contract`, lines 2566-2572) and per-entry-point table (lines 2623-2632) require every runtime entry — sling, API sling, formula-backed orders, convergence create/retry/speculative wisp, fanout fragments, `on_complete` continuations — to accept the alias and warn rather than block. External SHA-pinned packs are explicitly preserved through the alias window (lines 2637-2638, 2684-2688), and pinned-SHA immutability is respected by `formula.migration.pin_pack_revision`'s `edit_target` enumeration (lines 2766-2784) instead of asking authors to edit immutable objects.
- `contract`/`requires` conflicts are deterministic and hard-fatal, not silently accepted. Diagnostic ordering (lines 244-260, step 5) puts conflict ahead of host-capability satisfaction; the validation matrix locks `graph.v2 + >=1` to `formula.compiler_requirement_conflict` (line 1098), combined-defect precedence repeats it with disabled host (line 1118), and the compatibility-corpus row pins dual-disagreement spellings to the same fatal code (line 2588). Legacy `version` is structurally prevented from satisfying `requires` (lines 1006-1029, 3298) — that closes a real silent-acceptance vector.
- Alias removal is gated on durable evidence with a named owner, not the calendar. The five conditions (lines 2660-2680, 3256-3265), the sliding two-report alias-drain window (lines 5287-5293), and the explicit ban on process-local counters as drain evidence (lines 2967, 2970-2973) are concrete. The release captain is the sole gate-satisfier (line 2660-2662); each release artifact has a named owner, generator command, exit contract, and consumer (lines 3110-3123). The 60-calendar-day rule is positioned as a *floor*, not a sufficient trigger — exit `2` still fires while external-support is active or alias-drain is nonzero (lines 2961-2968).

**Critical risks:**
- **[Blocker] Host capability default can break every legacy `contract = "graph.v2"` formula on first upgrade.** New binary with `[daemon] formula_v2 = false` reading a `contract = "graph.v2"` formula fails with `formula.compiler_requirement_unsatisfied` (line 2569, 2802). The design specifies the diagnostic but never pins down the default value of `[daemon] formula_v2` on the binary that first ships requirements normalization, nor an upgrade-time migration step. If `false` is the binary default, every existing graph-v2 formula stops compiling the moment operators upgrade — the formula source is unchanged, the alias is honored by the parser, but the host gate kills the compile. That is exactly the "legacy formulas fail abruptly" red flag, and the alias-drain machinery cannot detect it because nothing successfully compiles to leave evidence. This single missing decision can invalidate the whole compatibility story for a fielded city that did not explicitly enable `formula_v2`.
- **[Major] Background accepted-alias counts require a durable write contract that the design names but does not guarantee.** The alias-removal gate insists counts come from "accepted compile artifacts, workflow-root metadata, producer diagnostic state, and saved release reports" (lines 2970-2973), and explicitly rejects in-memory LRUs, Event Bus warning payloads, and process counters. But warnings "do not publish Event Bus events" (lines 3352-3355), CLI suppression is per process invocation, and the producer-local LRU is capped at 4096 keys with oldest-entry eviction reset on restart and config reload (lines 3406-3413). No writer-side invariant is specified that every alias-accepting compile (CLI preview, sling, order dispatch, convergence retry, fanout fragment, `on_complete`, dashboard preview, retry continuation) emits a queryable durable record. Without an enforced producer-coverage gate, the drain can return zero by silently dropping evidence rather than by actually draining — and the gate passes false-positive.
- **[Major] Initial compatibility-corpus enumeration is load-bearing yet unsourced.** The corpus accepts whitespace and case variants "when accepted by the current released compiler" (lines 2547-2550, 2581) and narrowing the set requires saved external-support evidence (lines 2595-2598). But no artifact, generator, or test is named that *enumerates* every spelling each supported old binary actually accepts at alias-window start. If the corpus omits a spelling that fielded binaries currently accept, externally pinned packs that use that spelling will fail abruptly on the first new binary — and the deprecation warning never triggers because the parser routes them through `formula.compiler_requirement_unsupported` (line 2582) rather than the alias path.
- **[Minor] `last_validated_at` staleness has no numeric floor.** Lines 2716-2718 require non-stale `last_validated_at` for every external-support row, and the alias-removal gate exits `2` on "stale external-support row" (line 2987). But "stale" is never defined. A release captain could run discovery once at alias-window start, never refresh it, and pass the gate months later with technically-fresh-but-actually-static evidence — exactly the "deprecation scheduled by calendar rather than remaining-usage evidence" failure mode.
- **[Minor] Active-root waivers are unbounded.** Lines 5300-5302 allow waivers naming "the root id, owner, diagnostic, and operator-approved abandonment/cleanup plan," but no time-limit field, no re-review cadence, and no rule that an expired waiver flips the report to blocking. A release captain can land a waiver in one release and ride it indefinitely through Phase 8 conversion and into Phase 9 alias removal.
- **[Minor] Post-rollback drain-window boundary is ambiguous.** Lines 5244-5252 say a Phase 8 rollback "resets the alias-window clock" but the sliding drain window's "latest release candidate and the previous completed minor release" (lines 5287-5293) is not explicitly tied to the post-rollback alias-window-start digest. A drain pair spanning a rollback boundary could satisfy "two consecutive qualifying reports" while the previous report reflects pre-rollback state.

**Missing evidence:**
- The default value and upgrade-migration policy for `[daemon] formula_v2`. Without it, the compatibility matrix's `formula_v2 = false` column is the most likely real-world configuration.
- The procedure and checked-in artifact that seed `internal/formula/testdata/compat_corpus/` from each supported old binary's actual acceptance behavior at alias-window start.
- A writer-side invariant or test ensuring every alias-accepting compile path produces durable evidence queryable by the alias-drain gate. The `caller-preflight` matrix (lines 710-735) tests zero-write for fatal cases but does not assert that *accepted* alias compiles leave evidence.
- A numeric staleness threshold (days or release cadence) for external-discovery `last_validated_at`.
- An expiry/re-review rule for active-root waivers and a rule binding them to a specific Phase 8 release tag.
- An explicit rule that post-rollback drain reports must reference release candidates whose `formula-compiler-alias-window-start.json` digest is the post-rollback one.

**Required changes:**
- Add a rollout step (Phase 2 or Phase 5) that pins upgrade behavior of `[daemon] formula_v2`: either default it to `true` on binaries shipping requirements normalization, or auto-promote when the city's pack/formula inventory contains any `contract = "graph.v2"` or `[requires] formula_compiler = ">=2"` source. Either way, name the migration step that prevents implicit `formula_v2 = false` → "every legacy graph formula fails on upgrade." Include a release-gate test that boots a city with legacy formulas plus default config and asserts compile success with the deprecation warning, not `formula.compiler_requirement_unsatisfied`.
- Add an executable contract: every caller path in the `caller-preflight` matrix that can accept legacy `contract` must have a paired test asserting the accepted artifact, workflow-root metadata row, or producer diagnostic-group entry is written. CI fails when a new producer accepts an alias without registering a writer. Without this the `background_accepted_alias` count is unreliable by construction.
- Add a corpus-seeding artifact and command. Name the generator that runs each supported old binary against the full spelling matrix and writes the resulting accepted/rejected rows into `internal/formula/testdata/compat_corpus/`. Record the corpus digest in `formula-compiler-alias-window-start.json` so any narrowing during the window is auditable.
- Define a numeric `discovery_freshness_days` on `formula-compiler-external-support.json` (e.g., ≤30 days or the most recent supported minor, whichever is shorter) and bind the alias-removal exit to it.
- Bound active-root waivers with `expires_at`, `next_review_at`, and a rule that an expired waiver flips the active-root report to blocking. Cite the Phase 8 release tag in the waiver row.
- Tighten the sliding-window text: both reports in a post-rollback drain window must reference release candidates whose alias-window-start digest matches the post-rollback artifact.

**Questions:**
- Is the existing fielded population running with `[daemon] formula_v2 = true` already? If yes, document it as a Phase 2 precondition. If not, what's the migration step that prevents an upgrade from killing every `contract = "graph.v2"` compile?
- For SHA-pinned external packs whose maintainers never respond and that never surface in any discovery scan, the `unknown` status appears to block alias removal indefinitely (lines 2684-2686 vs lines 2759-2764). Is that intentional, or does the published `public_notice_url` plus the alias-window calendar/release floor cover the truly-undiscovered tail?
- The drain JSON's `background_accepted_alias` count — what producer writes it? Is the storage durable across controller restart and `config_generation` change? If not, a freshly-restarted release-candidate controller can show zero while live alias acceptance continues.
- Phase 9 requires "first-party `dual_declared == 0` for two consecutive minor releases" via the alias-removal gate (lines 2888-2893), but the drain definition cites "the latest release candidate and the previous completed minor release" (line 5287-5288). Is a release candidate a substitute for a published minor for Phase 9, or must Phase 9 wait for two *published* minors with `dual_declared == 0`?
- Who owns publishing compatibility branches for first-party packs that the `formula.migration.pin_pack_revision` hint points consumers at — Phase 3 (dual-floor declaration) or Phase 5 (compatibility bridge)?
