# Saoirse Raman - Claude

**Source reviewed:** `engdocs/design/formula-compiler-requirements.md` (6361 lines, current `main` checkout on branch `fix/560-sync-double-restart`). Context files read: `engdocs/architecture/formulas.md`, `engdocs/proposals/formula-migration.md`, `docs/reference/formula.md`. Lane: pack revision boundary, external author workflow, pinning and reproducibility, lint surface. This iteration of the document is materially longer than the prior snapshot my persona reviewed; several earlier pack-versioning findings (packman schema-2 provenance prerequisite, per-row external-support schema, `edit_target` for SHA-pinned packs, byte-stable `gc.formula_requirement_axes`) are now addressed in the body, and Phase 7 has been split into 7a (packman provenance contract) and 7b (pack-floor enforcement). The risks below are what survives that improvement and what is new.

**Verdict:** approve-with-risks

**Top strengths:**
- Formula-level artifact semver is *decisively* removed, not merely discouraged. It is a Goal (#3 — "Remove formula-level artifact versioning from the design"), a Non-Goal ("Versioning individual formulas independently from their containing pack"), an entire section ("Formula Artifact Versioning"), a glossary pair ("Pack revision | The artifact identity and reproducibility boundary for formulas"; "Formula `version` | Legacy formula metadata only; not semver and not a compiler selector"), persisted root provenance keys (`gc.formula_pack_name`/`gc.formula_pack_source`/`gc.formula_pack_ref`/`gc.formula_pack_revision`/`gc.formula_lockfile_key`/`gc.formula_reproducibility`), and a Consequence ("Formula consistency remains anchored at the pack revision boundary"). The surviving legacy `version` is actively neutered: matrix rows where `version = 1` or `version = 2` plus a v2-only construct without `>=2` fail with `formula.compiler_requirement_missing` regardless of `version`; `formula.version_misuse` warns on combinations that look like the author intended `version` as a selector; `--legacy-version-report` is a separately-owned release-gate command; and the stale-guidance CI has parsed-TOML-key matchers (`formula-version-selector`, `formula-version-prose`) that fail on docs presenting `version` as semver or compiler selection. There is no surviving path by which `version` becomes de facto formula semver — the first red flag is closed.
- The external-author workflow is a first-class deliverable rather than a changelog note. `gc formula validate --pack-path ./packs/acme --all --json` and `--pack-source <url> --ref <ref> --all --json` are explicitly "a stable user-facing surface, not only an internal release gate", returning a `schema_version`-stamped object (`pack`, `requested_ref`, `locked_revision`, `dirty`, `reproducibility`, `formulas`, `imports`, `shadowed_formulas`, `diagnostics`, `migration_hints`). `migration_hints` is versioned JSON with stable codes (`add_requires`, `keep_dual_declaration`, `remove_legacy_contract_when_floor_allows`, `raise_pack_requires_gc`, `pin_pack_revision`) and full binding identity (`binding_name`, `binding_path`, `parent_binding`, `lockfile_key`, `requested_ref`, `locked_revision`, `requires_gc_floor`). SHA-pinned-pack immutability is respected via the `edit_target` enum (`consumer_import_ref` / `consumer_lockfile` / `upstream_pack_branch` / `forked_pack_source` / `none`) plus the explicit rule "For SHA-pinned packs, the original source object is immutable. Validation must not suggest editing that SHA in place." The external-support artifact is per-row JSON (`docs/release/formula-compiler-external-support.json`) with `discovery_method`, `maintainer_contact_status`, `public_notice_url`, `migration_path`, `last_validated_at`, `blocks_alias_removal`, and deterministic `unknown`/`unreachable` expiry semantics ("network failure alone is exit `1`, not an expiry decision"). The acquisition contract is read-only — temp checkout, never mutates `packs.lock` or installed-pack cache, non-interactive credentials, no prompts. Lane question 2 is well-answered.
- Layer winners are designed to be traceable to pack revision. Typed `Provenance`, `PackProvenance`, `ImportProvenance`, and `ResolvedFormulaSource` carry pack name, source, requested ref/constraint, locked revision, content hash (lowercase-hex SHA-256 of canonical resolved bytes with a length-prefixed manifest for multi-file expansion/aspect contributions, *excluding* absolute staging paths), dirty state, transitive import chain, parent binding, and layer priority. "Layer resolution must preserve pack binding before symlink staging — a staged winner path under `.beads/formulas/` is not enough provenance by itself" is stated normatively. Shadowed-formula diagnostics report winner path / losing path / layer priority and may be promoted from warning to failure for first-party packs because a later layer-order change could expose them. Direct-pack validation has a synthetic binding identity (`__direct_pack__`) that is "never written to `packs.lock`, never treated as an installed import, and never reused as durable root provenance" — a nice guardrail against staging contamination. The canonical workflow-root predicate centralizes `gc.formula_*` matching with a deterministic 8 KiB / 1024-byte spillover to `gc.formula_compile_artifact`, and the four `reproducibility` values (`pinned` / `local-clean` / `local-dirty` / `local-not-reproducibly-pinned`) carry explicit release-gate policy. This closes the third red flag.

**Critical risks:**
- **[Major]** The resolver→compiler `ResolvedFormulaSource` plumbing is the foundation of *every* pack-revision-traceability claim, yet it is not explicitly named as a Phase 2 deliverable with its own rollback unit. Today (per `engdocs/architecture/formulas.md`) `ResolveFormulas()` symlinks winners into `.beads/formulas/` and the downstream `Store.MolCook` path receives a filename, not a typed source. Phase 2 ships `CompileResult`, `AcceptedCompileArtifact`, canonical metadata keys, dual-stamping, and the shared workflow-root predicate — but populating `CompileResult.Provenance` with real `PackProvenance` requires changing the resolver/import boundary to produce and thread `ResolvedFormulaSource`, with which fields are available from packman schema 1 (binding name, source URL, requested ref, locked `commit` from `[packs."<source>"]`) versus which require schema 2 (durable content hash, lockfile-stored `requires_gc`, parent binding, transitive depth). The design states the *requirement* ("the resolver/import layer owns `ResolvedFormulaSource` construction"; "no caller may reconstruct pack name, requested ref, locked revision, dirty status, or transitive import chain from a staged path … after compilation") and even gives packman schema 2 its own fail-closed phase (7a). It does not call out that a Phase 2 shipping `CompileResult` with a path-derived or empty `Provenance` would hollow out every downstream `--provenance` / `migration_hints` / `gc.formula_pack_revision` / `--requirement-diff` claim and would make the visible-diagnostic gate in Phase 2 ship deliverable but unverifiable. Name resolver→compiler `ResolvedFormulaSource` threading as an explicit Phase 2 deliverable with its own rollback unit, state which `ResolvedFormulaSource` fields come from schema-1 packman + resolver state versus which are blocked on schema 2, and add a test that a Phase 2 produced workflow root has a non-empty `PackProvenance.Source` and either a `LockedRevision` (schema 2) or a documented `Reproducibility=local-clean` with a VCS revision.
- **[Major]** Lane-question 3 is partly answered: the validation commands and `migration_hints` schema are excellent, but the document never uses the word "lint", and the `gc formula validate --pack-path` / `--pack-source` commands are not pinned to a rollout phase. They are absent from the `docs/reference/cli.md` required-update list (which names only `--provenance` and `--legacy-contract-report`), from the diagnostic-visibility matrix, and from the per-phase command table; the visibility matrix and rollout DAG do not say which release first gives external authors the migration tooling. An external pack author reading this design cannot tell which Gas City version first emits the typed `migration_hints` JSON they are being asked to consume — yet the design simultaneously prohibits first-party requires-only conversion until external consumers have a migration path. Pin `--pack-path` and `--pack-source --ref` to a phase no later than the first user-visible `[requires]` diagnostic (so it lands in the same release that starts the alias clock), add them to the diagnostic-visibility predecessor table, add them to the `docs/reference/cli.md` required-update row, and either explicitly state that `gc formula validate` *is* the lint surface (one sentence in the docs-rewrite skeleton, e.g. "this command is the canonical lint path for pack authors") or add a `gc formula lint` alias so external author tooling has a stable verb.
- **[Minor]** The `[pack] requires_gc` comparator behavior — release / prerelease / nightly / `main` / dirty / source-build / unknown rows, `pack.requires_gc_active_version_unknown`, `active_binary_dirty`, "build metadata ignored for ordering" — is specified in normative detail in a document that simultaneously disclaims ownership ("The comparator is centralized in the packman-owned compatibility package … Formula validation and pack acquisition call that comparator; they do not parse Gas City versions locally"). `requires_gc` is parsed today (`internal/config/config.go`, `cmd/gc/cmd_init.go`) but never compared, and the comparator package does not exist. Specifying behavior you say you do not own is a coordination hazard: if packman builds it differently these rows go stale, and pack-floor remediation hints will diverge from what the resolver actually does. The design already sequences schema-2 ownership well (7a before 7b); it should do the same for the comparator. Either move the comparator spec to a packman-owned doc and reference it (or mark these rows "proposed requirements for the packman comparator, pending ratification by the packman owner"), and add the explicit sequencing statement that 7b enforcement cannot ship in the same PR as the first packman comparator implementation. Without this, Phase 3 first-party `[pack] requires_gc` declarations may land against a comparator interface that does not exist or differs in dirty-source behavior.
- **[Minor]** `migration_hints[].safe_automatic_edit` ships in v0 with no in-tree consumer (the design correctly rules out an SDK `--apply` that rewrites third-party packs and explicitly says it needs a separate design), and `discovery_method: registry_query` / `registry_mirror` presuppose a pack registry Gas City does not have. The deferrals are reasonable, but a half-populated schema invites confusion and pushes the entire first-party fixture/example/tutorial/bundled-pack migration onto hand edits concentrated on the release captain. Either document the forward-compat justification in-schema (external tooling may consume `safe_automatic_edit`; `registry_query` is a future hook) and park a named follow-up — even just a Strict Review Risk Register row — for a first-party migration tool so requires-only conversion is not a manual-edit cliff, or drop the unused fields until they have a consumer.
- **[Minor]** `formula.version_deprecated` is explicitly suppressed on launch, order, retry, convergence, and controller paths and emitted only by validate/show/preview surfaces. The rationale ("operational logs are not polluted by an artifact-identity warning after a formula has already been accepted") is defensible for first-party operators, but it has a side effect for external pack consumers: a pack with stale `version = 2` and no v2-only constructs can ship for years through `gc sling --formula` with no warning the field is legacy. The `formula.version_misuse` warning covers the dangerous case (legacy `version` *plus* v2-only syntax without `[requires]`), but the merely-stale case is silent. Either confirm explicitly in the rollout section that merely-stale `version` is acceptable forever and silence is intentional, or fire `formula.version_deprecated` once per launch invocation through the same per-process LRU as `formula.contract_deprecated` so external consumers eventually see it during operator-driven runs.

**Missing evidence:**
- The concrete minimum-binary-floor version (`<minimum-floor-from-formula-compiler-min-floor.json>`) is genuinely unknowable until the feature ships, and the rollout correctly defers it. But every pinning/reproducibility guarantee for first-party requires-only packs — and every external pack that copies the modern requires-only example before Phase 7b lands — rides on a number that does not exist yet. Binaries that already parse `requires_gc` and ignore it sit in a "parse-but-don't-enforce" window where a requires-only formula with v2-only syntax will *mis-compile* rather than reject the pack. The design says "External SHA-pinned formulas that use legacy `contract = "graph.v2"` remain valid through the alias window" but does not state the converse: an external pack that copies the modern requires-only example today, on a non-enforcing binary, fails closed on the requirement but had no resolver-level safety net. Add an explicit row in the compatibility matrix (or external-author docs) that external authors should keep dual-declaring through the alias window for the same reason first-party packs do — not merely because `contract` is the alias.
- For `first_party_builtin` workflow roots, no persisted root metadata key records *which* Gas City build produced the root. The construct-registry release artifact records a "registry-owning binary SHA" and the active-binary version source/comparator are well-specified, but neither lands on the root. `gc.formula_pack_revision` for a builtin pack is presumably a content hash that does not identify the producing binary, and `gc.formula_pack_source = "builtin"` is a lossy classification. Builtin packs are how most users hit graph workflows; if builtin-pack workflow-root reproducibility matters, pin (in this design or as a Strict Review Risk Register row) what `gc.formula_pack_revision` means per `source_class`, whether the producing Gas City version belongs on the root, and how `gc formula repair-root-artifact` decides identity for a builtin root produced by an unknown earlier binary.
- The design lists `public_notice_url`, `notice_sent_at`, `maintainer_contact_status`, and `discovery_method` ∈ {`installed_pack_scan`, `lockfile_scan`, `registry_query`, `known_partner`, `support_request`, `manual_report`} as the outreach contract for external pack authors, but it does not name a publication channel. SHA-pinned external authors cannot subscribe to "a URL" — they need a specific surface (release notes? a `gascity-announce` discussion forum? a `migrations.md` file? the `docs/release/` directory in the public repo?). The design also does not say who scans for external packs in the wild and authors `support_request` / `manual_report` rows. Without a named channel, "the artifact mandates a public notice" is checklist-green but practically silent.
- Lane question 1 is well-answered by the prose, but I cannot find a fixture or test that asserts a v0 binary refuses to read `formula.version` *as* a semver in any production code path. The matrix locks the legacy-`version`-plus-construct combinations and the stale-guidance CI matches docs, but there is no `TestFormulaVersionIsNotABehavioralInput` or equivalent static guard that scans `internal/formula` and `internal/convergence` for branches on the decoded `Version` field. Without it, the prose statement that `version` is "never a compiler requirement" is enforced by code review rather than by CI.

**Required changes:**
- Name resolver→compiler `ResolvedFormulaSource` threading as an explicit Phase 2 deliverable in the Rollout Plan, with: (a) its own rollback unit (revert provenance plumbing, fall back to filename-only compile), (b) a table mapping each `ResolvedFormulaSource` field to "schema-1 packman + resolver state" vs "blocked on schema 2", and (c) a Phase 2 acceptance test that a real workflow root has non-empty `PackProvenance.Source` and either `LockedRevision` or a documented local-VCS reproducibility value.
- Pin `gc formula validate --pack-path` and `gc formula validate --pack-source --ref` to a specific rollout phase no later than the first user-visible `[requires]` diagnostic. Add them to the diagnostic-visibility predecessor table, to the `docs/reference/cli.md` required-update row, and to the per-phase command table. Add one sentence to the docs-rewrite skeleton naming `gc formula validate` as the lint surface for pack authors (closes the `lint surface` mandate), or add `gc formula lint` as an alias.
- Move the `[pack] requires_gc` comparator specification rows into a packman-owned doc (e.g., `engdocs/architecture/packman.md` or `docs/packv2/doc-packman.md`) and reference them from this document, OR mark the rows "proposed requirements for the packman comparator, pending ratification by the packman owner". Add the explicit sequencing rule that Phase 7b enforcement cannot ship in the same PR as the first packman comparator implementation, paralleling the existing rule for schema-2 lockfile writing.
- Add a Strict Review Risk Register row for "no first-party migration tool for converting `contract = "graph.v2"` → `[requires] formula_compiler = ">=2"` in bundled packs, examples, tutorials, fixtures, and Ralph demo formulas" — or commit to building one before Phase 8 conversion. Either document `safe_automatic_edit` as an external-tooling forward-compat field with a named follow-up design, or drop the field until a consumer exists. Same treatment for `registry_query` / `registry_mirror`.
- State the outreach channel for external pack authors: name the surface (release notes page, GitHub Discussions category, `docs/release/migrations.md`, or whatever the project chooses) and name the role that authors `support_request` / `manual_report` rows in `formula-compiler-external-support.json`. Without this, `public_notice_url` is a typed field with no publisher.
- Add a row to the compatibility matrix: "External SHA-pinned pack with requires-only graph formula, reader at parse-but-don't-enforce `[pack] requires_gc` binary | fails with `formula.compiler_requirement_unsatisfied` after passing pack load; recommended remediation is dual-declare during alias window". This is the implicit counterpart to the existing "Old binary reading old external SHA-pinned pack | Works through `contract` alias" row.
- Either explicitly state that merely-stale `version` is acceptable forever (and justify the silence on launch surfaces), or fire `formula.version_deprecated` once per launch invocation through the same per-process LRU used for `formula.contract_deprecated`.
- Add `TestFormulaVersionIsNotABehavioralInput` (or equivalent) to the static-guard inventory: production code outside the parser, raw scanner, and `--legacy-version-report` reader may not branch on the decoded `Version` field. This converts the lane-1 invariant from prose to CI.

**Questions:**
- Who owns the cadence of `gc formula validate --all-packs --external-support` and `--legacy-contract-report`? The release captain owns the artifact and exit contract, but the document is silent on whether these run in CI per PR, in nightly release validation, or only when the release captain manually invokes them. Stale evidence is fail-closed at the gate, but a "fresh enough" definition would prevent surprises at release time.
- Is `gc formula validate --pack-path` / `--pack-source --ref` runnable from a container image without a full Gas City city, or must external pack authors install a complete Gas City binary and initialize a city to validate their packs? If a full install is required, that is friction worth a follow-up. If a self-contained `gc formula validate` is the intent, that is worth stating.
- Why preserve `formula.version` *forever* as legacy metadata rather than scheduling its eventual removal alongside the parser-`contract` alias removal in Phase 9? Is there real-world tooling that requires the field, or is silence the lower-risk default? A single sentence ("we intentionally keep `version` as inert legacy metadata because [reason]") would close the question.
- Same-identity host-downgrade reuse permits already-accepted graph roots to continue mutating after `[daemon] formula_v2` is disabled. For *builtin*-pack roots, what happens if the Gas City binary itself is downgraded across the version that introduced `[pack] requires_gc` enforcement? The `host-downgrade-same-identity` artifact-reuse rule does not branch on builtin vs external — is a downgrade across the schema-version-bumping release a same-identity reuse, or does the `RequirementsSchemaVersion` mismatch invalidate the artifact? The forward-compat invariants suggest the latter; one sentence confirming would help.
- Does `gc formula repair-root-artifact` for a builtin-pack root depend on `internal/bootstrap/packs/` shipping the exact same source at the binary's checkout, or does it accept a content-hash match against a different vendored copy? Bundled-pack reproducibility identity is the implicit counterpart to external SHA pinning and deserves the same treatment.
