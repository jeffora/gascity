# Avery Brooks - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- A real glossary exists at lines 4639-4656 with explicit distinctions between *formula compiler capability*, *host capability*, *compiler implementation*, legacy `contract`, `schema`, *pack revision*, formula `version`, formula `[requires]`, pack `[[pack.requires]]`, pack `[pack].requires_gc`, host `[daemon] formula_v2`, *compiler capability 2*, *Formula v2*, and *graph workflow*. Lane question 2 ("one glossary-level distinction between capability, implementation, contract, schema, and pack revision") is structurally satisfied.
- The docs gate is executable, not advisory: a doctest fixture at `docs/reference/testdata/formula-requirements-doctest.yaml` parses every TOML block (lines 4737-4746); the blocking stale-guidance YAML at `docs/release/formula-compiler-stale-guidance.yaml` (lines 4879-4959) explicitly covers `contract`, `version = 2`, `GC_NATIVE_FORMULA=false`, `formula_v2` selection wording, and a `pack-requires-confusion` matcher; and `make formula-docs-check` is a hard predecessor for any user-visible diagnostic (lines 4988-5005, 5343-5356).
- The reference-page rewrite skeleton (lines 4697-4720) is built around two reader-facing tables — "which key do I edit" (lines 4747-4755) and "common-confusion" (lines 4759-4768) — that map directly to the surfaces a confused author would conflate. The same-PR docs-bundle rule and the diagnostic visibility matrix (lines 5343-5350) block any diagnostic surface that lacks paired docs/CLI-help/OpenAPI/dashboard-type updates.

**Critical risks:**

- **[Blocker] Warning-transport policy is decided for `formula.version_deprecated` only, not for `formula.contract_deprecated` or `formula.version_misuse`.** Lines 5036-5040 restrict `formula.version_deprecated` emission to validation/display surfaces (`gc formula validate`, `gc formula show`, formula API previews). Lines 1020-1023 anchor `formula.version_misuse` ordering relative to `version_deprecated` "on validation/display surfaces" — but the compatibility matrix at lines 1095-1098 and the alias-window prose at lines 2546-2553 emit `formula.contract_deprecated` whenever the alias is parsed, with no surface restriction. The In-Flight matrix (lines 5404-5416), the rollout visibility matrix (lines 5343-5350), and the per-entry-point table (lines 2625-2638) never name which surface emits which warning code. Operationally: a formula author who launches `version = 2` with v2 syntax but no `[requires]` will hit `formula.compiler_requirement_missing` (fatal) at launch yet never see `formula.version_misuse` unless they happen to run `gc formula validate`. This is the exact "open question about warning transport" called out in lane question 3, made user-visible by the rollout (red flag 3).

- **[Blocker] `[[pack.requires]]` is named in the glossary as a real, parallel surface, but the document never demonstrates its grammar, parser, or relationship to `[pack].requires_gc`.** Glossary line 4651 introduces "Pack `[[pack.requires]]` — Pack-owned dependency/capability declarations for pack ecosystem constraints; not the formula compiler requirement table." The common-confusion table (line 4762) and the requirement-surface table (line 4807) both reference it, but no TOML snippet shows what `[[pack.requires]]` looks like; no doctest fixture covers it; no PackV2 doc reference is cited; and the stale-guidance `pack-requires-confusion` matcher (lines 4943-4951) only checks prose for context phrases, not for a positive cross-reference. The asymmetric TOML shapes (`[requires]` table for formulas, `[[pack.requires]]` array-of-tables and `requires_gc` scalar for packs, all spelled with the same root word) are exactly the overload that bites authors who copy-paste. This is red flag 2 ("`requires` is overloaded with another pack concept without explanation") and a glossary line alone is insufficient explanation.

- **[Major] The reference-page rewrite skeleton requires `--legacy-contract-report`, `--provenance`, `--requires-only-conversion-gate`, `--alias-removal-gate`, `--compat-corpus`, and `--all-packs` CLI flags that the design never specifies.** Lines 4665, 4853, 5222, 5224, 5228, 5229, and 2671 reference these flags as release-gate inputs. None has a JSON shape, exit-code contract, or composition rule defined in this document. A docs reviewer cannot confirm "Diagnostics: stable code table, CLI/API/dashboard projection, and remediation wording" (line 4712) when the flag surfaces are dangling pointers. Reference page cannot be self-contained until each flag is either defined inline or cited to an existing CLI spec.

- **[Major] A snippet-level docs inventory is missing — only first-party packs and code callers are enumerated.** The `formula-compiler-first-party-inventory.json` artifact (lines 4814-4825) covers packs, examples, tutorials, and fixtures *as files*. The file-family update matrix (lines 4661-4671) and stale-guidance `path_globs` (lines 4882-4902) gesture at architecture docs, proposals, and PackV2 author docs, but a glob is not an enumeration. The existing `engdocs/architecture/formulas.md` (lines 47-50 in the architecture doc) currently presents `version` as "Optional formula version marker" with no legacy classification; that line must be actively *removed*, not merely paired with new content. Without a checked-in inventory of prose passages with per-passage migration outcomes (`rewrite-canonical`, `mark-legacy`, `delete-snippet`), the rollout gate cannot deterministically discharge lane question 1 ("Do docs/reference/formula.md, tutorials, architecture docs, and examples all teach `requires` as the canonical surface in the same change?"). The `formula-version-prose` matcher (lines 4914-4920) flags drift after the fact; it does not enumerate the migration plan.

- **[Major] The canonical pack example carries an unresolved placeholder that the design simultaneously bans in release artifacts.** Lines 4732-4736 and 4781-4786 show `requires_gc = ">=<minimum-floor-from-formula-compiler-min-floor.json>"` as the canonical TOML example. Line 4852 fails the stale-guidance check when `<minimum-floor-from-formula-compiler-min-floor.json>` "survives in release artifacts outside explicitly marked templates." The reference-page TOML is fed to a doctest (line 4744) that parses real syntax; the doctest fixture allows a `template_expected` class, but the design never says whether this canonical snippet is in that class. A pack author copy-pasting the canonical example writes invalid TOML. This is the "open question about exact values" from lane question 3 made concrete: the placeholder is half-canonical, half-template, and the docs gate cannot tell which.

- **[Minor] The design body uses "Formula v2", "formula-v2", "formula v2", and "compiler capability 2" interchangeably — exactly the inconsistency it polices in tutorials.** Glossary line 4655 designates "Formula v2" as prose shorthand and "compiler capability 2" as the executable-contract spelling. The body then uses "Formula-v2 compiler capability" as a heading (line 182), "formula-v2 graph compiler capability" in prose (line 40), and "formula v2" inside the `graph-workflow-host-confusion` matcher's *own pattern* (lines 4953-4955). Tutorials will copy whichever spelling appears nearest the snippet they cite; if the design itself does not model the discipline it demands, the discipline will not be obeyed.

- **[Minor] `formula-compiler-alias-window-start.json` example carries `release_date: "2026-05-11"` (today is 2026-05-12) which mixes fixture-looking data into the canonical design.** Line 5263 of the JSON sample reads `"release_date": "2026-05-11"`. If literal, this leaks a real release date one day before today into a design document. If illustrative, the design contradicts its own placeholder discipline (line 4852 et al.). Either replace with `"<release-date>"` (matching the placeholder style used elsewhere) or move the example into a checked fixture under `internal/formula/testdata/`.

- **[Minor] `engdocs/architecture/formulas.md` is currently stale against this design.** It lists `version` as an "Optional formula version marker" with no deprecation note (architecture/formulas.md lines 47-50), is stamped "Last verified against code: 2026-03-17", and frames Gas City as not owning the formula parser ("`bd` is the full-featured backend for real formula execution today"). The design's file-family table (line 4666) only requires adding "Internal ownership of compile result, diagnostics, and host capabilities"; it does not require removing the stale current-state prose, "Known Limitations", or "Last verified" framing in the same change. Three formula docs (`engdocs/architecture/formulas.md`, `engdocs/proposals/formula-migration.md`, `docs/reference/formula.md`) will end up disagreeing unless the architecture doc's reconciliation is explicit.

**Missing evidence:**

- No JSON shape, exit-code semantics, or example output for `gc formula validate --legacy-contract-report --json` or `--provenance`, despite Phase 3 (line 5222), Phase 5 (line 5224), Phase 8 (line 5228), and Phase 9 (line 5229) depending on these flag surfaces.

Cross-reference the per-entry-point table at lines 2625-2638 so the policy reads the same in both places.
2. **Define `[[pack.requires]]` or remove the row.** Either add a TOML example + doctest fixture + parser-ownership pointer that demonstrates `[[pack.requires]]` syntax and proves it cannot be mistaken for formula `[requires]`, or strike the row from the glossary and the requirement-surface comparison table (line 4807) and the common-confusion table (line 4762) until a separate design lands.
3. **Specify the `gc formula validate` flag surface.** Add a subsection to the reference-page rewrite skeleton (lines 4697-4720) that defines `--provenance`, `--legacy-contract-report`, `--requires-only-conversion-gate`, `--alias-removal-gate`, `--compat-corpus`, and `--all-packs` (input scope, JSON shape, exit-code contract, composition rules with each other).
4. **Add a prose-docs inventory artifact.** Extend `docs/release/formula-compiler-first-party-inventory.json` (or add a sibling) to enumerate every existing docs/tutorial/architecture-doc passage that teaches `contract` or `version` as canonical, each paired with a per-passage migration outcome. At minimum the inventory must include `engdocs/architecture/formulas.md` line 47-50 and the `bd`-shell-out framing in the same file, plus any tutorial snippet that currently teaches `version`. Make the inventory a hard input to `make formula-docs-check`.
5. **Decide the canonical-example placeholder policy.** Either commit the doctest fixture to classify the reference-page `requires_gc` snippet as `template_expected` (and update the example so a reader cannot accidentally copy-paste a placeholder as valid TOML), or replace the placeholder with a concrete minimum-floor value tied to a release artifact. Current state breaks the docs gate.
6. **Pick one canonical spelling in the design body.** Use "compiler capability 2" in all executable-contract prose and reserve "Formula v2" for explicit shorthand boxes. Audit headings (line 182), inline references (line 40), and matcher patterns (lines 4953-4955) to comply.
7. **Replace `release_date: "2026-05-11"`** in the alias-window-start JSON example (line 5263) with `"<release-date>"` matching placeholder style, or convert the example to a checked fixture.
8. **Add `[daemon] graph_workflows` to the glossary** as a deprecated host alias, since the stale-guidance gate treats it as user-visible legacy surface.

**Questions:**

1. Is `[[pack.requires]]` real today (PackV2 land) or forward-looking? The glossary names it as if it exists; the rest of the doc never demonstrates it. Lane question 2 cannot be answered until this is resolved.
2. Are `formula.contract_deprecated` warnings emitted on launch / order / convergence surfaces, or only on validation/display surfaces? The doc currently restricts `version_deprecated` only; two distinct policies for warnings about deprecated declarations will confuse operators.
3. Should `requires_gc = ">=<minimum-floor-from-formula-compiler-min-floor.json>"` ship as a template (with explicit template-class markup) or as a concrete release-anchored value? The current state allows both interpretations.
4. Does the migration section of `docs/reference/formula.md` have to surface alias-window timing and external-support status, or do external pack authors learn the `contract` deprecation deadline only from `docs/release/*.json` artifacts? Learning a deprecation deadline from a release-internal JSON file is fragile for external consumers.
5. Should the authored TOML key be `formula_compiler` or `formula_compiler_capability`? The persisted root metadata key is `gc.formula_compiler_capability` while the authored key drops "capability"; the worked future example uses `state_store` (authored) and `gc.formula_state_store_requirement` (persisted, ends `_requirement`). The naming convention across authored / `*_requirement` / `*_capability` should be stated once so future axes follow it.
