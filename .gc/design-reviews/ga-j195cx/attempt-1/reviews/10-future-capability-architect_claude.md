**Verdict:** approve-with-risks

**Top strengths:**

- Default-capability identity is locked structurally, not by prose. `CompileIdentity.CompileID`, `AcceptedArtifactIdentity` hash, accepted-artifact dedup ref, and `workflow-root kind=default_capability_workflow` are explicitly required to be identical for omitted `[requires]`, empty `[requires]`, and `formula_compiler = ">=1"`. `RequirementSource` is excluded from the behavioral identity record, and the doc names a CI guard rejecting runtime branches on `RequirementSourceRequires|Contract|Dual|Omitted` outside diagnostic/provenance/migration/compat paths.
- Forward-compatibility invariants are spelled out as binding rules: monotonic capability numbers, released constructs immutable, released axis byte grammars frozen, and `docs/release/formula-compiler-construct-registry.json` becoming an immutable release artifact. Old-reader fixtures cross source-parse, persisted root metadata, accepted artifact load, API/dashboard projection, retry, fanout, convergence, and cleanup for future capability `3`, future axes, and schema-version bumps. The "old binary classifies as `unknown_future_capability_workflow` for observation only and fails closed for graph-specific writes" rule is unambiguous.
- Unknown axes and unsupported future minima are deterministic hard errors with fixed diagnostic ordering (unknown axis â†’ unsupported future â†’ host satisfaction). Axis identifier grammar is normative ASCII (`[a-z][a-z0-9_]{0,63}`) and the future-axis shape is normative ("flat scalar string directly under the top-level `[requires]` table â€¦ No user-defined axes in v0 â€¦ A future extension axis must be a first-class typed field, not `x-*` passthrough"). The combined-defect suite explicitly states "unknown axis remains fatal; accepted known axes do not mask unsupported axes."

**Critical risks:**

- [Major] Default-capability runtime equivalence is described in prose ("Tests fail if any runtime path â€¦ differs across those three inputs") but no test name pins it. The named tests in the doc cover registry/matrix shape (`TestConstructRegistryMinimumMatchesCapability`, `TestConstructRegistryRowsCoveredByMatrix`, `TestCompilerRequirementMatrixCountsLocked`, `TestCallerManifestDrivesPreflightMatrix`, `TestWorkflowControlMetadataRegistryComplete`) â€” none of them cover the omitted-vs-`>=1`-vs-empty-table parity assertion across convergence, fanout, retry, dispatch, dashboard projection, and accepted-artifact dedup. The single point where the persona's red-flag divergence could be introduced (omitted and `>=1` accidentally drift) has the strongest prose statement and the weakest named contract. This is the highest leverage gap given my mandate.
- [Major] The "no runtime branch on `RequirementSource`" CI guard is described but its mechanism is unnamed â€” no grep pattern, no AST scanner, no `TestRequirementSourceBranchGuard` is identified. Combined with the prose-only equivalence assertion above, the doc relies on reviewer discipline to keep omitted/`>=1` identical forever. Both should be promoted from prose to named, executable contracts (a test plus a static-analysis matcher) sitting next to the equivalence claim.
- [Major] The future-axis worked example (`state_store = ">=2"`) is itself a `>=N` axis. The doc states "If a future requirement is not monotonic, it must use a different axis," but never demonstrates what a non-monotonic axis grammar looks like, who owns its parser, or how its diagnostics align with the `formula_compiler` family. The persona's mandate covers "reserved requirement axes" â€” concrete near-term candidates the project might want (`tools`, `runtime`, `pack_format`) are not necessarily `>=N`-shaped. Without one worked non-monotonic example, the next axis author has to invent grammar conventions and risks bending `formula_compiler` semantics or producing ad-hoc per-axis grammars. Add one normative non-monotonic worked example covering grammar, owning package, host-capability accessor shape, diagnostic codes, projection behavior, and provenance fields.
- [Minor] Requirement axis identifiers are `[a-z][a-z0-9_]{0,63}` with no reserved namespace. The doc forbids `x-*` passthrough and states "No user-defined axes in v0," but it does not commit to a permanent third-party policy. If third-party packs ever want their own axes (and the typed-field path the doc requires implies that is at least possible), every name is up for grabs first-come-first-served. Either reserve a third-party prefix now (e.g., axes owned by Gas City are unprefixed; third-party axes must use `<owner>_<name>` or are forever rejected) or state explicitly that all axes are forever Gas-City-owned. Today's grammar leaves both doors open and that is itself a forward-compatibility risk.
- [Minor] The "narrow grammar is permanent" contract is implicit. A casual reader of the Requirement Expression table could conclude that `">=2.0"` and `">=2.1"` are merely v0-rejected and could later be accepted by extending the parser. The Forward Compatibility section forbids this ("Released axis byte grammars are frozen â€¦ If a richer grammar is needed, it becomes a new typed axis or a schema-versioned extension"), but the two statements live far apart in a 6,361-line doc. Add a single sentence in "Requirement Expression" stating the rejection set is permanent for the `formula_compiler` axis; richer expressivity requires either a schema-version bump with old-reader fixtures or a new axis. This closes the narrowness-is-caution misreading.

**Missing evidence:**

- No `[requires]` axis is reserved by name beyond `formula_compiler`. The persona's mandate explicitly lists "reserved requirement axes." The doc enables future axes but reserves none. If `tools`, `runtime`, or `pack_format` are anticipated even loosely, listing them now as reserved-future identifiers (with stub `formula.requirement_unknown_axis` diagnostics in v0) would prevent later collisions and would let old binaries surface a more targeted upgrade hint.
- The "old reader sees multi-axis manifest with an unknown axis it doesn't understand" path is described â€” old readers classify the root as `unknown_future_capability_workflow` â€” but the specific old-reader cross-product table at line 6296 covers future compiler capability and future axis, not the case where a known axis (`formula_compiler = ">=1"`) is present alongside an unknown axis in persisted metadata. The matrix calls this out in `combined-defect` ("unknown axis remains fatal; accepted known axes do not mask unsupported axes") but the persisted-metadata old-reader scenario is missing one row.
- The `gc.formula_requirement_axes` manifest is normatively defined (sorted, comma-separated, no padding) but the doc does not show an example with two axes, nor a fixture row asserting an old reader's behavior when it sees an axis it does not understand in the manifest. The "byte-stable" claim should be paired with one concrete two-axis fixture.
- The doc says "no boolean 'is v2' helpers in new code" but does not name the CI matcher that enforces it. A future developer could still write `if cap >= 2 { â€¦ }` outside `internal/formula` and create exactly the omitted-vs-`>=1` divergence my mandate warns about (since today `cap == 1` for both, but a refactor could conceivably introduce a "treat omitted as cap=0 implicitly" optimization). State the scanner rule (e.g., "comparisons against `CompilerCapability` literals outside `internal/formula` fail CI").

**Required changes:**

- Add named, executable contracts for the omitted/`>=1`/empty-table equivalence: a single `TestDefaultCapabilityRuntimeEquivalence` (or named family) that exercises every runtime surface listed in the default-capability identity table (CompileID, accepted-artifact hash and dedup ref, workflow-root kind, convergence/fanout/retry validation, dispatch, dashboard projection) across the three input spellings, plus a `TestRequirementSourceBranchGuard` static check matching the prose CI guard. Reference both names next to the prose so the contract is visible at the point of claim.
- Add one worked non-monotonic future axis example, including: grammar fragment, owning package, host-capability accessor, three diagnostic codes mirroring `formula.<axis>_requirement_invalid_syntax|invalid_type|unsupported_future|unsatisfied`, persisted metadata key, and projection behavior. Keep `state_store` as the monotonic example and add e.g. `tools = "ripgrep,jq"` or `runtime = "javascript"` as a non-monotonic example so future axis authors have a template that doesn't accidentally inherit `>=N` assumptions.
- Add an explicit axis-identifier namespace policy. Either: (a) all axes are forever Gas-City-owned and the doc says so; or (b) third-party axes must use a reserved prefix/suffix shape, and v0 rejects any axis violating that shape. The current "byte-exact lowercase ASCII â€¦ no user namespace prefixes" is consistent with (a) but the doc does not commit, and "A future extension axis must be a first-class typed field" implies extension by parties other than Gas City is at least possible.
- Add one sentence in "Requirement Expression" stating the rejection set for `formula_compiler` is permanent â€” richer grammar requires a new axis or schema-version bump, never a relaxation of `formula_compiler` parsing. The Forward Compatibility section says this; lifting one line into the section a reader hits first prevents misreading the rejection table as a v0-only conservatism.
