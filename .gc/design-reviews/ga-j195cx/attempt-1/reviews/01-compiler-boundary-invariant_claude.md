# Nadia Sorenson - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- Compiler authority is anchored in syntax, semantics, and CI. `[requires]` is unambiguously a minimum-capability statement, not an implementation selector ("The active Gas City binary decides which compiler path to use"; "the formula does not choose the compiler at execution time"). The glossary fixes "Compiler implementation" as binary-chosen, the stale-guidance matchers (`compiler-selector-wording`, `formula-v2-positive-content`) refuse docs that imply otherwise, and named static guards (`TestNoFormulaPackageGlobalCapabilityReads`, `TestNoFormulaHostCapabilityBypass`, `TestRequirementSourceProvenanceOnly`) prevent regression at the code level.
- Single normalization path is structurally enforced. The two-stage parser (raw `RawRequirementField` capture → `NormalizeRequirements`) preserves source attribution; `NormalizedRequirements` is unexported-fields with accessor-only readers; `formulahost.FromCityConfig` is the only edge that converts `[daemon] formula_v2`/`graph_workflows` into a `HostCapabilityInput`; per-occurrence caller manifest + `raw_consumer_allowlist.yaml` + `TestNoNewFormulaRawConsumers` reduce raw-string consumers to a small allowlisted set with owner/expiry/replacement rows.
- The preview/accepted projection split is structurally different, not just nominally. `PreviewProjectionSnapshot` and `AcceptedProjectionSnapshot` are distinct types; `AcceptedCompileArtifact.proof` is unexported with a byte-deterministic identity hash and an invalid zero value; `ValidateAcceptedArtifact(writeIntent)` is the only authorization gate. Combined with `TestDurableWritersRequireAcceptedArtifact`, durable callers cannot fabricate authority from a `*Recipe`, a preview `CompileResult`, root metadata, or a `bd` probe.

**Critical risks:**
- [Major] **Two compile entry points create a soft drift surface.** The design designates `PreflightFormula` as "the canonical entry point for new behavioral callers" (line ~1514) yet `CompileWithResult` is the public API used throughout the per-caller migration table, the convergence executable pseudocode (lines ~5567–5586), and the durable preflight contract (line ~2233). The legacy `Compile(...) (*Recipe, error)` wrapper is *also* retained as "preview compatibility only." Three compile-shaped exported entry points (`PreflightFormula`, `CompileWithResult`, `Compile`), each with different guarantees, is exactly the shape this lane is meant to prevent. A new caller can wire `CompileWithResult` → `AcceptCompileResult` → `ValidateAcceptedArtifact` by hand, get past every static guard, and still skew identity construction. There is no test named that prevents a new producer from constructing `CompileIdentity` outside `internal/formula`.
- [Major] **Legacy host-capability globals have no hard expiry.** `formulaV2Enabled`, `IsFormulaV2Enabled`, `SetFormulaV2Enabled`, `SetGraphApplyEnabled`, and `IsGraphApplyEnabled` are "temporary wrappers"/"edge-only shims" with the migration table saying "removed after CLI/API/controller callers pass explicit host capability." The `host_capability_adapter_allowlist.yaml` is mentioned, but no named test asserts the shims are gone by a specific caller sub-phase. Compared with the rest of the design — which pins every expiring helper to an `expiry_phase` and a blocking test — this is the one place where a process-global capability read can outlive its window without breaking the build.
- [Major] **`Recipe.GraphWorkflow` is a behavioral switch on a public mutable type.** `Recipe.GraphWorkflow` is correctly described as "a read-only projection," and `TestNoRecipeGraphWorkflowSynthesis` blocks lexical assignment outside `internal/formula`. But `Recipe` is otherwise passed freely across the molecule/dispatch boundary, the field is a plain exported Go bool (no unexported wrapper as `NormalizedRequirements` has), and `GraphWorkflow=true` is a genuine behavioral decision in execution code. Static guards catch direct assignment; they do not catch aliasing, struct copy-and-modify, or a future caller reading `Recipe.GraphWorkflow` as authority instead of going through the accepted-artifact accessor. The asymmetry with `NormalizedRequirements` is a deliberate downgrade in discipline and should be justified or fixed.
- [Minor] **Phase 4a is monolithic and exists alongside the legacy path for a long window.** Phase 4a delivers `CompileResult`, `AcceptedCompileArtifact`, the projection snapshot, every accepted-artifact writer signature, the shared workflow-root predicate, the repository-wide caller manifest, the report-mode allowlist, and the blocking guard for new raw consumers — while every existing producer still calls the legacy wrappers (line ~5163). For a lane whose mandate is keeping the boundary clean, that's a long period in which both paths coexist and the new path has zero exercised durable writers. The DAG (line ~5137) shows 4b–4e proceed in parallel after 4a, which is good, but a stronger invariant — "no producer sub-phase merges until at least one accepted-artifact zero-write fixture is green end-to-end in 4a" — would prevent the new APIs from accumulating untested signatures.
- [Minor] **Compose-traversal-vs-construct-identity invariant is not pinned as a generator test.** The rule that `contribution_path ∈ {compose_expand, compose_map, aspect, transitive_import}` rows must source their `construct_identity` from the *contributed template* (not the host formula) is stated in prose (lines ~860–867) and partly enforced by `TestConstructRegistryRowsCoveredByMatrix`. But there is no named fixture-or-test that asserts the negative case: a host formula with no v2-only construct and a contributed template with one must emit `formula.compiler_requirement_missing` *at the contributed source*. That's the single rule that, if violated, lets an aspect smuggle a v2 construct past requirement validation.

**Missing evidence:**
- The design references `internal/formula/testonly` as the package home for test-only constructors (`HostCapabilitiesForTest`, accepted-artifact builders, normalized-requirement builders) but does not say whether that package exists today or whether Phase 1 creates it. The "production cannot mint artifacts for tests" invariant depends on this boundary; without an explicit Phase-1 creation row, the testonly path is a hole.
- No reflection-backed inventory test is named that proves every exported field on `internal/formula.Formula`, `Step`, inline expansion, `compose.expand/map`, aspect advice, and import contribution is either in the construct registry, explicitly v1-compatible, or rejected as unsupported. Prose says "Generator self-tests fail closed" (line ~944), but the actual "added a new decoded field without a registry row" guard is not named.
- The initial allowlist/manifest files (`raw_consumer_allowlist.yaml`, `host_capability_adapter_allowlist.yaml`, `caller_paths.yaml`, `contribution_paths.yaml`, `coverage_intents.yaml`, `formula_projection_equivalence.yaml`) are all named but no Phase-1 PR deliverable commits an initial seed. The report-mode guard in Phase 1 has nothing to compare against until those files exist; the design should pin the seed commit explicitly.
- The convergence transition fence (lines ~5443–5451) requires shadow-compiling every selected formula through `CompileWithResult` while the legacy subset parser still owns writes during 4b–4f. It is fail-closed on shadow fatal diagnostics, but the design does not require *divergence* between shadow and legacy decisions to itself be fail-closed. A v2-only construct that the legacy parser silently accepts and the shadow result also accepts (because it is not yet treated as fatal in the matrix) would still write durable state.

**Required changes:**
- Collapse the public compile API to one canonical durable entry point. Either keep `PreflightFormula` and demote `CompileWithResult` to `// internal use only` (updating the convergence pseudocode and per-caller migration table accordingly), or commit to `CompileWithResult` + `AcceptCompileResult` and delete `PreflightFormula` from the public surface. Delete `Compile(...) (*Recipe, error)` entirely or move it behind a `testonly`/`compatpreview` build tag. Add `TestNoExternalCompileIdentityConstruction` that fails when production code outside `internal/formula` constructs `CompileIdentity`.
- Pin removal of the legacy host-flag globals to a named blocking test and an explicit phase. Example: `TestNoFormulaV2GlobalReadersAfterPhase4d` fails the build once caller sub-phase 4d is declared complete in the rollout artifact; the `host_capability_adapter_allowlist.yaml` rows for these globals must carry `expiry_phase: 4d` and the test enforces the expiry.
- Either unexport `Recipe.GraphWorkflow` (replace with `Recipe.GraphWorkflow()` accessor reading from an unexported field set only by the formula package) or add a sibling guard `TestNoExternalRecipeGraphWorkflowReadsForBehavior` that catches reads as authority. The current `TestNoRecipeGraphWorkflowSynthesis` only catches assignment.
- Add a named test that exercises the compose-traversal invariant negatively. Fixture: a host formula with no v2 construct + an aspect that contributes a `retry` table + no `[requires]` on either. Expected diagnostic: `formula.compiler_requirement_missing` at the aspect source, not at the host root. The generator must produce one such row per contribution-path × v2-construct pair.
- Require shadow-vs-legacy divergence inside the convergence transition fence to itself be a fail-closed event (named test: `TestConvergenceTransitionFenceNoNativeLegacyDrift`) so the fence cannot mask a bypass of `internal/formula`'s authority. Without this, the fence is only a partial safety net.
- Commit the initial seed for every allowlist/manifest file in the first implementation PR, even if empty with frozen `schema_version`. The Phase-1 acceptance evidence relies on report-mode comparisons that need a baseline.

**Questions:**
- If `CompileWithResult` remains exported, what is the test that prevents a new durable producer from calling it and `AcceptCompileResult` separately with its own `CompileIdentity` construction? The static guard list blocks bare `*Recipe`/`*CompileResult` as write authority but does not block caller-side identity construction, which is the same hole at a higher level.
- Why is `NormalizedRequirements` unexported-fields with accessors but `Recipe`, `CompiledStep`, and `CompiledConvergenceProjection` are not? The compiler-owned predicates (`Recipe.GraphWorkflow`, `CompiledStep.GraphCapability`) live on those structs. Is the intent that these are projection-only and the type system is enough, or is it a deliberate trade-off worth documenting?
- `internal/formula.RootMetadataFacts` and `internal/sourceworkflow.ClassifyWorkflowRoot` both classify metadata. The split (formula owns key meanings; sourceworkflow owns persistence query criteria) is sensible, but the design does not say which package extends first when a new `gc.formula_axis_*` key lands. Naming the order of operations would prevent parallel classifiers from growing for the same input.
- Is there a planned retirement of the `[daemon] formula_v2` boolean in favor of a typed capability config (`[daemon].formula_compiler_capability = 2`)? The edge adapter currently accepts two legacy spellings (`formula_v2` and `graph_workflows`); a long-term plan to consolidate to one typed key would simplify the host-capability surface and eliminate the deprecated-alias source kind.
