# Lena Driscoll - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- The phase DAG is explicit and well-bounded: Phases 1 and 2 are additive (revertible without source edits), 4a is a hard "writer lockdown" predecessor before any producer migration (lines 5199-5212), and Phases 8 and 9 are separated into distinct release gates with distinct rollback units (source conversion vs. parser alias removal). The "Phase | PR home | Owner | Required local command | Rollback unit" table (lines 5218-5229) gives every phase a named rollback control instead of leaving rollback as release-time judgment.
- Built-in pack compatibility is enforced as a cross-phase invariant rather than a per-phase reminder: "No first-party graph formula, example, tutorial, or fixture may drop `contract = "graph.v2"` while `docs/release/formula-compiler-min-floor.json` says `first_party_requires_only_allowed: false`" (lines 5128-5133). Combined with the `dual-declared-graph-v2` row in the compatibility corpus (line 3029) and the explicit per-entry-point matrix at lines 2625-2632, dual-declaration is locked from Phase 2 through the alias window.
- The migration-window exit is measurable, not narrative: alias-window-start is a JSON artifact with required fields (lines 5256-5283), the clock requires "two completed minor releases AND 60 calendar days" measured from a recorded release tag/date, alias-drain is a sliding window requiring two consecutive qualifying reports (lines 5287-5293, 3135-3142), and `--alias-removal-gate --json` has typed exit codes (0/1/2) with seeded conservative-blocking defaults (lines 2960-3001, 3152-3160).

**Critical risks:**
- **[Major] No invariant that a first-party `[pack] requires_gc` floor is <= the version that ships it.** Phase 3 lands "dormant" first-party floor declarations and Phase 7b enforces them at the resolver. The min-floor schema at lines 3071-3082 contains `"minimum_gc_for_requires_only": "0.x.y"` as a placeholder and CI rejects literal placeholders (lines 3103-3108), but there is no gate that compares the declared floor to the shipping release version. If Phase 7b enforces a floor of e.g. `>=0.7.0` while the resolver-enforcement release is `0.7.0`, a single packaging/version-drift bug ships a pack that rejects every operator on `0.6.x` who hasn't yet upgraded. This violates the spirit of the "incompatible packs are rejected before formula selection... durable writes" clause (5115-5118) by punishing operators rather than incompatible packs.
- **[Major] Phase 8 rollback cost is buried.** Lines 5249-5252 state "If Phase 8 rolls back after any requires-only source has been published, the rollback clock resets to the next release that republishes dual declarations and dual root stamps." This is correct, but the operational impact - rollback effectively delays alias removal by another two minor releases plus 60 calendar days - is only readable after cross-referencing the alias-window clock definition. A release captain skimming the "Rollback notes" (lines 5304-5318) does not see the multi-month cost. Without an explicit callout, Phase 8 will be rolled back optimistically under the assumption it can be retried in the next release.
- **[Major] Phase 2's "lands first-party dual declarations" deliverable is not gated on the dual-declared compat-corpus fixture passing on prior released binaries.** Line 5089-5090 lists "lands first-party dual declarations, and saves a green first-party inventory" as Phase 2 deliverables. The compatibility corpus at lines 3022-3033 includes `dual-declared-graph-v2` with required readers "latest two published Gas City minor releases, current `main`, `bd >= 1.0.0` probe", but the rollout plan does not explicitly require the dual-declared fixture pass on those prior released binaries *before* Phase 2 ships dual declarations into first-party packs. If a prior released binary's TOML decoder is strict about unknown top-level tables, adding `[requires]` to a first-party formula could regress that binary's ability to read the formula even though `contract` is still present. The fixture exists; the gate connecting Phase 2 ship-readiness to that fixture's green status is implicit.
- **[Minor] Sub-phase 4 docs partition vs. Phase 6 docs partition is unclear.** Line 5106-5111 says "no Phase 4 diagnostic surface may wait until Phase 6 for its required docs, help, schema, OpenAPI, dashboard, example, or stale-guidance artifact" - good. But Phase 6's scope is "stale live docs, generated CLI/config docs, architecture docs, PackV2 author docs, tutorials, examples, testdata, and dashboard/API generated types that were not tied to an earlier visible surface" (lines 5105-5108). That overlaps with surfaces 4d/4g already own. If a Phase 4 sub-phase regresses and is rolled back via feature gate, what guarantees Phase 6's general-purpose docs do not still reference the (now-disabled) diagnostic surface? The partition rule should be a typed manifest, not prose.
- **[Minor] Phase 4 mid-flight rollback for stateful producers (4f convergence/molecule/fanout) lacks an explicit "what continues, what pauses" playbook.** Line 5329 names the rollback unit as "reject new convergence/fanout formulas while legacy roots continue" - acceptable. But "legacy roots continue" is undefined when a root was created by Phase 4f after migration and carries an accepted-artifact ref. Dual-stamping at the writer covers most of this (line 1881 asserts canonical + legacy keys during alias window), but the design does not enumerate, per producer in 4f, what happens to in-flight convergence iterations whose parent root used the new artifact path when 4f is disabled. This is the highest-blast-radius rollback in the plan and deserves its own table.
- **[Minor] Parallel sub-phases 4b-4e share an implicit non-overlap requirement.** Lines 5154-5155 say "Sub-phases 4b through 4e may proceed in parallel only after 4a is merged and only if their manifest rows and write boundaries do not overlap." This is a hard correctness rule (concurrent writers on the same boundary could double-stamp or skip dual emission). There is no named CI test or build-time check that asserts no overlap; the caller manifest exists (lines 2015-2030, 2092-2127) but a "no two producer sub-phases own the same `durable_boundaries[]` entry concurrently" assertion is not specified.

**Missing evidence:**
- No fixture or release-gate proof that a first-party `[pack] requires_gc` floor declared in Phase 3 is satisfied by the binary that ships Phase 3 itself. The min-floor schema records `minimum_gc_for_requires_only` and `first_gc_with_requires`, but there's no `floor_is_satisfied_by_shipping_release` assertion.
- No "old binary at the prior published minor release reads a Phase-2 dual-declared first-party formula" fixture explicitly cited as a Phase 2 ship gate. The `dual-declared-graph-v2` fixture exists but is not tied to Phase 2's `--first-party-inventory` green-state gate.
- No rollback-replay test: if Phase 4 sub-phase X is rolled back (feature gate disabled), can the system still observe and clean up roots that sub-phase X created? The design references dual-stamping and the shared workflow-root predicate, but a checked-in "Phase 4x rollback replay" test (create root with X enabled, disable X, prove read/cleanup paths still work) is not named.
- The active-root report (lines 3119, 3122-3133) is required before Phase 8, but the design does not state how operators acquire an active-root report for a multi-controller, mixed-version fleet during the window before all controllers have shipped Phase 4a's shared predicate. The report exists per-store; aggregation across stores is not addressed.
- No explicit statement on what happens to the rollout if `engdocs/proposals/formula-migration.md` (the context document) is not superseded in Phase 2 as required by lines 5088-5089. Phase 2's ship gate references this update, but the consequence ("Phase 2 cannot ship until the proposal is marked superseded") is not stated as a CI assertion.

**Required changes:**
- Add a release-gate check that fails closed when `formula-compiler-min-floor.json` declares a `minimum_gc_for_requires_only` greater than the version of the release publishing the floor. Run it in Phase 3 and Phase 7b; fail Phase 8's `--requires-only-conversion-gate --json` when the floor binary is unshipped or unsupported.
- Add a "Rollback cost" callout to the "Rollback notes" block (around line 5304) for Phase 8 specifically, stating the alias-window-clock reset consequence in operator-facing terms (e.g., "Phase 8 rollback after publication delays alias removal by at least two completed minor releases plus 60 days from the next dual-republishing release"). Link to the alias-window-start clock anchor.
- Add an explicit ship gate to Phase 2 in the per-phase table (line 5221) requiring `gc formula validate --compat-corpus internal/formula/testdata/compat_corpus --json` to pass with `dual-declared-graph-v2` accepted by every reader in `formula-compiler-compatibility.yaml` `supported_readers`, before any first-party pack's `[requires]` table is merged. State this connection in the prose at 5088-5090.
- Add a "Phase 4 sub-phase rollback playbook" table that, for each of 4a-4g, names: (1) what feature flag/control disables the new producer surface, (2) which existing producers continue dual-stamping during the disabled period, (3) which roots created during the enabled period remain readable through the shared workflow-root predicate, (4) the zero-write fixture proving no protected writes occur while the surface is disabled. The information is scattered across lines 5202-5212 and 5320-5330; consolidate.
- Add a CI test `TestNoOverlappingProducerWriteBoundariesInParallelSubPhases` that consumes the caller manifest and asserts that no two simultaneously-merged Phase 4 sub-phases own a row whose `durable_boundaries[]` intersects. Name the test in the lockdown table at lines 5202-5208.
- Specify the docs partition between Phase 4 and Phase 6 as a manifest: every doc page is owned either by exactly one Phase 4 sub-phase or by Phase 6 cleanup. A rollback of a sub-phase reverts its owned docs together. Pages claimed by Phase 6 cannot reference a diagnostic surface owned by a not-yet-shipped Phase 4 sub-phase.
- Add an explicit invariant in the Rollout Plan that `engdocs/proposals/formula-migration.md` either is updated to align with the canonical design OR carries an `OBSOLETE: superseded by engdocs/design/formula-compiler-requirements.md` banner, gated by Phase 2 ship.

**Questions:**
- What is the intended behavior when a controller fleet is mid-Phase-4 rollout and the operators on the older binary cannot yet read accepted-artifact metadata? The shared workflow-root predicate handles legacy metadata reads, but does it handle reads of *new* artifacts written by Phase 4-shipped controllers? If so, that needs a fixture; if not, Phase 4 sub-phases require a fleet-wide upgrade ordering not stated in the plan.
- Is "two completed minor releases plus 60 days" the same clock for the alias window start AND for re-arming after Phase 8 rollback, or do they run independently? Line 5249-5252 implies the clock resets entirely; confirm whether the re-armed clock still requires the same evidence bundle (alias-window-start.json, compat artifact, external-support.json) to be re-saved with new release tags.
- Phase 7a (packman provenance) and Phase 7b (pack-floor enforcement) are listed as separate phases but share an owner (packman owner + release captain) and live in `internal/packman`. Is 7a/7b genuinely split across two PRs/releases, or is "7a/7b" a logical split within one release? The rollback unit for 7b is "disable resolver enforcement before formula selection" - if 7a and 7b ship in the same release, that rollback also disables 7a's provenance reads, which Phase 8 requires. Clarify.
- The "first-party dual-declaration guard" (line 2816) blocks releases where a first-party graph formula drops `contract` before the floor allows it. Does this guard run pre-merge (blocking the PR) or pre-release (blocking the tag)? Phase 3 introduces the floor but it is dormant; until Phase 7b, the floor cannot reject anything. What stops a contributor from removing `contract` from a first-party formula between Phase 3 and Phase 7b?
- Phase 8 source conversion edits N first-party pack files. Is the design's intent that all first-party `contract` removals land in a single PR (atomic conversion, single rollback unit), or is per-pack conversion allowed? The plan says "Phase 8... republish dual declarations and reset alias-window clock" for rollback - if conversion is multi-PR, partial rollback is possible and the alias-window-clock reset rule needs to specify whether any one pack's reversion to dual-declaration is sufficient or whether all packs must be dual again.
