# Priya Zielinski - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- The byte-exact v0 grammar is closed and fixture-locked: the accepted-shape matrix (`>=1`, `>=2`, omitted, empty table) plus the rejected-shape rows for `"2"`, `">= 2"`, `" >=2"`, `">=2 "`, `">=2.0"`, `">=2.1"`, `">=+2"`, `">=0"`, `">=-1"`, `">=01"`/`">=02"`, NUL/Unicode/control bytes, TOML integers, floats, bools, arrays, inline tables, dotted tables, and `[[requires]]` arrays of tables are enumerated with named diagnostic codes. Unsupported future minima (`>=3`, `>=4`, `>=99`, max-in-range ASCII) emit `formula.compiler_requirement_unsupported_future` rather than being accepted optimistically. Trimming, semver parsing, numeric coercion, Unicode normalization, and overflow wrap-around are explicitly forbidden, with `TestRawScannerPositionRoundTrip` and `FuzzRawRequirementScannerSourceAttribution` to lock the contract.
- The parser is genuinely two-stage: a `RawRequirementField` capture from original bytes runs before typed decoding, recording byte position, raw value, shape, and duplicate flag for `[requires]`, legacy `contract`, and legacy `version`. Diagnostics point at the raw field, never at a lossy zero value. `TestRawRequirementScannerMatchesTOMLMetadata` reconciles raw capture against `toml.MetaData` so the implementation cannot quietly drop position data, and `TestJSONRequirementScannerDuplicateMembersFatal` blocks last-writer-wins for JSON object members.
- V2-only construct enforcement is generator-driven, not prose: the construct registry is seeded from `workflow_control_metadata.yaml` and decoded field paths under `compiler_requirements_matrix.yaml`. CI count locks (`workflow_control_key_count`, `workflow_control_value_count`, `workflow_control_generator_count`, `workflow_control_callsite_count`) plus the reflection-backed `TestConstructRegistryRowsCoveredByMatrix` and `TestConstructRegistryMinimumMatchesCapability` make it impossible to add a `gc.*` workflow-control key or a new decoded field without registry + positive + missing-requirement rows. The three-state classifier (`authored_present` / `materialized` / `condition_disabled`) closes the lane-3 hole: a v2-only construct hidden behind `condition`, `if`, retry conditions, disabled steps, skipped compose branches, or order gates still requires `>=2`. The `caller-preflight` suite count of 18, derived from the per-occurrence caller manifest, asserts zero durable writes per durable boundary so missing-requirement diagnostics fire identically across `validate_only`, `show_preview`, `cook_root`, `cook_attach`, sling launch/attach, order dispatch, retry/on-complete, fanout parent/fragment, convergence create/retry/next, convoy scan, and dashboard preview.

**Critical risks:**
- **[Major] Two competing matrix-row schemas.** The doc defines one schema at lines 596-652 keyed by `dimensions` + `suites` with `count_lock`, and a second, more detailed schema at lines 1049-1079 keyed by `source`/`legacy`/`construct`/`caller`/`host`/`expect`. Both claim the same file `internal/formula/testdata/compiler_requirements_matrix.yaml` as normative. Implementers will fork: the first schema captures dimension vocabulary but doesn't have `expect.diagnostics[].source_path`/`line`/`column`; the second has the diagnostic projection but lacks the executable `count_lock`/`suites` and the `coverage_intent_id` keying. Without one canonical schema, the generator self-tests `TestCompilerRequirementMatrixSchema` and `TestCompilerRequirementMatrixGoldenDiagnostics` cannot both pass on the first PR.
- **[Major] `step.expand`/`expand_vars` vs compose traversal contradiction.** Construct identity rules at lines 540-552 say compose traversal (`compose.expand`, `compose.map`, `compose.aspects`) is v1-compatible until a contributed template contains a registered v2-only construct, and authored `Step.Expand` plus `expand_vars` are v2-only because they materialize subgraph topology directly. But the v2-only construct registry entry for `expand_vars` lists "same locations as authored step `expand`, plus compose expansion/map vars" — i.e. `expand_vars` is v2-only even when it lives on a compose entry whose traversal is v1-compatible. Line 866 then says a row with `contribution_path=compose_expand` plus `construct_identity=step_expand` is valid only when the contributed template contains an authored `Step.Expand`. The semantics for `expand_vars` on a compose entry without a contributed v2-only construct are not pinned. Pick one: either `expand_vars` is always v2-only regardless of carrier (matches registry), or it is v1-compatible on compose entries until a v2-only contributed construct appears (matches traversal rule). Today both rules coexist and matrix rows will diverge.
- **[Major] Misplaced-requirements coverage gap.** The `misplaced-requirements` suite lists `[steps.requires]`, `[compose.expand.requires]`, dotted `requires.formula_compiler` + bracket `[requires]`, and inline-table conflicts, but does not enumerate `[children.requires]`, `[loop.body.requires]`, `[compose.map.requires]`, `[compose.aspects.requires]`, or `[expand_vars.requires]`. The rule "non-top-level requirement tables never satisfy the root" is correct, but a v2-only formula whose only requirement table is `[steps.requires]` must emit `formula.compiler_requirement_missing` (not `requirement_invalid_type`) because the nested table does not satisfy the construct. The doc names this combined case for `[steps.requires]` but not for `children`, `loop.body`, or compose map/aspects. A fixture-locked combined-defect row for each nested location is required, otherwise an attacker (or careless author) can hide a `step.retry` under `[children.requires]` and produce ambiguous diagnostics.
- **[Major] Duplicate `[requires]` / dotted vs bracket conflict — undefined behavior leak.** Line 832 says "TOML parser-boundary collision when the parser rejects it, otherwise structured duplicate/collision diagnostic." BurntSushi/toml v1.6.0 actually does reject `[requires]` table redefinition and `requires.formula_compiler` dotted key colliding with a `[requires]` bracket table — but it returns the error as a generic decode error string, not a typed `KeyConflict` error. Without a frozen TOML parser version *and* a checked test asserting "v1.6.0 returns error X for fixture Y", the diagnostic class (parser-boundary vs structured) is implementation-defined. A future patch of BurntSushi/toml that changes the error wording or accepts a previously rejected form would silently move this from `parser-boundary` to `structured` (or back), without any visible matrix change. Pin `github.com/BurntSushi/toml@v1.6.0` in `go.mod` *and* assert it via a checked-in fixture, or replace the prose with a generated raw-shape row whose `diagnostic_code` is fixture-locked to the v1.6.0 output.
- **[Major] V1-compatible field manifest is not introduced.** The generator self-test at lines 944-952 fails closed when a decoded field is "not registered with `min_compiler_capability`, explicitly v1-compatible, or rejected as unsupported." The v2-only registry has a checked-in YAML, but the v1-compatible manifest has no file path. Without `internal/formula/testdata/v1_compatible_fields.yaml` (or equivalent), `TestEveryDecodedFormulaFieldClassified` will reject every existing field on first run. Either the manifest file needs a path here or the test contract needs a "seeded from current decoded types in the same PR" rule.
- **[Major] `>=N` parser semantics across binary versions are underspecified.** Line 1162-1167 says "The parser may recognize `>=<integer>` only to emit the machine-distinguishable `formula.compiler_requirement_unsupported_future` diagnostic for future minimums above the host binary's known maximum." But the matrix table at lines 270-275 unconditionally maps `">=3"` to `formula.compiler_requirement_unsupported_future` — implying the parser does not branch on the host's known max. The forward-compatibility section then says "Compiler capability numbers are stable once released and are never reused for different semantics. If a future capability such as `3` is introduced, older binaries reject `>=3` with `formula.compiler_requirement_unsupported_future`; newer binaries may accept it only after adding typed support." So the accepted v0 grammar is *closed by binary version*. The matrix needs a `host_known_max` dimension or an explicit invariant: "the accepted grammar is exactly omitted, `>=1`, `>=N` for every N where N ≤ binary's known max, and emits `_unsupported_future` for all other valid ASCII-integer forms." Today a reader could conclude `">=3"` will always emit `_unsupported_future` even after capability 3 ships.
- **[Minor] `internal/formula/testonly` has no build-tag guard.** Lines 1180 and 1486 say test-only constructors live in `internal/formula/testonly` or `_test.go` and "production constructors stay unexported." But there is no build-tag rule (`//go:build !production` or equivalent) and no static guard like `TestProductionDoesNotImportTestOnly`. A future PR that imports `internal/formula/testonly` from `cmd/gc` or `internal/api` would compile cleanly and bypass `acceptedCompileProof`. Add a static guard and a build tag, or move `testonly` under `internal/formula/internal/testonly` so the import path is unreachable from outside.
- **[Minor] `coverage_intent` vs `coverage_intent_id` drift.** Line 617 uses free-text `coverage_intent: "every durable writer has one fatal diagnostic row..."` inside `suites`. Lines 880-882 then introduce `internal/formula/testdata/coverage_intents.yaml` and say CI keys on `coverage_intent_id`. The two schemas need merging: either every suite carries both fields (intent text + checked id) or the free-text becomes a comment derived from the id. Reviewing a generator that flips between them will be painful.
- **[Minor] `formula_compiler = ">="` and overflow strings have ambiguous suite ownership.** Line 977 says `">="` (empty operand) or overflow-sized ASCII integers are "negative or unsupported according to parser boundary; never accepted; diagnostic code and parser boundary are fixture-locked." But the `grammar` and `raw-shape` suite ownership is split: grammar owns accepted/rejected byte strings, raw-shape owns wrong-decoded shapes. `">="` is a valid TOML string with the wrong byte content — that belongs in `grammar` as a negative row, not in `raw-shape`. Lock this explicitly so the generator does not emit duplicate rows in both suites.

**Missing evidence:**
- The fixture files the design depends on do not yet exist in the tree: `internal/formula/testdata/compiler_requirements_matrix.yaml`, `internal/formula/testdata/workflow_control_metadata.yaml`, `internal/formula/testdata/caller_paths.yaml`, `internal/formula/testdata/contribution_paths.yaml`, `internal/formula/testdata/coverage_intents.yaml`, `internal/formula/testdata/host_capability_adapter_allowlist.yaml`, `internal/formula/testdata/raw_consumer_allowlist.yaml`, and the construct-registry release JSON `docs/release/formula-compiler-construct-registry.json`. Phase 1 cannot start until at least the matrix, workflow-control, caller-paths, and v1-compatible-fields fixtures are seeded.
- No worked golden example for the `condition_disabled` row family. The doc says condition-disabled constructs still require `>=2` because old compilers may fail to parse, but there is no concrete fixture showing a `[[steps]] check = {...} condition = "{{x}} == 'off'"` row evaluating to `formula.compiler_requirement_missing` with the construct source attribution preserved. This is the load-bearing test for the three-state classifier and it's described only in prose.
- No fixture asserting that an `[steps.retry]` decoded by BurntSushi/toml as a *partial* table (e.g. malformed numeric field) is rejected with `requirement_invalid_type` *before* construct-registry detection runs. The decoded-path predicate at lines 481-495 assumes a clean decode. If a malformed retry table decodes into a zero-value struct, construct detection will fire on a phantom `step.retry`.
- The `legacy-check-compatibility` suite is named at line 1046 but its row vocabulary is not specified. What concrete `check` shapes from old public examples must round-trip? Without that, the suite is a placeholder.

**Required changes:**
1. Collapse the two matrix-row schemas (lines 596-652 vs 1049-1079) into one normative schema in `internal/formula/testdata/compiler_requirements_matrix.yaml`. Pick the detailed `source`/`legacy`/`construct`/`caller`/`host`/`expect` schema, add `suite_id` + `count_lock` + `coverage_intent_id`, and delete the older shape from the design doc in the same change.
2. Pin `github.com/BurntSushi/toml v1.6.0` in `go.mod` and replace every "TOML parser-boundary collision" prose row with a fixture-locked `raw-shape` row whose `diagnostic_code`, `source_line`, `source_column`, and exact error wording come from the pinned parser version. Add `TestTOMLDuplicateTableRejected`, `TestTOMLDottedBracketCollisionRejected`, and `TestTOMLArrayOfTablesRequiresRejected` driven by checked-in fixtures.
3. Resolve `expand_vars` on compose entries: either edit the construct registry so `expand_vars` is v2-only only when carried by an authored `Step.Expand` (drop the "plus compose expansion/map vars" clause), or edit the compose-traversal rule so any decoded `expand_vars` field on a compose entry promotes the contributed template to v2-only. Update lines 322-332, 540-552, and 860-867 in lockstep, and add fixtures for `compose.expand[].vars` + no v2-only template content covering the chosen outcome.
4. Add `misplaced-requirements` matrix rows for `[children.requires]`, `[loop.body.requires]`, `[compose.map.requires]`, `[compose.aspects.requires]`, and `[expand_vars.requires]`, each paired with at least one positive (`condition_disabled` v2-only construct hidden beneath) row that asserts `formula.compiler_requirement_missing` rather than `requirement_invalid_type`.
5. Introduce `internal/formula/testdata/v1_compatible_fields.yaml` as a checked-in companion to `workflow_control_metadata.yaml` and the v2-only construct registry. Seed it from the current decoded types in the same PR that lands the reflection guard `TestEveryDecodedFormulaFieldClassified`.
6. Pin the `>=N` parser invariant: add an explicit statement that "the accepted grammar is exactly omitted, `>=1`, and `>=N` for every N where 1 ≤ N ≤ binary's known maximum capability; every other ASCII-integer form `>=M` with M > known max emits `formula.compiler_requirement_unsupported_future`." Add `future-boundary` rows that cross `host_known_max ∈ {2, 3}` with `requirement_value ∈ {">=2", ">=3", ">=4"}` so the generator captures the "known max grows with binary" axis.
7. Move `internal/formula/testonly` under `internal/formula/internal/testonly` (Go enforces the import path is unreachable) and add `TestProductionDoesNotImportTestOnly` against `cmd/gc`, `internal/api`, `internal/sling`, `internal/dispatch`, `internal/molecule`, `internal/convergence`, `internal/order`, and `internal/sourceworkflow`.
8. Add a fixture proving that BurntSushi/toml partial-decoded malformed `[steps.retry]` is rejected as `requirement_invalid_type` (or the equivalent step-level diagnostic) *before* the v2-only construct-registry detector runs. Otherwise the detector can fire on phantom decoded fields.

**Questions:**
- For the `coverage_intent_id` schema: should `coverage_intent_id` and free-text `coverage_intent` both be required per row (id + reviewer-readable annotation), or does the id alone suffice with intent text rendered into generated tests only?
- For `[requires.formula_compiler]` as a dotted/nested table: does BurntSushi/toml v1.6.0 actually return a typed key-conflict error, or only a generic decode error? The matrix row claims `formula.requirement_invalid_type` with `diagnostic_count=1`, but the raw scanner can produce a position-attributed `requirement_invalid_type` only if it sees the dotted/nested shape before the parser bails. Confirm whether the raw scanner reads the bytes ahead of typed decoding for the `[requires.*]` family specifically.
- The `legacy-check-compatibility` suite is named but its row vocabulary is not enumerated. What is the canonical list of public `check` syntax variants from existing released documentation that must round-trip? Without that inventory, alias-window release evidence cannot be measured.
- For the `caller-preflight` count lock of `18`: when sub-phase 4f adds operator-invoked `gc formula repair-root-artifact`, does the repair path become caller row 19, or does it reuse one of the existing rows (e.g. `controller_validation`)? Locking this now prevents a future PR from accidentally shrinking the count by merging two rows.
