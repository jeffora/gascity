# Yuki Patel

**Persona verdict:** block

**Sources:** Claude, Codex

**Consensus findings:**
- **Blocker:** The caller-facing compile contract is still not singular. Codex flags two incompatible `CompileResult` shapes, while Claude independently finds the same drift in the in-flight/convergence section. Callers cannot safely migrate if convergence, fanout, retry, API preview, and durable writers disagree on whether `Steps` and `RuntimeVars` live directly on `CompileResult`, under `Projection`, or behind an accepted-artifact API.
- **Blocker:** Workflow-root predicate consolidation lacks a behavior-change ledger. Both reviews identify divergent current semantics across sling, sourceworkflow, graphroute, API order feeds, convoy dispatch, and SQL paths: exact vs trimmed, case-folded vs case-sensitive, `kind && contract`, `kind || contract`, and `kind`-only. The design must name which call sites preserve behavior, which intentionally narrow or widen behavior, and which `WorkflowRootKind` or compatibility wrapper each caller uses.
- **Major:** The sourceworkflow/API migration plan is not concrete enough for caller integration. Codex requires a before/after table for `IsWorkflowRoot`, `ListLiveRoots`, `WorkflowMatchesSource`, local API predicates, convoy dispatch, and order/feed projections. Claude requires the generated caller manifest to reconcile cited symbol names against the live tree and to distinguish net-new APIs from existing compatibility helpers.
- **Major:** Fanout and convergence in-flight semantics can still diverge after a host capability downgrade. Claude finds that accepted-projection-snapshot reuse and current-host fragment recompilation give opposite answers for not-yet-materialized fanout. The design must choose the rule for same-identity roots, changed fragments, inline expansion templates, and separately resolved `compose.expand` formulas, then put the fanout rule next to the convergence rule.
- **Major:** Transitional in-flight behavior is underspecified for rollout phases before accepted artifacts and `gc formula repair-root-artifact` exist. A v2 root created before those mechanisms can still stall if `[daemon] formula_v2` is disabled mid-expansion. The design either needs an explicit Phase 1 through 4e caveat or must move repair support earlier.
- **Major:** The prompt/template and first-party producer inventory needs deterministic binding. Both reviews want prompt, tutorial, example, and formula producer references such as `bd mol`, `gc bd mol`, `mol wisp`, `mol bond`, `mol cook`, and `mol cook-on` to be mechanically classified. Codex adds that external or materialized workflow-pack prompts need a pinned source, lockfile, or checked-in fixture contract instead of relying on ignored local runtime state.
- **Major:** The raw-consumer allowlist and caller manifest need enforceable expiry mechanics. Claude notes that `TestRawConsumerAllowlistExpiry` has no concrete rollout-phase state to read and that package boundaries do not align with sub-phase ownership. Codex asks for a checked-in manifest excerpt covering API metadata queries, sourceworkflow predicates, fanout compilation, prompt producers, and validation-only `bd` probes.
- **Major:** Stale public guidance can reintroduce removed caller paths. Claude says the native-only runtime decision is structurally right but should be stated plainly up front. Codex identifies `engdocs/architecture/formulas.md` and `docs/reference/formula.md` as still describing the old `bd`/`Store.MolCook*` production path and requiring explicit update or supersession.

**Disagreements:**
- Claude's verdict is `approve-with-risks`; Codex's verdict is `block`. Assessment: block. The remaining issues include contradictory public caller contracts and workflow-root migration semantics, which must be resolved before downstream caller work can proceed safely.
- Claude says the live `internal/sourceworkflow` package lacks `IsWorkflowRoot` and `ListLiveRoots`; Codex says those helpers exist and currently case-fold/trim an OR of `gc.kind=workflow` and `gc.formula_contract=graph.v2`. A narrow local check confirms Codex's current-tree observation, so the synthesis treats Claude's point as a stale-symbol/inventory problem: the design still must mark net-new APIs clearly and keep current compatibility helpers explicit.
- Claude emphasizes fanout host-downgrade ambiguity as a major unresolved risk; Codex describes the in-flight/fanout section as much more executable than earlier versions. Assessment: the improvement is real, but the unanswered same-identity fanout vs current-host recompile rule is still caller-blocking.
- Claude treats the `bd` runtime fallback as already removed and validation-only; Codex focuses on stale docs and prompt/template sources that can reintroduce `bd` assumptions. Assessment: these are compatible. The design should state native-only production once, then update or supersede stale docs and classify producer references mechanically.

**Missing evidence:**
- One authoritative `CompileResult` and projection snippet used by convergence, fanout, retry, API preview, durable writers, and accepted-artifact repair.
- A per-call-site workflow-root reconciliation ledger covering sling launch, sling attachment, sourceworkflow, graphroute, API order/feed paths, convoy dispatch, SQL queries, and source-scoped closed-history checks.
- Golden fixtures for exact canonical metadata, whitespace-only legacy compatibility, case-folded legacy roots, graph.v2-only roots, kind-only roots, and `kind || contract` attachment behavior, with an explicit decision for each behavior change.
- Zero-write fixtures for host-disabled behavior across fanout, retry/`on_complete`, attach, order scans, convergence create/retry, and root-only wisps, including pre-artifact and artifact-stamped roots.
- A checked-in caller-manifest excerpt that covers at least one API direct metadata query, one sourceworkflow predicate, one fanout fragment compile, one prompt/template producer, and one validation-only `bd` probe.
- A pinned source, lockfile, or deterministic fixture-generation contract for external-but-first-party workflow pack prompts and formulas.
- Concrete rollout-phase state for `TestRawConsumerAllowlistExpiry`, plus exact owning sub-phase tags for each manifest or allowlist row.

**Required changes:**
- Define `CompileResult` exactly once. If convergence needs `Steps` and `RuntimeVars`, put them in the canonical projection path or add them to the canonical type and update every consumer contract to use that same path.
- Resolve the sourceworkflow API plan: either keep `IsWorkflowRoot`, `ListLiveRoots`, and related helpers as named compatibility wrappers over new `WorkflowRootFacts`, or replace them with new criteria APIs. In either case, specify wrapper semantics, expiry phase, tests, and the before/after mapping for each caller.
- Add the workflow-root predicate behavior ledger, including exact/trim/case-fold behavior, `kind` only, `contract` only, `kind && contract`, `kind || contract`, graph-only vs any-workflow distinctions, and source-scoped closed-history queries.
- State the fanout parent rule in one place: whether same-identity existing roots load the accepted projection snapshot or recompile against the current host, how changed fragment formulas are handled, and how inline expansion templates differ from separately resolved `compose.expand` formulas.
- Add a "transitional in-flight behavior, Phases 1 through 4e" subsection for legacy v2 roots without accepted artifacts, or move `gc formula repair-root-artifact` early enough to give operators a remedy before those paths depend on accepted artifacts.
- Make the generated caller manifest the single normative per-occurrence source. Include explicit seed literals for `bd mol`, `gc bd mol`, `mol wisp`, `mol bond`, `mol cook`, and `mol cook-on`, and reconcile all cited Go symbols against the current tree.
- Define deterministic scanning for first-party workflow-pack prompts and formulas through tracked sources, a pinned source revision, or checked-in generated fixtures. Do not make ignored local runtime state a mandatory CI input.
- Specify how allowlist expiry reads rollout-phase state, replace broad file/package allowances with exact file plus symbol or line-pattern constraints, and tag each row with its owning sub-phase.
- State up front that the `bd` shell-out runtime path is already removed, then update or mark superseded `engdocs/architecture/formulas.md`, `docs/reference/formula.md`, and any other docs that still describe `Store.MolCook*` or `bd` as production caller paths.
