# Marta Hidalgo - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- One `Diagnostic` / `FormulaDiagnostic` shape, projected uniformly to CLI stderr, Huma response bodies, dashboard state, and registered event payloads, with a fixture-driven parity contract (`internal/formula/testdata/diagnostics/`) whose golden tests compare direct CLI, API-routed CLI, Huma bodies, generated TS, dashboard state, and event payloads field-for-field (`Code`, canonical message, remediation, source path/key/value, normalized requirement, host capability, severity, `OnceKey`). This is the right structural answer to "dashboard and CLI give contradictory remedies" — the wording lives in one struct and every surface is asserted against it.
- The host gate is applied to the *normalized* requirement (`NormalizedRequirements.FormulaCompiler()` vs `HostCapabilities`); legacy `contract = "graph.v2"` is normalized to `>=2` *before* the host check (diagnostic-order step 5 then step 7), and the "Forward Compatibility" guardrail adds a CI guard forbidding runtime branching on `RequirementSource`. So `requires`-source and `contract`-source formulas reach the gate identically. The gate also fails *before any durable write* at every entry point ("Durable preflight contract" table; "no beads created"; convergence/fanout zero-write rows), so a disabled host never leaves a half-built molecule for an operator to clean up.
- Disabled capability gets its own code (`formula.compiler_requirement_unsatisfied`) with explicit remediation, and on the dashboard it renders as a distinct "disabled-capability state with remediation" rather than the generic "formula preview validation error" used for parse-class diagnostics. `version_deprecated` is never emitted on runtime paths (launch/order/retry/convergence/controller), only on validate/show/preview — exactly the right call for keeping operational logs clean.

**Critical risks:**
- **[Major] The background-suppression cache can re-flood deprecation warnings, and "config generation" — the linchpin of the dedup contract — is undefined.** Per "Host Capability And Diagnostics", background suppression is "an in-memory LRU per city and producer, capped at 4096 keys with oldest-entry eviction" that holds *both* `OnceKey`-suppressed warnings *and* fatal-diagnostic grouping entries — and the fatal entries are keyed by `(producer, subject id, OnceKey, host capability, config generation)`, which has far higher cardinality because it includes `subject id` (order id, convergence id, fragment target). A burst of distinct fatal diagnostics — e.g., `daemon.formula_v2` flipped off while thousands of graph workflows / formula-backed orders are live — can evict the warning-suppression keys, so the next compile of a legacy-`contract` formula re-emits `formula.contract_deprecated` on every poll, precisely when operators are most flooded. Separately, "config generation" is used as a grouping/dedup dimension in at least five places (the order-event row, the `OnceKey`/grouping paragraph, the registered-event dedup key, the repeated-loop tests, and the dashboard grouping rule) but is never defined; if an implementer makes it a per-reload or per-tick counter, fatal grouping resets every controller tick and the rate-limiting is gone. Lane red flag 3 ("deprecation warnings fire on every resolve cycle"), directly.
- **[Major] Launch commands collapse formula/host diagnostics into the generic failure exit code.** "CLI exit code mapping is fixed: `0` … `1` for process/internal I/O failures, and `2` for formula diagnostics on validation/report commands. Runtime launch commands such as `gc sling --formula` and `gc order` use exit `1` for a failed operation …". So a script driving `gc sling` in CI cannot tell "the operator needs to enable `[daemon] formula_v2`" from "the bead store is unreachable" — both are exit `1` — without parsing stderr. The mandate is "actionable errors" and "API/CLI/order consistency"; an exit-code contract that is uniform on validation commands (`2` == formula diagnostic) but lossy on launch commands is the wrong default and is exactly the kind of cross-surface inconsistency the lane is meant to catch.
- **[Major] "Increment a grouped occurrence counter on … the next emitted event" reads as mutating an emitted event, which conflicts with the append-only Event Bus invariant.** Order dispatch is specified to emit "at most one typed order failure event per `(order id, formula, OnceKey, host capability, config generation)` in a scan series" and "repeated scans increment a grouped occurrence counter on the producer state or next emitted event", and the repeated-loop test asserts "one grouped `order.formula_compile_failed`, occurrence count `3`" after three scans. The Event Bus is append-only (AGENTS.md: "append-only pub/sub log"), so the count on an already-emitted event cannot be bumped. The doc must say the authoritative count lives in producer state and is surfaced via a read-model join (or that a fresh event is emitted only when the count crosses thresholds), and add the corresponding test — otherwise an implementer either violates the event invariant or emits one event per scan, defeating the bound.

**Missing evidence:**
- The definition of "config generation" — is it a content hash of the formula-relevant config slice (`daemon.formula_v2`, formula layers, pack pins) or a coarser reload/tick counter? It is load-bearing for every dedup/grouping rule in the design.
- The cardinality bound for deprecation-warning `OnceKey`s (≈ legacy-formula-count × host-capability-count, host-capability ≤ 2 in v0) and confirmation of whether warnings and fatal grouping really share the single 4096-key per-(city, producer) LRU. Without this the 4096 cap is unjustified and the eviction-churn risk above is unbounded.
- A fixture asserting byte-identical projection (code, canonical message, remediation, CLI exit, HTTP status, dashboard group title, event payload code, host source attribution, `OnceKey`) between a host-disabled `requires`-source formula, a host-disabled `contract`-only formula, and a `dual`-source one. The shown `disabled-v2-host` fixture covers only the `requires` source — so lane question 2 ("does the gate apply regardless of `requires` vs `contract`") is contract-by-prose, not contract-by-test.
- A "many distinct subject ids, host disabled" fixture (the LRU-thrash scenario), a process-restart fixture, and a config-flap (off → on → off) fixture — the repeated-loop test table covers config reload but not restart, not flap, and not the high-subject-cardinality eviction case.
- The exact canonical message/remediation strings as normative constants. The doc shows examples and at least two phrasings of the `unsatisfied` remedy: "enable `[daemon] formula_v2` or choose a v1 formula" in the projection matrix / parity fixture, vs "enable `[daemon] formula_v2` or use a formula that does not require compiler v2" in the Decision section.
- An operator-facing read surface for grouped failure state and "how much was hidden by cadence/eviction". The eviction counter is "a producer-local counter and debug log field" only; there is no dashboard/API field for events-emitted-vs-suppressed and no `gc … status`-style command for a dashboard-less operator to see the grouped rows synchronously.
- (Baseline note: the iteration's snapshot `design-before.md` for attempt-94 contains `engdocs/proposals/formula-migration.md` — a context file — not `formula-compiler-requirements.md`. I reviewed the most recent valid snapshot of the design doc, the 2166-line "Formula Compiler Requirements v0" (attempt-92/attempt-93 `design-before.md`). If synthesis diffs against attempt-94's snapshot, the comparison baseline is wrong.)

**Required changes:**
- Define "config generation" precisely in the "Host Capability And Diagnostics" section: a hash/version that changes *only* when formula-relevant config changes (`daemon.formula_v2`, formula layers, pack pins), never a per-reload or per-tick counter. State it once and reference that definition everywhere it is used as a grouping dimension.
- Separate the warning-suppression store from the fatal-grouping store, or cap them independently, or add a fixture proving that a burst of distinct fatal diagnostics cannot evict warning-suppression keys under the shared 4096-key LRU. Add a "many distinct subject ids, host disabled" fixture asserting `formula.contract_deprecated` is still emitted at most once per `(code, source path, source key, host capability, config generation)` while the fatal failures churn.
- Give `gc sling --formula` and `gc order` exit code `2` (matching validation/report commands) when the failure is a formula/host diagnostic, reserving exit `1` for process/I/O failures — or document explicitly, in both the design and the generated CLI help, why launch commands collapse them and that automation must read the typed body rather than `$?`.
- Resolve the occurrence-counter / append-only-event tension: state that emitted event payloads are immutable, that the authoritative occurrence count lives in producer state and is surfaced to the dashboard/API via a read-model join (or that a fresh event is emitted only on threshold crossings), and add the test that asserts the Event Bus does not gain one entry per scan.
- Pin the canonical messages and remediations as constants/fixtures referenced from every projection; remove the second phrasing of the `unsatisfied` remedy so there is one byte-exact string.
- Add the cross-source parity fixtures (`requires`-source vs `contract`-only vs `dual`, host disabled) plus the process-restart and config-flap fixtures to `internal/formula/testdata/diagnostics/` and the repeated-loop tests table.

**Questions:**
- Is "config generation" a content hash of the formula-relevant config slice, or a coarser counter? What exactly bumps it, and does a no-op `city.toml` touch bump it?
- Do warning suppression and fatal grouping share the 4096-key per-(city, producer) LRU? If yes, what prevents fatal-diagnostic churn (keyed including `subject id`) from evicting the warning-suppression keys and re-flooding deprecation warnings?
- On launch endpoints, why HTTP `409` for `formula.compiler_requirement_unsatisfied` while preview returns `400` for the identical cause? Is `409` chosen to align with other launch conflicts, or should the host-disabled case be `422` consistently on both surfaces?
- For `gc sling` / `gc order`, is exit `1` for a formula-diagnostic failure intentional, given validation commands use exit `2`? If intentional, where is that documented for operators writing automation around these commands?
- Is a dashboard-less operator expected to read grouped failure state and suppression/eviction counts from somewhere (a `gc` command, an API endpoint), or only from debug logs and the Event Bus?
