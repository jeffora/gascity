# Priya Zielinski

**Persona verdict:** approve-with-risks

**Sources:** Claude, Codex

**Consensus findings:**
- [Major] Raw requirement extraction is not yet an executable parser contract. Both reviews flag that `RawRequirementField` requires source text, TOML/JSON shape, line, and column before typed decoding, while the selected BurntSushi/toml path does not provide stable per-key positions or source slices for valid decoded fields. Without a named scanner, AST parser, pre-pass, or equivalent API, diagnostics can degrade into lossy decoded zero values while the prose still appears satisfied.
- [Major] Parser edge cases are named but not pinned to exact diagnostic rows. The design needs explicit row IDs, diagnostic codes, source keys, escaped source values, and ordering for signed values, `>=0`, `>=-1`, `>=+2`, leading-zero values, overflow-sized integers, tab/newline whitespace, Unicode whitespace, look-alike operators, full-width digits, NUL/control strings, inline tables, nested/dotted tables, and duplicate-key/table boundaries.
- [Major] JSON parity and scope are under-specified. Claude notes that the matrix schema is TOML-shaped while prose mentions JSON null/object/array/duplicate-member cases; Codex adds that Go's standard JSON decoder will not detect duplicate object members by default. The design must either make JSON formulas explicitly out of scope for v0 enforcement or add a source-format-specific JSON shape axis and a duplicate-detecting raw pass.
- [Major] Durable-writer caller coverage is not mechanically tied to the full writer inventory. Codex flags that `caller_path` can drift from retry continuations, `on_complete`, Ralph continuation, workflow-control appenders, active-root repair, fanout repair, missing-child repair, convergence retry, and molecule-instantiation writers. Claude sees the durable preflight design as strong, but the matrix still needs generated manifest linkage so new writers cannot bypass `CompileWithResult -> AcceptCompileResult -> ValidateAcceptedArtifact`.
- [Major] Count-lock governance needs stronger mechanical guarantees. Codex says several normative suites do not visibly carry concrete lock values or generated expressions; Claude says existing count locks still allow asymmetric shrinkage, such as deleting a negative row while adding a positive row. Locks should be required for every suite and split or generated by row kind so coverage loss cannot hide behind a stable total count.
- [Minor] Contract and TOML spelling boundaries need explicit classification. Claude calls out empty, whitespace-padded, case-folded, and future `contract` values; Codex calls out legal TOML dotted-key spelling such as `requires.formula_compiler = ">=2"`, quoted keys, and case-sensitive key spelling. These should be accepted or rejected by fixture, not left to implementation inference.
- [Minor] Disabled-by-condition v2 constructs are correct but need pinned fixtures. Claude agrees with requiring `>=2` when a v2-only field is present even if a condition disables the step, but that behavior reverses an intuitive author expectation and should be represented by explicit condition/no-condition row pairs.

**Disagreements:**
- There is no verdict disagreement: both reviewers recommend `approve-with-risks`.
- Claude treats the durable preflight architecture as a strength, while Codex sees the caller-path matrix as a drift risk. Assessment: the architecture direction is acceptable, but approval depends on adding a generated caller manifest or equivalent CI guard tying the matrix to every durable writer and active-root mutator.
- Claude indicates normative suite count locks exist but need positive/negative/unsupported/impossible sub-counts; Codex reports that only some visible suites have concrete lock values. Assessment: both point to the same control failure. The final design should require every suite to declare either concrete count locks or deterministic generated-count expressions, with row-kind protection.
- For overflow and malformed integer-looking requirements, Claude asks the design to choose between invalid syntax and unsupported future; Codex recommends reserving `formula.compiler_requirement_unsupported_future` for byte-exact ASCII `>=N` where `N` is greater than the host-supported maximum, with malformed values as invalid syntax. Assessment: adopt Codex's narrower rule unless the design explicitly documents a different monotonic future-capability policy.
- JSON treatment is framed differently: Claude wants either JSON shapes in the schema or JSON declared out of scope; Codex questions whether JSON support is first-class and whether duplicate-member detection is required. Assessment: not contradictory, but the design must make one binding decision before implementation.

**Missing evidence:**
- A concrete raw-source extraction strategy for BurntSushi/toml and JSON inputs, including how valid scalar fields receive source text and line/column, how parser-level duplicate key/table errors remain outside formula diagnostics, and how typed decoding is prevented from running before raw requirement capture.
- Exact matrix rows and golden diagnostics for malformed requirement strings, overflow-sized ASCII integers, TOML inline/nested/dotted table shapes, dotted-key spelling, duplicate scalar keys, duplicate tables, quoted/case-variant keys, escaped NUL/control strings, and contract-value variants.
- A JSON v0 scope decision, plus either JSON `null`/array/object/nested-object/duplicate-member fixtures with JSON pointer positions or an explicit statement that JSON formulas are out of scope for this change.
- A generated `caller_paths.yaml` or equivalent inventory proving that every durable writer, continuation launcher, active-root mutator, fanout/convergence repair path, and metadata-only caller is represented in the preflight matrix or deliberately marked impossible.
- Required count-lock fields or generated-count expressions for `grammar`, `raw-shape`, `legacy-alias`, `construct-registry`, `contribution-traversal`, `caller-preflight`, and `projection-parity`, including row-kind sub-counts.
- The scope of the reflection-backed completeness test: the root decoded types it walks, nested construct types it follows, and the handling of `map[string]string` metadata fields against `workflow_control_metadata.yaml`.
- An authoritative `internal/formula/testdata/` index naming every YAML fixture file, its schema, consuming tests, and the locks it controls.

**Required changes:**
- Specify the raw-source extraction implementation as an API contract: chosen TOML scanner/parser or pre-pass, JSON raw parser behavior, returned field model, source-position guarantees, fallback if positions cannot be recovered, duplicate parse-error boundary, and tests proving typed decoding cannot bypass raw capture.
- Expand the grammar/raw-shape matrix with exact row IDs and expected diagnostics for signed values, zero/lower-than-supported values, leading zeros, overflow, ASCII and Unicode whitespace, Unicode operators/digits, NUL/control strings, inline tables, nested/dotted tables, duplicate scalar/table errors, dotted-key spelling, quoted keys, and case variants.
- Pin `contract` compatibility rows for empty string, whitespace-padded `graph.v2`, case-folded `GRAPH.V2`, and future `graph.v3`, and state whether trimming or case-folding is ever permitted for the contract field itself.
- Decide JSON scope. If JSON remains supported, add a `json_shape` or source-format-conditional shape axis covering null, array, object, nested object, and duplicate member behavior with JSON pointer diagnostics; if not, mark JSON formula inputs out of scope for v0 enforcement.
- Tie `caller_path` to a generated durable-writer manifest or equivalent CI guard, and require fatal-diagnostic zero-write rows for retry, scope-check, fanout, `on_complete`, Ralph continuation, active-root repair, missing-child repair, convergence retry, order dispatch, and any metadata-only impossible paths.
- Require count locks or generated-count expressions for every normative suite, split by positive, negative, unsupported, and impossible row kinds, and make the generator fail on missing locks, undeclared dimension values, or dimension growth without an intentional lock update.
- Add disabled-by-condition construct-registry fixtures proving a v2-only field demands `>=2` whether or not the enclosing step condition can run.
- Name and scope the reflection completeness test, including the decoded types it walks (`Formula`, `Step`, `Compose`, `Advice`, `Aspect`, `Import`, expansion templates, loop bodies, and nested construct types) and how metadata maps are matched against the workflow-control registry.
- Add or generate `internal/formula/testdata/README.md` listing each fixture file, schema version, consuming test functions, and associated count locks.
