# Avery Brooks - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- The Terminology glossary (lines 636-648) explicitly distinguishes formula compiler capability, host capability, compiler implementation, `contract`, `schema`, pack revision, formula `version`, pack `requires`/`requires_gc`, and graph workflow. It also calls the formula-vs-pack `requires` overload out by name — directly answering the persona's lane question on glossary distinctions.
- The Diagnostic Projection Matrix (lines 525-535) gives operators one stable wording across CLI, API, dashboard, and events with user-oriented remediation strings (`use [requires] formula_compiler = ">=2"`). That preserves source spelling and remediation across surfaces, which is the only way the deprecation story stays honest as it propagates.
- The Terminology And Documentation Contract section (lines 631-661) commits docs/reference, architecture, tutorials, examples, test fixtures, and dashboard to the *same release phase* as user-visible diagnostics. The intent is exactly right: docs and diagnostics ship together.

**Critical risks:**

- [Major] **The Rollout Plan contradicts the same-release-phase doc contract.** Phase 5 (lines 686-689) says "after parser and caller support ships, update `docs/reference/formula.md`, architecture docs, tutorials, examples..." But line 650 says "Docs and examples must be updated in the same release phase that makes diagnostics user-visible." Phase 3 (lines 681-682) is when user-visible diagnostics start emitting (`formula.contract_deprecated`, `formula.compiler_requirement_unsatisfied`, `formula.version_deprecated`). If Phase 5 trails Phase 3 by even one release, operators will see deprecation warnings while `docs/reference/formula.md` still teaches `version = 1` and never mentions `[requires]`. This is the persona's red flag #1 (docs continue recommending version/contract after behavior changes), and the design contradicts itself on it.
- [Major] **`docs/reference/formula.md` is currently a counter-example to the design.** Today's file teaches `version = 1` as part of the minimal example (line 23) and lists `version` as a canonical Common Top-Level Key (line 48) with no deprecation note. It never mentions `[requires]`, `contract`, `formula_compiler`, capability, or graph workflows. The design defines `formula.version_deprecated` (line 521) and projects it to validate/show, but does not call out that the reference doc is *currently teaching the deprecated surface*. The doc flip must be coupled to Phase 3, not deferred.
- [Major] **First-party examples are 100% legacy-only and the design does not show what dual-declared looks like.** Five in-repo formulas (`examples/dolt/formulas/mol-dog-compactor.toml`, `examples/gastown/packs/gastown/formulas/mol-{digest-generate,idea-to-plan,refinery-patrol}.toml`, `examples/gastown/packs/maintenance/formulas/mol-dog-reaper.toml`) carry `contract = "graph.v2"` and `version = 2`. Line 441 says first-party formulas "must stay dual-declared until the minimum supported Gas City binary understands `[requires]`," but neither the design nor a fixture shows the dual-declared shape. New users grepping examples will copy legacy-only formulas; new authors won't know whether to keep `contract`, drop `version`, or both.
- [Minor] **Pack `requires`/`requires_gc` overload is acknowledged in the glossary but not disambiguated where users meet it.** `pack.requires_gc = ">=0.20"` is real syntax (`docs/packv2/doc-loader-v2.md:356`, `docs/guides/migrating-to-pack-vnext.md:673`). A user editing a TOML with `[requires]` near the top has no quick way to distinguish formula-level `[requires]` from pack-level `[pack].requires_gc` without cross-reading two reference docs. The design names the overload (line 646) but does not require a cross-link or callout in either reference page. This is the persona's red flag #2.
- [Minor] **`engdocs/architecture/formulas.md` is dated 2026-03-17 and silent on the entire compiler-capability story.** It still teaches `Store.MolCook` as the runtime seam (line 13-14), never mentions `contract`, `[requires]`, capability, or diagnostics, and the design only lists it as one row in the file-family table (line 657) without specifying the content delta. The architecture doc currently advertises a model the design is replacing. Reading both today produces a contradiction.
- [Minor] **`docs/reference/config.md:259-260` and `docs/reference/cli.md:2495-2496` use the deprecated user-facing term.** Config doc describes `formula_v2` as enabling "graph.v2 formula compilation"; CLI help calls flags "for graph.v2 launches." The glossary deprecates `graph.v2` as user-facing terminology (line 642). After Phase 3 ships, generated CLI help and config reference will be teaching the deprecated term while diagnostics tell users to use `[requires] formula_compiler = ">=2"`. Stale-guidance check (line 457) needs to catch these specific strings.

**Missing evidence:**

- A worked TOML example of a dual-declared formula (both `contract = "graph.v2"` and `[requires] formula_compiler = ">=2"`). The design jumps from default-capability example (line 120) to v2-only example (line 134) to migration matrices, without ever showing the alias-window shape that first-party packs are required to write.
- A specific update plan for the five in-repo legacy-only example formulas. The design's caller inventory and provenance sections list internals exhaustively, but the examples row in the Terminology section is one line ("Dual declarations during alias window, `[requires]` as canonical prose") with no per-file checklist.
- The exact content delta for each user-facing reference file. The Terminology table (line 654-661) names files but not what changes inside them. For `docs/reference/formula.md`: which rows in Common Top-Level Keys move from canonical to deprecated, what new rows are added, where the disambiguation callout for pack-level `requires_gc` lives. For `docs/reference/config.md`: the replacement wording for "graph.v2 formula compilation". For `docs/reference/cli.md`: the replacement flag-help text. Without this, the doc author cannot make the change Phase 3 requires.
- A list of rejected expression forms in the user-facing reference doc. Lines 168-172 list rejected forms (`>=2.0`, `>= 2`, `">=2 "`, `">=3"`, integer `2`) with rationale, but `docs/reference/formula.md` is what users will read when their formula fails with `formula.compiler_requirement_unsupported`. The reference doc must list the accepted forms (`omitted`, `>=1`, `>=2`) and explicitly call out the most likely typos.
- A re-confirmation that long-running daemons preserve deprecation visibility under `OnceKey` LRU suppression (lines 562-568). Per-process CLI suppression is fine; daemon in-memory LRU per city is fine in principle, but the design does not commit to a minimum LRU size or a re-emission window. If a city processes thousands of unique `(code, source_path, source_key, source_value, host_capability)` keys before reset, the LRU could evict warnings the operator needs to see during the alias window. The design says "bounded LRU" without a number.

**Required changes:**

- Reconcile the Rollout Plan with the Terminology contract: either move the doc updates listed in lines 654-661 from Phase 5 to Phase 3 (so user-visible diagnostics and user-facing docs ship together), or remove "in the same release phase" from line 650 and accept the gap explicitly. Pick one. Right now the document promises both.
- Add a canonical TOML example block to the design (under "Compatibility With `contract`") showing the dual-declared formula shape — `formula = "..."`, `[requires] formula_compiler = ">=2"`, *and* `contract = "graph.v2"` — with a one-paragraph note on when authors should drop `contract`. Reference this block from the examples row of the Terminology table.
- Expand the file-family table (line 654-661) into a per-file content delta. For `docs/reference/formula.md`: drop `version = 1` from the minimal example, mark `version` as deprecated in the Common Top-Level Keys table with a one-line note pointing at pack revision, add a `[requires]` row, add a callout disambiguating from `[pack].requires_gc`. For `docs/reference/config.md`: replace "graph.v2 formula compilation" wording. For `docs/reference/cli.md` (and any generator that produces it): replace "graph.v2 launches" with "formula compiler capability >=2 launches" or equivalent.
- Add `engdocs/architecture/formulas.md` to Phase 3 (not Phase 5) with an explicit content gap note: it currently teaches the `Store.MolCook` seam that the migration proposal removes and the contract field that this design replaces. Either supersede the doc or rewrite the Architecture and Instantiation sections; do not silently land contradictions.
- Specify the OnceKey LRU bound and re-emission policy explicitly. Either pick a minimum bound and a periodic re-emission window for long-running daemons during the alias window, or commit to fatal-only suppression for daemons (warnings always emit) until alias removal.
- Add a per-file plan for the five in-repo example formulas (`examples/dolt/formulas/mol-dog-compactor.toml`, `examples/gastown/packs/gastown/formulas/mol-{digest-generate,idea-to-plan,refinery-patrol}.toml`, `examples/gastown/packs/maintenance/formulas/mol-dog-reaper.toml`). They must move to dual-declared in lockstep with the docs flip.
- Add a stale-guidance CI check entry that fails on new docs containing `graph.v2` outside the deprecation/migration sections (line 457 promises this for `contract` as canonical surface; the matching check for `graph.v2` user-facing terminology is implied but not committed).

**Questions:**

- Lane question 1 (canonical surface taught across docs/reference, tutorials, architecture, examples in the same change): Does the Phase 5 deferral of doc updates (line 686) take priority over the Terminology contract's "same release phase" claim (line 650)? These are mutually exclusive as written.
- Lane question 2 (one glossary-level distinction across capability/implementation/contract/schema/pack revision): The glossary itself is good, but does the design require the same disambiguation to land in the user-facing reference docs (`docs/reference/formula.md`, pack reference) and not only in `engdocs/design/`?
- Lane question 3 (open questions resolved before shipping): The design closes the exact-values question (lines 167-172) and the warning-transport question (lines 525-568) and the legacy `version` question (lines 521, 627-629, 663-667) on the implementation side. But all three remain *user-visibly open* as long as `docs/reference/formula.md`, the example files, and `engdocs/architecture/formulas.md` have not been updated. Required answer: docs land in Phase 3, not Phase 5.
- Is `gc formula validate --provenance` and `gc formula validate --all-packs --legacy-contract-report` (lines 612-613, 474-486) supposed to appear in `docs/reference/cli.md` in the same phase that the binary supports them? The design lists the CLI reference in the file-family table but does not commit to a phase for the new flags.
- Does `formula.version_deprecated` ever fire on third-party formulas the operator did not author? If so, the remediation string ("use pack version/ref/SHA for artifact identity") is correct guidance for first-party authors but wrong guidance for an operator who is just running someone else's pack. Worth a sentence on whose action the warning is requesting.
