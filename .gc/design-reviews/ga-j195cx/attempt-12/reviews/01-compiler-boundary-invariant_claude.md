# Nadia Sorenson - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- The boundary is named in the design vocabulary itself: a glossary distinguishes formula compiler *capability* (declared by the formula), *host capability* (what the active binary can satisfy), and *compiler implementation* (chosen by the binary, never by the formula). Combined with the explicit Non-Goals "Adding runtime compiler selection" and "Adding multiple formula compiler implementations", this leaves the active Gas City binary as the sole selector and structurally prevents a future axis from sliding into formula-side selection.
- `internal/formula` is named the only legal interpreter of raw `contract`, `requires.formula_compiler`, `version`, and v2-only construct strings, with a normalized `CompileResult`, a typed `HostCapabilities` input that is "an explicit input, not package-global state", an explicit deterministic diagnostic order, an exhaustive validation matrix, and a per-surface caller-migration table that names every projection (sling, orders, convoy, API, convergence, dashboard, molecule, graphroute, sourceworkflow) that must consume the typed result instead of branching on raw fields.
- A static CI guard is required for new behavioral uses of `contract = "graph.v2"`, `declaresGraphV2Contract`, `Requires.FormulaCompiler`, and `gc.formula_contract` outside an explicit allow-list, plus a "no event/wire path may hand-write JSON or `map[string]any`" rule, both of which match the project's typed-wire/typed-events invariants and are exactly the kind of structural enforcement that prevents drift from creeping back across the alias window.

**Critical risks:**
- [Blocker] The v2-only construct registry that drives the entire missing-requirement diagnostic and static guard is internally inconsistent. The table's column header is "Locations scanned", but two rows are labeled "Graph workflow metadata keys" with different RHS contents (one giving locations, one giving key names) and a third row "Graph workflow metadata values" puts `gc.kind` *values* in the locations column. A reader cannot tell whether the trigger is "any metadata key in {`steps`, `children`, `loop.body`, expansion `template`}", "metadata values like `gc.kind = scope`", or "metadata keys named `gc.scope_name`/`gc.scope_role`/...", or all three. Because this registry is the contract for "what triggers `requires.formula_compiler = ">=2"`", any ambiguity here is a contract defect — the parser, the missing-requirement diagnostic, and the future-capability guard all consume it.
- [Major] The shared workflow-root predicate's owning package is left as an "or": `internal/sourceworkflow.IsWorkflowRoot` and `ListLiveRoots` may either move to `internal/formula` *or* remain in `internal/sourceworkflow` "as the sole persistence predicate that calls formula metadata helpers". The whole point of the boundary invariant is that *one* package owns requirement interpretation; leaving the canonical predicate's home unresolved means two packages can each plausibly claim ownership, and the predicate is exactly the kind of helper that gets quietly forked when callers need slightly different fallback behavior across the alias window. The predicate is also referenced four separate times without a named symbol, package home, or signature — without `func IsGraphWorkflowRoot(meta map[string]string) bool` (or similar) pinned down, every consumer (sling, convoy cleanup, API projection, dashboard) will re-implement a slight variant.
- [Major] Convergence validation is also left as an "or": "Validate through `internal/formula` preflight *or* an adapter over `CompileResult`". `internal/convergence/formula.go` already exists today as a "convergence formula subset" validator (per `engdocs/architecture/formulas.md`) and has historical drift incentives (subset semantics deliberately diverge from the full compiler). Without committing to "convergence calls `CompileWithResult` for every create/retry/speculative-wisp entry point and any convergence-specific checks are strictly post-compile domain checks operating on `CompileResult.Recipe`", the design re-creates the duplicated-requirement-check failure mode this lane is meant to prevent.
- [Major] The static guard's symbol list is too narrow to enforce the boundary invariant against typed-field drift. It catches raw `contract`, `declaresGraphV2Contract`, raw `Requires.FormulaCompiler`, and `gc.formula_contract`, but not callers that consume the *typed* normalized fields and re-implement requirement reasoning — for example, comparisons against `CompilerCapabilityGraphV2` outside `internal/formula`/`internal/molecule`, switches on `RequirementSource`, reads of `Diagnostic.Code` to recompute satisfaction, reads of `HostCapabilities.FormulaV2` (the legacy bool) instead of `FormulaCompiler`, or hand-built `Recipe.GraphWorkflow` checks. Without these in the guard, "no caller branches on raw contract" can hold while "callers re-implement `CheckRequirements` against typed fields" silently regresses. The design also does not specify the *mechanism* of the guard (vet analyzer, golangci-lint plugin, AST-aware tool, or grep) — a string-match grep is brittle against trivial renames and cannot distinguish field-access from string occurrences.
- [Major] `Recipe`'s raw-field exposure is unspecified. The design says "`Recipe` may carry the normalized fields for instantiation, but it must not force consumers to re-parse raw TOML strings." This does not answer whether `Recipe.Contract`, `Recipe.Requires`, or `Recipe.Version` are exported on the public type. If they remain public, a caller can reach `recipe.Requires.FormulaCompiler` directly and bypass the `CompileResult.Requirements` projection. The design must either remove these fields from the public `Recipe` surface or explicitly enumerate every public field name in the static guard with a field-access-aware analyzer.
- [Minor] `CompilerCapabilityGraphV2 = 2` names the canonical capability constant after an implementation. The design otherwise insists the formula declares capability and the binary chooses implementation; encoding "GraphV2" into the constant invites future maintainers to read the capability number as an implementation selector and makes the implementation-neutral framing in the glossary inconsistent with the Go-level vocabulary callers actually see. The constant should be implementation-neutral (e.g., `CompilerCapabilityV2`).
- [Minor] `HostCapabilities.FormulaV2 bool` dual-purposes the typed capability. The design says the bool "exists only as legacy config vocabulary" and `FormulaCompiler` is canonical, but the bool sits in the same struct, is reachable by every call site that holds a `HostCapabilities`, and the design does not forbid `CheckRequirements` from looking at it. Either remove the bool from the type and build it at the config edge, or add it to the static guard so reads outside the config-edge wrapper fail.
- [Minor] `Recipe.GraphWorkflow` and `CompileResult.GraphWorkflow` are documented as parallel surfaces (both readable by callers per the migration table). Both are derivable from `Requirements.FormulaCompiler >= CompilerCapabilityGraphV2`. Two derived bool fields with no enforced single source of truth is a footgun for tests that hand-construct a `Recipe` with mismatched fields and for any future caller that reads `Recipe.GraphWorkflow` while `CompileResult.Requirements` is the load-bearing source.
- [Minor] `needs`, `depends_on`, `children`, `loop`, `gate`, `condition`, `compose.expand`, `compose.map`, and `compose.aspects` are "v2-only only when the resolved formula, expansion, aspect, or nested body also contains one of the v2-only constructs above". This conditional rule conflates two questions ("does this construct require v2?" vs "does this formula as a whole require v2?"); any future v2-only construct addition silently changes whether a formula using only `needs` requires v2, and a missing-requirement diagnostic may end up pointing at a `needs` line whose author never opted into v2 syntax. The cleaner invariant is that the registry lists constructs that always trigger v2, and these dependent constructs never themselves trigger.
- [Minor] `CompilerCapability(0)` is documented as "uninitialized internal value" that "is never a valid normalized capability". The type system does not enforce this — any caller that constructs `NormalizedRequirements{}` or `HostCapabilities{}` and calls `CheckRequirements` is in undefined territory. A constructor-only path or a defensive normalization in `CheckRequirements` is needed; otherwise zero-valued capabilities can flow through the system and bypass requirement satisfaction silently.

**Missing evidence:**
- The static guard's allow-list is informal: "outside the parser, compatibility metadata writer, shared workflow-root predicate, tests, fixtures, and docs". "The parser" is a package, "compatibility metadata writer" is unnamed, the shared predicate's home is itself "or", and "tests/fixtures/docs" is path-based. There is no concrete file/symbol allow-list, so the guard cannot be implemented unambiguously and reviewers cannot tell whether a new caller is in or out of compliance.
- The home of the legacy `SetFormulaV2Enabled`/`IsFormulaV2Enabled` shims during the migration is unspecified. The doc says they "may not be read inside requirement normalization or satisfaction logic" but does not say *where they live* — if they live in `internal/formula`, the package itself exposes the package-global capability surface it forbids itself from reading, which is a maintenance trap.
- There is no test specified that asserts `CheckRequirements` is sensitive *only* to `HostCapabilities.FormulaCompiler` and ignores `HostCapabilities.FormulaV2`, package globals, and `Requirements.Source`. The "two compiles in the same process can evaluate the same formula against different host capabilities deterministically" test is good for per-call isolation but does not prove the canonical capability is the *only* input to satisfaction.
- There is no test specified that asserts a formula whose `[requires]` syntax tries to select an implementation (e.g., a hostile or accidental future axis like `requires.formula_compiler_implementation = "graph.v2"`) is rejected by `formula.requirement_unknown_axis` rather than honored. This pins the binary-owns-implementation invariant against future axis additions.
- There is no concrete plan for verifying the `bd` shell-out path (`GC_NATIVE_FORMULA=false`) parses `[requires]` identically before any first-party formula becomes requires-only. The design says either the dual `contract` declaration must be preserved or `bd` must be proven to parse `[requires]` identically, but does not say who proves it or where the test lives, leaving Phase 7 (alias removal) without a firing condition.
- There is no phase that turns off the `gc.formula_contract` legacy dual-stamp on new producers. New producers will keep dual-stamping forever unless an explicit phase ends it; without that phase, the static guard's compatibility allow-list becomes permanent and the predicate's legacy fallback never sunsets.

**Required changes:**
- Restructure the v2-only construct registry table so each row's RHS matches the column header. Either change the column to "Trigger description" (locations + values + keys mixed) or, preferably, split the table into three: constructs scanned at named locations, metadata-value triggers, and metadata-key triggers. Make the registry the unambiguous source of truth that the parser, the missing-requirement diagnostic, the static guard, and the future-capability guard all reference by name.
- Commit `internal/sourceworkflow.IsWorkflowRoot`/`ListLiveRoots` to a single home and pin a named, signed predicate (e.g., `formula.IsGraphWorkflowRoot(meta map[string]string) bool` in `internal/formula`, used by sling, convoy cleanup, API projections, and dashboard mappers). Either move the existing predicates to that home, or designate `internal/sourceworkflow` as the persistence-edge predicate that calls only `internal/formula` helpers, and add the chosen owner to the static guard's whitelist by name. Remove the "or".
- Commit convergence to "calls `CompileWithResult` for every create/retry/speculative-wisp entry point; convergence-only validation is strictly post-compile and operates on `CompileResult.Recipe` fields, not raw TOML." Remove the "or an adapter" language. Add `internal/convergence/formula.go` to the static guard's no-formula-parsing scope (not on the allowlist) so future convergence code cannot import a TOML decoder or interpret raw `contract`/`requires` strings.
- Resolve `Recipe`'s raw-field exposure: either declare `Recipe.Contract`, `Recipe.Requires`, and `Recipe.Version` unexported (preferred), or enumerate every public field name in the static guard's deny-list with a field-access-aware analyzer.
- Specify the static guard's implementation mechanism (vet analyzer, golangci-lint plugin, custom AST tool, or grep) and the exact symbols/strings on the deny-list with the file allowlist (parser, compatibility metadata writer, named predicate package, tests, fixtures, docs). String-match grep is acceptable only for string literals like `"graph.v2"`; field-access enforcement requires AST-aware tooling.
- Extend the static guard symbol list, outside the explicit allow-list, to also fail on:
  - `formula.CompilerCapabilityGraphV2` comparisons (callers branching on capability),
  - `formula.RequirementSource*` reads (callers branching on legacy source),
  - `formula.HostCapabilities.FormulaV2` reads (legacy bool),
  - `Recipe.GraphWorkflow` reads outside `internal/molecule`/`internal/formula` (force callers to consume `CompileResult.GraphWorkflow`),
  - `Diagnostic.Code` reads in code that creates beads, wisps, attached molecules, or convergence instances (requirement satisfaction must come from `CheckRequirements`, not from inspecting diagnostic codes after the fact),
  - direct reads of `metadata["gc.formula_contract"]` outside the named predicate's package.
- Rename `CompilerCapabilityGraphV2` to a number-only or implementation-neutral identifier (e.g., `CompilerCapabilityV2`) so the canonical capability constant does not encode an implementation name. Update glossary, validation matrix, and all examples accordingly.
- Make `Recipe.GraphWorkflow` either unexported or strictly populated by `formula.CompileWithResult`/`Compile`, and document a single derivation rule: `GraphWorkflow == (Requirements.FormulaCompiler >= CompilerCapabilityV2)`. Add a unit test that asserts the invariant for every code path that constructs a `Recipe`, including test helpers.
- Drop `needs`, `depends_on`, `children`, `loop`, `gate`, `condition`, `compose.expand`, `compose.map`, and `compose.aspects` from the conditional v2-only treatment. The "they trigger v2 only if another v2 construct is present" rule reduces to "the other v2 construct is what triggers v2" — that is the rule to keep.
- Forbid `internal/formula` from depending on its own `SetFormulaV2Enabled`/`IsFormulaV2Enabled` package globals during the migration. Either delete the globals (build `CompileOptions.HostCapabilities` at the CLI/API edge from `[daemon] formula_v2`), or keep the globals only in CLI/API-edge files that are explicitly named in the static-guard allow-list. The package that forbids reading them must not also expose them.
- Add a phase aligned with Phase 7 that also stops writing `gc.formula_contract` on new roots, with a defined firing condition tied to the `bd`-compatibility-complete trigger. Without this, dual-stamping is permanent.
- Add a Phase 4.5 (or extend Phase 4) that defines the "bd compatibility complete" trigger as a concrete, automatable check (e.g., "CI exercises the bd shell-out path against a corpus of `[requires]`-only formulas and verifies parity with native compilation"). Without this, Phase 7 has no firing condition.
- Add a test that asserts `CheckRequirements(req, host)` produces identical diagnostics regardless of `host.FormulaV2`'s value when `host.FormulaCompiler` is held constant, and identical diagnostics regardless of `req.Source` when `req.FormulaCompiler` is held constant. This proves capability is the only input that drives satisfaction.
- Add a test that asserts `NormalizedRequirements{FormulaCompiler: 0}` and `HostCapabilities{FormulaCompiler: 0}` either fail loudly (panic, error, or required diagnostic) or are normalized to a defined value before reaching `CheckRequirements`. Zero values must not silently behave as `CompilerCapabilityDefault` because the type already has an explicit `RequirementSourceOmitted` for that case.
- Add a test that asserts an unknown `[requires]` axis (e.g., `requires.formula_compiler_implementation = "graph.v2"`) is rejected by `formula.requirement_unknown_axis` and never reaches any compiler-implementation selection path. This pins down the "binary owns implementation" invariant against a hostile or accidental future axis.

**Questions:**
- Which package owns the shared workflow-root predicate after the alias window closes? The design's "Move predicate to `internal/formula` or keep this package as the sole persistence predicate" strongly suggests a future re-org. If the answer is "stays in `internal/sourceworkflow`" indefinitely, what stops a future caller from inlining the canonical-vs-legacy decision tree as a "small optimization"?
- Once first-party formulas become requires-only (Phase 6), is the `gc.formula_contract` dual-stamp removed from new producers, or does it remain forever to support externally pinned legacy packs reading new bead stores? If it remains, the predicate must remain too — and the static guard's compatibility allow-list becomes permanent rather than transitional. The boundary invariant is easier to defend if this is decided now.
- Does the `bd` shell-out path (`GC_NATIVE_FORMULA=false`) ever observe the `[requires]` syntax? The compatibility matrix says it must "be proven to parse `[requires]` identically before source conversion" *or* the dual declaration must be preserved. If `bd` never parses `[requires]`, the dual declaration is permanent for any formula that may be compiled by `bd`, which contradicts the "first-party requires-only conversion" phase goal. Resolving this affects whether the alias window can ever close.
- The active formula-migration proposal (`engdocs/proposals/formula-migration.md`) is removing `bd mol wisp`/`bd mol bond` shell-out in favor of native `internal/formula.Compile` + `internal/molecule.Cook`. If that migration completes, the "old binary that only knows `contract`" target may not exist for first-party use, which could let Phase 7 fire as soon as the native migration completes plus the first-party two-minor-release window. Is that the intent, and how do the two designs sequence?
- Is `internal/convergence/formula.go` (the "convergence formula subset" validator) intended to remain as a package once `CompileWithResult` is universally adopted, or is the long-term goal to delete it and have convergence consume `CompileResult` directly? Settling this now changes whether code reviews flag every new field added to the subset as a drift signal.
- The compatibility matrix lists "`[requires] formula_compiler = ">=2"` only" against "old binary that only knows `contract`" as "Not supported for v2 formulas; old binary can miss the requirement." Is "miss the requirement" specifically defined? If the old binary parses the formula, ignores the unknown `[requires]` table, and proceeds with default-capability compilation against v2-only constructs, the failure mode could be silent corruption (steps that look right but compile under the wrong rules) rather than a clean fail-stop. The hazard description should be explicit so first-party-pack maintainers don't relax dual-declaration prematurely.
