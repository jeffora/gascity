# Ibrahim Park - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- Fail-closed grammar with an enumerated rejected-form table. The Requirement Expression section names every variant the parser rejects in v0 (`"2"`, `">= 2"`, `" >=2"`, `">=2 "`, `">=2.0"`, `">=2.1"`, `">=3"`, plus every non-string TOML type and every dotted/nested table). Combined with the Forward Compatibility section's explicit "intentionally fail-closed" framing, this is the load-bearing fence that prevents future binaries from inheriting accidental compatibility, and old binaries can distinguish "this needs a newer compiler" from "this is shaped wrong."
- Unknown `[requires]` keys produce a hard validation error (`formula.requirement_unknown_axis`, error severity, deterministic position 2 in the diagnostic order). This directly closes the "unsupported requirement axes silently ignored" red flag and forces every new axis to ship as a coordinated parser change rather than slipping in as data the host quietly drops. The "Every future requirement axis must add..." checklist (typed normalized state, accepted/rejected grammar tests, diagnostic codes, provenance fields, persisted metadata, docs/schemas) is the strongest forward-extension contract in the document.
- Capability is normalized into a typed `NormalizedRequirements` with an integer `CompilerCapability` plus a `RequirementSource` discriminator, and the canonical-compile-result invariant (with a CI static guard against `Contract`/`declaresGraphV2Contract`/`Requires.FormulaCompiler`/`gc.formula_contract` outside the parser and shared predicate) prevents drift in callers. Future axes can extend the typed state without re-shaping consumers or sliding back into ad-hoc TOML re-parsing.

**Critical risks:**
- [Major] **Capability monotonicity under `>=N` is implicit and unstated.** The grammar adopts `>=N` syntax and accepts `>=1` and `>=2`. Two important questions are not answered: (1) when v3 ships, does a binary with host capability `3` automatically satisfy `>=2` (the natural reading of `>=`), and (2) is a formula authored against capability 2 guaranteed to keep working under capability 3 forever? Today the design says nothing about whether compiler capability is monotonically backwards compatible — yet the syntax invites that assumption. If capability `3` ever introduces a breaking change, formulas with `>=2` will silently compile under v3 with different semantics. The mandate explicitly asks whether `>=1` and `>=2` semantics are defined tightly enough to avoid divergence later; today the answer is no, because the meaning of `>=N` under future N is left for the future to pick. Required: state the monotonicity invariant explicitly (or commit to "future capabilities never break older requirements"), or replace the `>=` syntax with an enum-shaped form (e.g., a bare integer `2` or `compiler_capability_v2`) that does not promise comparator semantics.
- [Major] **Grammar shape adopts comparator-like syntax for an enum-shaped parser, with no forward-compat plan.** `>=2` looks like a comparator but the parser is a flat closed allowlist; `">=2.1"`, `">=2,<4"`, `^2`, `==2`, `>2`, `!=3` are all rejected today. When the next axis lands or v3 ships, contributors must decide whether to keep extending the enum (`>=3` joins `>=2` joins `>=1`) or transition to a real range parser. The design says only "Future compiler capability expressions can be added only by extending this grammar and its validation matrix in the same change" — that is a process commitment, not a syntactic one. The semver-shaped surface also invites well-meaning operators to type `>=2.1` thinking the parser handles minor versions; a clear "minimum-major-only enum, ever" commitment (or, conversely, a clear "we will transition to a comparator parser at v3" plan) is needed before the v0 surface is published. The mandate calls out "future capability ranges"; the design does not say whether v0 is the permanent shape or a transitional one.
- [Major] **No reservation policy for future requirement axes.** Strict-reject is the right default, but the design does not enumerate planned future axes (`tools`, `pack_format`, `runtime`, `agents` are mentioned in the persona mandate; the Terminology section notes pack-level `requires` / `requires_gc` exist as separate constructs but does not list intended formula-level axes), does not reserve a vendor-prefix or extension namespace (e.g., `requires.x_*` for ignored-by-design experimental keys), and does not state whether strict-reject is permanent or transitional. The consequence: every future axis is a coordinated binary-floor break, every binary below the new floor hard-fails on every formula that uses the new axis, and there is no graceful-degradation lever for vendor experimentation. At minimum the design should publish a short "axes we intend to add" list and decide whether vendor-prefixed keys are reserved, ignored, or rejected.
- [Major] **`formula.compiler_requirement_unsupported` cannot distinguish "binary too old" from "operator typo".** The diagnostic code and remediation are the same for `[requires] formula_compiler = ">=3"` (v3 binary needed) as for `[requires] formula_compiler = ">=2.0"` (typo). The remediation text — "use omitted, `>=1`, or `>=2`" — is actively wrong for the binary-too-old case (the operator should upgrade `gc`, not edit the formula). With strict-reject as the canonical forward-compat strategy, this collapses the most common operator-experienced upgrade signal into the same surface as a misspelling. The design should split the diagnostic: one code for "syntactically invalid" with the existing remediation, one code for "syntactically valid but binary cannot satisfy" (e.g., `formula.compiler_requirement_unknown_capability`) with an "upgrade gc" remediation, distinguished by whether the value matches the `>=N` shape with an unrecognized `N`.
- [Major] **`CompilerCapabilityGraphV2` couples the constant name to a feature, not to a revision.** Every other piece of the design treats compiler capability as a numeric scale (`gc.formula_compiler_capability = "2"` is canonical persisted metadata; capability numbers are stable wire values per the Forward Compatibility section). But the Go constant is named after the v2 feature (`graph`), not the revision. If v3 brings a non-graph capability, the constant set drifts (`CompilerCapabilityGraphV2`, `CompilerCapabilityV3`?), and renaming the constant later does not change the persisted metadata. This is the kind of name that becomes a permanent wart. Required: rename to `CompilerCapabilityV2` (or `CompilerCapabilityRev2`) and document the v2-feature association in the comment, before the symbol becomes load-bearing in tests and call sites.

**Missing evidence:**
- The compatibility matrix in "Compatibility With `contract`" enumerates "no requirement" and `>=2` rows but does not include an explicit `[requires] formula_compiler = ">=1"` row across the three binary states (old, new with `formula_v2 = false`, new with `formula_v2 = true`). The Requirement Expression table asserts `>=1` is equivalent to omitted, but the per-binary behavior is not pinned in the migration matrix. This is the exact place omitted-vs-`>=1` divergence could leak in.
- The design does not specify whether `RequirementSource` for explicit `[requires] formula_compiler = ">=1"` is `RequirementSourceRequires` or `RequirementSourceOmitted`. The constants exist but the mapping is implicit. Two semantically-equivalent formulas (omitted vs `>=1`) may produce different `Source` values, and downstream consumers (telemetry, deprecation policy, dashboard filters) may treat them differently — exactly the "omitted and `>=1` diverge accidentally" red flag.
- `CompilerCapability(0)` is documented as "uninitialized internal value" in Forward Compatibility, but the design does not say what happens if the parser produces a value of `0` due to a bug, nor does it require parser code to never emit it. There is no constant for it (no `CompilerCapabilityUnknown`), so a zero-valued struct (`NormalizedRequirements{}`) is a silent footgun. Tests should assert that no public API ever returns `FormulaCompiler == 0`.
- The required validation matrix has rows for `[requires]` with no v2-only construct and v2-only construct without `[requires]`, but no row for "`[requires] formula_compiler = ">=1"` with v2-only construct present" (presumably `formula.compiler_requirement_missing` like the omitted case, but the matrix collapses both into "omitted or `>=1`"). The collapsed row is fine if the source attribution is also collapsed; it is misleading if `RequirementSource` differs. Pin both behaviors in the same row.
- Inheritance semantics for source attribution are not specified. If a parent has `>=1` (explicit) and a child has omitted, the child's normalized requirement is `>=1`. What is the source — `requires` (inherited from parent) or `omitted` (the child's own input)? The design specifies value inheritance ("a child may raise the requirement, but it may not lower a parent requirement") but not provenance inheritance.
- No explicit policy on whitespace, leading-zero, or canonicalization rules. The rejection list rejects `">= 2"` and `">=2 "` but does not say whether `">=02"`, `">=2\n"`, or other unicode whitespace are also rejected. A formal grammar (BNF) for the accepted strings would close this, especially as future maintainers add `>=3`.

**Required changes:**
- Add an explicit "Capability Monotonicity" subsection stating whether `>=N` means "minimum capability N, monotonically backward compatible" or "minimum capability N as a flat enum lookup." If the former, document that compiler capabilities must remain backward compatible forever (no breaking changes), and require future capability work to publish a compatibility statement. If the latter, change the grammar to a non-comparator form before v0 ships (e.g., bare `"2"` or `"capability_2"`) so the syntax does not promise comparator semantics it does not deliver.
- Split `formula.compiler_requirement_unsupported` into two codes: one for syntactically invalid expressions ("use omitted, `>=1`, or `>=2`") and one for syntactically valid but unrecognized capabilities (`formula.compiler_requirement_unknown_capability` with remediation "upgrade gc to a binary that supports formula compiler capability N"). Distinguish by whether the value parses as `>=([1-9][0-9]*)`. This makes the v0 fail-closed strategy operator-actionable instead of misleading.
- Add the missing `[requires] formula_compiler = ">=1"` row to the compatibility matrix (across old binary, new binary `formula_v2 = false`, new binary `formula_v2 = true`), and add a row for `[requires] formula_compiler = ">=1"` with a v2-only construct present. Pin both `Source` and behavior.
- Specify `RequirementSource` for every accepted input form (omitted, empty table, `>=1`, `>=2`, `contract = "graph.v2"` only, dual). Add a normative table like: `omitted → omitted`, `empty table → omitted`, `>=1 → requires`, `>=2 → requires`, `contract only → contract`, `dual → dual`. Specify inheritance source semantics (does a child inherit the parent's source, or is the source always the formula that produced the winning normalized value?).
- Rename `CompilerCapabilityGraphV2` to `CompilerCapabilityV2` (or `CompilerCapabilityRev2`) and add a comment that v2 currently denotes graph-workflow capability. Apply the same revision-numbered convention to all future constants.
- Reserve a vendor-prefix or extension namespace explicitly: either commit "all unknown keys are rejected, forever" or carve out `requires.x_*` / `requires.experimental_*` as ignored-by-design. Either commitment is acceptable; silence is not, because operator behavior diverges based on which interpretation is assumed.
- Publish a short "intended future axes" list in the design (or in `engdocs/architecture/formulas.md`): for example, `tools`, `pack_format`, `runtime`, `agents`. This does not commit to implementing any of them in v0, but it warns formula authors and pack maintainers not to use those key names for unrelated purposes.
- Add a normative grammar (BNF or regex) for accepted requirement strings, e.g., `^>=([1-9][0-9]*)$`, and require parser tests to fuzz alternate whitespace, leading zeros, and unicode forms against the grammar. This prevents drift when capability `3` is added and someone forgets to update the rejected-string list.
- Define `CompilerCapability(0)` policy: either add a `CompilerCapabilityUnknown CompilerCapability = 0` constant and require parser tests to assert public APIs never return it, or document that `0` is an internal invariant violation that triggers a panic in normalization. Pick one.
- Specify a per-axis diagnostic code convention now (before any future axis ships). Either all axes share `formula.requirement_unknown_axis` and `formula.requirement_invalid_type`, or each axis gets a per-axis code (`formula.requirement_tools_unsupported`). Picking the convention upfront prevents retroactive renames once dashboards and release-note tooling latch onto the v0 codes.
- State the snake_case axis-naming convention. `formula_compiler` is snake_case; commit that all future axes will be too. Otherwise the namespace fragments (`packFormat`, `pack-format`, `pack_format`).

**Questions:**
- Is compiler capability guaranteed to be monotonically backward compatible (capability `N+1` always satisfies `>=N`), or does the design reserve the right to break older capability requirements at major boundaries? The answer determines whether `>=N` syntax is honest.
- Are there capabilities below `1`? The design treats `1` as the floor (`CompilerCapabilityDefault = 1`); is this permanent? If a future "minimal" or "no-compile" mode arrives, can it be expressed?
- Is `RequirementSourceDual` permanent or transition-only? Once `contract` is removed, `dual` becomes dead code. If transition-only, where is its removal tracked, and what is the deprecation criterion?
- What is the canonical relationship between formula-level `[requires] formula_compiler` and pack-level `requires` / `requires_gc`? The Terminology section notes they are distinct, but does a pack-level requirement ever subsume or shadow a formula-level requirement? If a pack declares `requires_gc = ">=2"` and a contained formula declares `[requires] formula_compiler = ">=1"`, which wins, and is that combination even allowed?
- Are vendor-prefixed keys (e.g., `requires.x_org_internal`) intended to be supported, ignored, or rejected? The doc says all unknown keys fail; the question is whether that is permanent.
- When `gc.formula_compiler_capability` is persisted as canonical metadata and the alias `gc.formula_contract` is removed, is the persisted capability number frozen forever, or can a future migration remap (e.g., compress capability `2` into capability `2`-with-aspects)? The Forward Compatibility section says "Compiler capability numbers are stable once released" but does not say persisted metadata values are immutable.
- Should `CompileResult` include the raw rejected-form value when a diagnostic is fatal, so that operators can correlate "I typed `>= 2` and got rejected" with the diagnostic without reading the formula file? `Diagnostic.SourceValue` exists but its required population for unsupported-expression cases is not stated.
