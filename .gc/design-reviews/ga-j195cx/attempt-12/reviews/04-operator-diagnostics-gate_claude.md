# Marta Hidalgo - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- Diagnostic codes cleanly separate parser failures from disabled-capability state. Each fatal code gets a canonical message + remediation pair (lines 525-535), so an operator who hits `formula.compiler_requirement_unsatisfied` sees "enable `[daemon] formula_v2` or choose a v1 formula" rather than a parse trace. The `Diagnostic` struct's structured fields (`Code`, `Remediation`, `OnceKey`, `HostCapability`, `NormalizedRequirement`) make CLI/API/dashboard projection a structural property, not a string-replication discipline.
- The host capability gate operates on the normalized canonical form, not on the raw source spelling. `CheckRequirements(NormalizedRequirements, HostCapabilities)` (line 348) plus the validation matrix (lines 236-249) and the compatibility matrix (line 437) all confirm that `contract = "graph.v2"` and `[requires] formula_compiler = ">=2"` flow through the same gate path with the same diagnostic. The `RequirementSource` enum (`requires`/`contract`/`dual`/`omitted`) preserves provenance for diagnostics without leaking into satisfaction logic.
- `OnceKey` shape includes `host_capability` (line 562-563), so suppression keys reset on flag flips and on config reload. That means an operator who toggles `[daemon] formula_v2` sees fresh warnings reflecting the new state instead of stale silence — a non-obvious but operationally correct choice for the "warnings fire on every resolve cycle" red flag.

**Critical risks:**
- **[Major] Fatal diagnostic events have no rate-limiting under poll-driven order dispatch.** Lines 547-548 say "Order dispatch emits one typed order failure event per failed dispatch attempt and continues scanning later orders" and lines 567-568 say "Fatal diagnostics are emitted once per failed dispatch/launch attempt so operators can correlate failed work without warning floods." For an order with a poll cadence (orders are scheduled, poll-driven work in this codebase), `formula_v2 = false` against a graph-required formula produces a fresh `formula.compiler_requirement_unsatisfied` event on every poll tick, indefinitely, until config is fixed. The design does not specify whether the dashboard projection coalesces by `once_key`, whether downstream alerting consumers must dedupe, or whether the event bus has any per-source throttle. The `once_key` field is present in the event payload (line 559-561) but no surface is named as the deduper. This is the exact "deprecation warnings fire on every resolve cycle" red flag generalized to fatal events, which the design's own warnings-only suppression policy leaves uncovered.
- **[Major] HTTP 400 vs 409 split for `formula.compiler_requirement_unsatisfied` is undocumented at the actor boundary.** Line 532 maps the same diagnostic to "HTTP 400 for formula input or 409 for launch conflict plus diagnostic body." The design does not name which endpoint paths use which status, what makes a request "input" vs. "launch conflict," or how the dashboard renders both into a single "disabled-capability state with remediation" card. A dashboard implementation that branches on HTTP status (a common pattern) will produce two different operator UX paths for the same root cause. The typed projection of `code` mitigates this only if every dashboard surface reads `code` and ignores status — which is a discipline, not a structural guarantee.
- **[Major] Warning-suppression LRU is "bounded" but unsized and the rotation behavior is undefined.** Lines 564-568: "in-memory, bounded LRU per city, reset on process restart and config reload, and scoped to warnings only." A long-running controller will hold suppression entries with shape `<code>|<source_path>|<source_key>|<source_value>|<host_capability>`. The cardinality is finite per process, but if the bound is small and entries are evicted under load, the same `formula.contract_deprecated` warning re-fires each time it rotates back in. There is no specified bound, no rotation policy, and no test gate proving that a controller running for N hours emits at most one warning per `OnceKey`. The lane question "Are deprecation warnings emitted at a controlled cadence that will not flood poll-driven dispatch or CI logs?" is not answered for the "long-lived controller" case.
- **[Minor] CLI "warning stderr once per invocation" wording is misleading.** Line 533 says `formula.contract_deprecated` is "warning stderr once per invocation" in CLI. The OnceKey-based suppression keyed on `(code, source_path, source_key, source_value, host_capability)` actually means "once per unique source per invocation." A `gc formula validate --all-packs` run touching 200 legacy formulas will emit 200 deprecation warnings, not one. That is correct behavior; the matrix wording should be "once per unique source per invocation" so operators reading the spec do not expect a single line.
- **[Minor] Migration call-site table mandates dashboard typed projection but no static guard enforces it.** The static guard at lines 399-402 covers `Contract`, `declaresGraphV2Contract`, `Requires.FormulaCompiler`, and `gc.formula_contract` outside `internal/formula` and the shared predicate. It does not guard against the dashboard reading raw root-metadata strings to infer disabled-capability state. Line 372 mandates "Use API-projected typed fields and diagnostics" but lists no CI gate. This is the "dashboard and CLI give contradictory remedies" red flag with no automated backstop.

**Missing evidence:**
- The current state of controller and convergence error/event surfaces is not enumerated. Line 552 mandates "they must not synthesize alternate wording that loses the remediation," but the design does not say whether existing surfaces emit typed errors today or raw strings. Without that inventory, a reader cannot tell whether the migration is a one-line wrap or a ground-up rewrite for those callers.
- The exit-code contract for `gc formula validate <name>` (general invocation, no `--legacy-contract-report`) is unspecified. Lines 491-493 give 0/1/2 semantics for the legacy-contract report, but the general validate path's exit codes for `formula.compiler_requirement_unsatisfied`, `formula.compiler_requirement_missing`, etc. are only implied by the matrix's "fatal stderr, exit 1" projection rule. CI scripts and operator runbooks need an explicit table.
- Concrete dashboard card structure for "formula preview validation error" vs. "disabled-capability state with remediation" is not shown. These are listed as distinct UI states (lines 527, 532) but the design provides no mock, schema, or screenshot. Without a concrete projection target, two dashboard implementers can satisfy the spec with incompatible UIs, both technically conforming to "typed diagnostic" and "remediation field present."
- Whether the dashboard generated types migration has a parity test against API responses is not stated. Line 392 lists "Dashboard generated types | Consume typed API diagnostics and workflow requirement fields" with required tests, but the test description "Dashboard tests for diagnostic rendering without metadata/string inference" does not specify whether those tests catch HTTP status branching, raw-metadata reads, or only end-state rendering.

**Required changes:**
- Specify the warning-suppression LRU bound size, eviction policy, and a unit test proving no warning re-fires for the same `OnceKey` within one controller process across at least the maximum legitimate cardinality (number of distinct legacy formulas times number of distinct source paths). A simple "no eviction; entries persist for process lifetime, capped at 4096" with a test would close this. Tie the cap to memory accounting in `engdocs/architecture` so future increases are tracked.
- Add an explicit emission-cadence policy for fatal diagnostic events under sustained failure. Either: (a) require the producer (controller / order dispatch / convergence) to apply a backoff on identical `(once_key, target_id)` pairs (e.g., emit on first failure, suppress for the next M minutes, re-emit on M-minute boundaries until resolved), or (b) name the dashboard / API projection as the dedup owner and require it to group by `once_key` for display. Pick one and write it into the "Projection rules" section, with a corresponding test in the order-dispatch suite for `formula_v2 = false` over 60 polls.
- Disambiguate HTTP 400 vs 409 in the projection matrix. Add a row per endpoint path (e.g., `POST /sling/formula` is 400 for unsatisfied; `POST /workflow/launch` is 409 for unsatisfied) and require the dashboard to project both into the same operator card via a contract test that hits both endpoints and asserts the same rendered remediation.
- Add a CI gate that fails when dashboard code in `cmd/gc/dashboard/web/` reads `gc.formula_contract` or other root-metadata strings to infer disabled-capability state. The gate may be a grep or AST check over the generated TS types directory and the panel sources. This is the structural counterpart to the existing static guard for raw `Contract` / `declaresGraphV2Contract` use.
- Specify exit codes for `gc formula validate <name>` general invocation in a dedicated table: which diagnostic codes return 1, which return 2 (if any), and how this aligns with the `--legacy-contract-report` JSON exit codes. Operators writing CI gates should not have to read the projection matrix and infer.
- Update line 533's wording from "warning stderr once per invocation" to "warning stderr once per unique `(code, source_path, source_key)` per invocation" so the OnceKey semantics are visible at the matrix level.

**Questions:**
- Is the design intent that operators monitor poll-driven order failures by raw event count (so each occurrence matters for SLA tracking) or that the dashboard surfaces one card per unique failure (so volume reflects distinct issues, not poll cadence)? These are incompatible philosophies and the design currently allows both. Settling this changes whether the producer or the consumer owns deduplication.
- When `[daemon] formula_v2` flips from true to false at runtime, lines 716-718 confirm existing graph-workflow roots continue. But lines 728-730 say a retry or `on_complete` action that compiles a new formula uses the current host capability and can fail. Should the controller emit a one-shot "configuration drift" notification when it detects the toggle has invalidated previously-running formulas, so operators see the impact of their flag flip before the next retry attempt fails? Without it, the failure surfaces only when work tries to make progress, which can be hours later.
- The persisted metadata distinguishes canonical (`gc.formula_compiler_capability = "2"`) from deprecated (`gc.formula_contract = "graph.v2"`). During the alias window, dashboard cards reading legacy-only roots (canonical absent, legacy present) still need to render the "this root requires graph capability" affordance. Is there a dashboard test that verifies a legacy-only root still produces the correct operator remedy card with no canonical metadata at all, simulating an older controller writing into a shared bead store?
- Convergence preflight (lines 728-731) creates "no active loop with missing children" on failed preflight. What does the operator see in the dashboard for a convergence whose preflight failed? Is there a distinct convergence-failed card type, or does it surface as a generic failed-dispatch event? If the latter, the convergence-specific operator remedy ("the convergence root never landed because formula_v2 is off") may be lost in a generic "order failed" card.
