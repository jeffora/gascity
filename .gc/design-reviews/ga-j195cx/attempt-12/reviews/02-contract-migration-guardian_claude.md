# Elias Vega - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- Conflict detection between `contract` and `[requires]` is a first-class hard error (`formula.compiler_requirement_conflict`) with a deterministic position in the diagnostic ordering (rule 4, before host-capability checks). The "both present and disagree" case cannot land silently — directly closing the "conflicting declarations are silently accepted" red flag for the cells the matrix names.
- Alias removal is conditioned on a multi-axis evidence trigger that explicitly rejects calendar deadlines: zero first-party legacy-only formulas (criterion 2, with a JSON exit-code-2 release-gate command), an external-pack support commitment (criterion 3), and a CI stale-guidance check that blocks new docs from teaching `contract` (criterion 4). The "release captain" is named as a deciding owner.
- The compatibility matrix names externally-pinned packs explicitly ("External SHA-pinned formulas that use legacy `contract = "graph.v2"` remain valid through the alias window") and pairs that with a first-party dual-declaration mandate so old binaries never see a requires-only built-in graph formula. Existing legacy formulas continue compiling; nothing fails abruptly.

**Critical risks:**
- [Major] **The "documented compatibility branch" referenced in alias-removal criterion 3 is never defined.** Criterion 3 reads "externally pinned legacy formulas remain supported by either the alias or a documented compatibility branch." That branch is the only escape hatch for SHA-pinned external packs after alias removal, and the design says nothing about its shape (long-lived release line? versioned shim? frozen tag?), its owner, what it backports, how long it lives, or its retirement criteria. As written, the gate is unevaluable on paper and externally-pinned consumers have no concrete forward-compat promise.
- [Major] **`gc formula validate --all-packs --legacy-contract-report` is the load-bearing gate but is itself under-specified.** The doc gives an output JSON shape and exit codes, but does not state which sources the command walks (city packs only? imported packs? aspect/expansion source files? convergence subset? test fixtures with intentional legacy?), what counts as "first-party" (vendored copies of external packs?), and whether it is a CI gate or a release-checklist artifact. Criterion 2 hinges on this command reporting zero — its semantics must be precise.
- [Major] **`contract × [requires]` interaction has unstated edge cells.** The matrix enumerates "only `contract`", "both present and agree", and "both present and disagree (>=1 vs graph.v2)", but several real authoring shapes are ambiguous:
  - `contract = "graph.v2"` + empty `[requires]` table — the empty-equals-default rule means `[requires]` normalizes to `>=1`, which disagrees with `graph.v2`. The doc could equally be read as "only `contract` present" (alias wins, deprecation warning). Different parsers will diverge.
  - `contract = "graph.v2"` + `[requires]` containing only unrelated keys — does the unknown-key error fire before or after the "alias accepted" path?
  - `contract = "graph.v2"` + `formula_compiler = ">=3"` — diagnostic-order rule 3 (unsupported expression) fires before rule 4 (conflict), but the user-facing precedence is not stated.
  - `contract = ""`, `contract = " graph.v2 "`, `contract = "Graph.v2"` — normalization (trim vs reject, case folding) is unstated.
  This is exactly where "conflicting declarations silently accepted" reappears: any implementer who reads the empty-table case as alias-wins disagrees with one who reads it as conflict.
- [Major] **No diagnostic code covers an invalid `contract` value.** The taxonomy has `formula.compiler_requirement_unsupported` (scoped to `[requires]` expressions) and `formula.compiler_requirement_conflict`. Neither cleanly fits `contract = "graph.v3"`, `contract = "garbage"`, or `contract = ""`. The compatibility matrix says "other value | any | any | any | `formula.compiler_requirement_unsupported` on `contract`" — but that code's canonical message ("unsupported formula compiler requirement <value>" / "use omitted, `>=1`, or `>=2`") is wrong for the `contract` field. CLI/API/dashboard projections will improvise — exactly what the canonical diagnostic taxonomy is meant to prevent.
- [Major] **Dual-stamping `gc.formula_contract = "graph.v2"` has no defined sunset.** Workflow roots are required to dual-stamp "during the alias window," but the doc never says whether dual-stamping ends with source-alias removal in Phase 7, has its own readiness criteria (e.g. shared workflow-root predicate fully deployed across every root-consuming surface), or persists indefinitely on long-lived graph workflows. Long-running cities will outlive the alias window with ambiguously stamped roots, and the shared predicate's *legacy fallback* removal date is similarly unspecified.
- [Major] **Warning reach is gated on fresh compilation, which long-running cities never trigger.** Deprecation diagnostics fire at compile time. A long-running city with stable workflows will not surface `formula.contract_deprecated` until something forces a recompile (new sling, new convergence root, retry that recompiles a fresh formula). Operators who exclusively run through orders/daemon may go weeks without seeing the warning. The events-column for `formula.contract_deprecated` reads "suppressed warning event only for daemon/order paths" — ambiguous between rate-limited and fully-suppressed, and either reading widens the gap. The release captain therefore has no empirical signal that external usage is winding down before alias removal; the design substitutes "we kept a fallback" for "we measured remaining usage." This partly reintroduces the calendar-style deprecation that the criteria otherwise reject.
- [Major] **External `bd cook --persist` consumers — the population most likely to be SHA-pinned to legacy `contract`** — never see Gas City's diagnostics at all, because they are running `bd`'s own copy of the formula package (per `engdocs/proposals/formula-migration.md` §Ownership: "Beads retains: `internal/formula/` as a frozen copy"). The doc does not say whether `bd` is being taught `[requires]`, whether `bd cook` will surface `formula.contract_deprecated`, or whether the alias window includes a `bd`-version floor. Without that, "warning reach" excludes a real and migration-relevant audience.
- [Minor] **`OnceKey` deduplication scope is undefined per projection.** The diagnostic-projection rules pin CLI dedupe ("once per `(code, source path, source key)` per command invocation"), but controller, order dispatch, API, dashboard, and convergence projections do not pin a scope. A per-controller-restart scope makes long-running daemons emit one warning early then go silent for weeks; a per-event scope floods the bus during retries. Without explicit scopes, "warning reach" is non-deterministic and not reviewable. HA semantics (dedupe within one process or across replicas) are also unspecified.
- [Minor] **Criterion 1 ("two minor releases") is the closest cell to calendar-style deprecation in the gate.** It is the right thing for caller-integration soak time, but it does not test for remaining usage. The conjunction with criteria 2–4 saves it, but the rationale should make explicit that criterion 1 is *only* a soak-time floor, not evidence of migration completion — otherwise future readers may treat it as the calendar deprecation the design otherwise rejects.
- [Minor] **Old-binary forward compat for dual-declared formulas rests on an unverified assumption.** The matrix asserts old binaries "ignore `[requires]`" — but the doc cites no test or audit that current `bd`/`gc` binaries use non-strict TOML decoding for top-level tables. A regression test pinned against the oldest supported binary should accompany this design before dual-declaration becomes the migration linchpin. If TOML strictness was tightened in either repo, this assumption fails silently.
- [Minor] **Over-declaration is silent.** The validation contract says v2-only constructs *require* a v2 requirement, but never says whether a formula with no v2-only constructs is allowed to declare `requires.formula_compiler = ">=2"` defensively. Forward-compatible pack authors need this answer; leaving it to implementer discretion produces drift.
- [Minor] **Convergence subset validation is hedged.** The caller-inventory table says "Validate through `internal/formula` preflight **or** an adapter over `CompileResult`." The "or" allows two implementations and therefore drift between what convergence accepts and what the canonical compiler accepts — exactly the failure mode the canonical-result rule is meant to eliminate elsewhere.

**Missing evidence:**
- A concrete definition of the "compatibility branch" (scope, lifetime, backport policy, owner) — without it, criterion 3 cannot be enforced.
- The specification for `gc formula validate --all-packs --legacy-contract-report`: what it walks, what counts as "first-party legacy-only", how aspects/expansions/convergence-subset formulas are handled, and whether it gates CI or only the release checklist.
- An expanded contract×requires interaction matrix that pins down: empty `[requires]` + `contract`; `[requires]` with only unrelated keys + `contract`; dual + `>=3`; over-declaration on a non-v2 formula; whitespace/case variations in `contract`.
- A diagnostic code (and matrix row) for invalid `contract` values, distinct from `formula.compiler_requirement_unsupported` whose canonical message is wrong for the `contract` field.
- An explicit endpoint for `gc.formula_contract` dual-stamping and for the shared workflow-root predicate's legacy-fallback removal.
- A "warning reach" analysis covering long-running cities (cadence guidance for forced revalidation), order/daemon paths (precise scope of `OnceKey` dedupe), and external `bd cook --persist` consumers (whether `bd` is expected to surface the deprecation, and what `bd` version floor is required).
- A quantified expectation for deprecation-warning volume during the alias window. With first-party graph formulas dual-declared by mandate, every cook of a built-in formula emits `formula.contract_deprecated`. Without a ceiling under realistic load, "must not flood the event bus" is aspirational.
- Worked test cases for the failure modes real consumers will hit: legacy parent + new-style child via `extends`; dual-declared root + legacy-only aspect via `compose.aspects`; expansion template that contains a v2-only construct while the root carries only `contract`.
- A statement of the "minimum binary floor": neither a version, a commit SHA, nor a procedure for selecting one is provided in the design. The doc says it is "recorded in the release checklist and `docs/reference/config.md`," but the value/procedure is not in this design — and the release captain needs it before Phase 6 conversion can proceed.

**Required changes:**
1. Define the "documented compatibility branch" referenced in criterion 3: what it is, who owns it, what it backports, the consumer-facing migration step, and the criteria under which it can be retired. If no such branch will exist, replace the phrase in criterion 3 with the concrete fallback. The current text leaves the most consequential external-compat lever undefined.
2. Specify `gc formula validate --all-packs --legacy-contract-report` in full: sources walked (city packs, imported packs, aspects, expansion templates, convergence subset, test fixtures with explicit opt-out), the definition of "first-party legacy-only", the JSON shape's full schema, and exit-code semantics. Tie criterion 2 to the specified report.
3. Extend the compatibility section with an explicit edge-case table that resolves: `contract = "graph.v2"` + empty `[requires]`; `contract = "graph.v2"` + `[requires]` with only non-`formula_compiler` keys; missing `contract` + `[requires] formula_compiler = ">=2"` (already partially covered); dual + mismatched (`>=3`, `>=1`); over-declaration on a non-v2 formula; whitespace and case variations of `contract`. State the diagnostic outcome and code per row. The empty-table case must explicitly state whether the alias wins, the requires wins, or it is a conflict — the current ambiguity is exactly the silent-acceptance failure mode.
4. Add a diagnostic code (e.g., `formula.contract_invalid_value`) for `contract` with a value other than `graph.v2`, with its own canonical message and remediation. Reference it from the compatibility-with-`contract` rules so projections converge.
5. Define the sunset of dual-metadata stamping (`gc.formula_contract = "graph.v2"`). Either tie it to source alias removal in Phase 7, or state separate criteria (shared workflow-root predicate present in every root-consuming surface, including dashboard projection). Specify the retirement date for the predicate's legacy-fallback in the same change.
6. Add a "Warning reach" sub-section: cadence guidance for `gc formula validate --all-packs` in operator runbooks; recommendations for long-running cities to schedule periodic revalidation; an explicit acknowledgement (with mitigation) that external `bd cook --persist` consumers do not see Gas City diagnostics. If `bd` is being asked to surface the deprecation itself, say so and tie it to a `bd` version floor that is part of the alias-window definition.
7. Pin the `OnceKey` scope per projection: CLI = per command invocation, API = per request, controller/convergence = per controller restart with explicit re-emit cadence (or a one-line statement that there is no re-emit), order dispatch = per dispatch attempt or per day. State HA semantics (dedupe within one process or across replicas, and where shared state lives if across).
8. Annotate criterion 1 ("two minor releases") to clarify it is a soak-time gate for downstream integrators, not a usage-evidence gate. Future readers should not interpret it as the calendar-style deprecation the document otherwise rejects. Pair it with a real production-telemetry signal — for instance, `formula.contract_deprecated` event volume from instrumented deployments must fall below a stated threshold for N consecutive releases.
9. State explicitly whether over-declaration (`requires.formula_compiler = ">=2"` on a formula with no v2-only constructs) is permitted. The recommended answer is yes, with a test row in the validation matrix, so forward-compatible packs are sanctioned.
10. Pick one strategy for the convergence subset: preflight via `internal/formula` *or* adapter over `CompileResult`. Remove the "or". Convergence parity with the canonical compiler is exactly what the canonical-result rule is meant to enforce.
11. Add a one-sentence directive for external pack authors during the alias window: "Until Gas City vN is the minimum supported binary, dual-declare `contract` and `[requires]` in externally distributed graph packs." State the value of N, or commit to publishing it before alias removal. Without this, the alias-window contract for external authors is missing.
12. Add a regression test that loads a dual-declared formula with the oldest supported `bd`/`gc` binary and asserts compilation succeeds. Reference the test in the compatibility-matrix row for "Old binary reading dual-declared pack." If the test cannot pass, dual-declaration is not viable and the design must change (e.g., gate `[requires]` adoption on a binary floor that already understands the table).

**Questions:**
- What is the support commitment for the "documented compatibility branch"? Long-lived release line, shim package, or frozen tag with security-only patches? At what point may the commitment be retired?
- For external SHA-pinned packs that never migrate (abandoned forks, archived repos), does the design accept inevitable breakage at alias removal, or does the compatibility branch become effectively permanent?
- The formula-migration proposal still describes a `bd` shell-out path and a `GC_NATIVE_FORMULA=false` toggle. After Gas City removes the `contract` alias, must `bd cook` / `bd mol wisp` still accept `contract = "graph.v2"`? If yes, the alias-window definition must include a `bd`-version floor; if no, external `bd` consumers need a migration story before alias removal, not after.
- For dual-declared formulas (which are *intentionally required* for first-party packs until the binary floor moves), the design emits `formula.contract_deprecated` on every compile. Should that warning suggest *removing* `contract` as the canonical fix, given that doing so would violate the dual-declaration mandate? The current canonical message could push contributors to remove `contract` prematurely.
- Does `gc formula validate` exit non-zero on warnings (including `formula.contract_deprecated`) or only on errors? If non-zero, dual-declared first-party formulas mandated by the design will fail CI; if zero, criterion 4's "stale-guidance check" needs a separate mechanism that can fail builds.
- Who is the "release captain" — a designated person, an on-call rotation, or "whoever is shepherding this migration"? The alias window may outlive any one person's tenure; the role needs a succession protocol if criterion 3 hinges on a single owner's judgment.
- What happens if an external pack ships `[requires] formula_compiler = ">=2"` *only* (no `contract`)? It cannot compile on old binaries, but it is a legitimate authoring choice for a brand-new external pack that targets new gc only. Is the design's recommendation to dual-declare *external* packs unconditionally during the window, or is it acceptable for new external packs to require new-binary readers? If acceptable, what advisory landing zone tells external authors when it is safe to drop `contract`?
