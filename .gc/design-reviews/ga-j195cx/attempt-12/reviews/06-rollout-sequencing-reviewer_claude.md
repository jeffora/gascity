# Lena Driscoll - Claude

**Verdict:** approve-with-risks

**Top strengths:**
- Phases 1–2 are genuinely additive: parser, `NormalizedRequirements`, `CompileResult`, dual-stamping, and the shared workflow-root predicate land before any caller branching changes. A revert touches only Go code, never formula source files. Combined with Phase 3's dual-stamping fallback, the first three phases pass the "rollback without ripping up TOML" bar that the lane questions ask about.
- Alias-window exit criteria are measurable, not aspirational. `gc formula validate --all-packs --legacy-contract-report` is specified as a release-gate JSON command with concrete exit codes (`0`/`2`/`1`) and item shape; the "two minor releases of `[requires]` shipped" rule plus the CI stale-guidance check together turn "we're done migrating" into something CI can answer. The compatibility matrix in §"Compatibility With `contract`" enumerates the old-binary × `formula_v2` × source-shape combinations explicitly rather than hand-waving.
- The doc preserves dual-declaration for first-party graph formulas across the entire alias window and gates first-party requires-only conversion (Phase 6) on an explicit "minimum binary floor" recorded in the release checklist plus `bd` shell-out compatibility. Built-in syntax is never flipped before compatibility support ships, which directly defuses red flag #1.

**Critical risks:**
- **[Major] Phase 3 atomicity is undefined and the static guard forces a flag day.** Phase 3 enumerates eight caller surfaces (`internal/sling`, orders, API, convoy, convergence, molecule cook/cook-on, dashboard projections, tests) plus "add the static no-raw-consumer guard." The guard fails CI on any remaining behavioral use of `Contract` / `declaresGraphV2Contract` / `Requires.FormulaCompiler` / `gc.formula_contract` outside the parser, the compatibility metadata writer, the shared predicate, tests, fixtures, and docs. As written, that means the guard cannot land until the last caller is migrated — which makes Phase 3 a single landing window across parser, dispatch, convoy, API, dashboard, and convergence. The doc never says whether Phase 3 is one PR or many, and it does not name a sub-phase order. Lane question 1 is unanswered: if Phase 3 must land atomically, this is a flag day across parser, callers, and docs in the same release, and "main stays green" depends on a single very large PR.
- **[Major] Phase 6 first-party requires-only conversion is a multi-file revert.** Phase 6 removes `contract = "graph.v2"` from every first-party pack, example, test fixture, and tutorial. Reverting Phase 6 requires editing every one of those TOML files back, and there is no executable mechanism to do it in one step (no `gc formula migrate --revert`, no scripted dual-restore). The Rollout Plan acknowledges Phase 6 "requires an explicit release decision" but does not address rollback granularity. Lane question 2 fails for Phase 6: if a regression appears post-Phase-6 that traces to caller behavior, the operational fix path is "re-edit N first-party formulas." This is the textbook multi-step revert in red flag #2.
- **[Major] No release-floor enforcement on the alias window or dual-stamping.** The "minimum binary floor" is described as "recorded in the release checklist and `docs/reference/config.md` before any first-party requires-only conversion lands." That is a process commitment, not a test. There is no automated gate (build-time check, runtime version assertion, CI test) that fails if Phase 6 ships without the floor enforced, and no specification of where dual-stamping STOPS in metadata writers — the doc says new roots are dual-stamped "during the compatibility window" but never names the phase that ends it. Operators currently cannot answer "are we still dual-stamping?" from code without reading commit history. Red flag #3 (migration window lacks a measurable exit criterion) is partially defused by the legacy-contract-report tool but reappears for the dual-stamping side of the bridge.
- **[Minor] Docs/diagnostic visibility ordering is internally inconsistent.** §"Terminology And Documentation Contract" line "Docs and examples must be updated in the same release phase that makes diagnostics user-visible" conflicts with Rollout Plan Phase 5 ("Docs and examples: after parser and caller support ships"). Phase 3 is what makes diagnostics user-visible (it routes them through CLI/API/dashboard); Phase 5 lands docs in a later release. Either docs fold into Phase 3 or the doc must explicitly accept a one-release gap during which operators see diagnostic codes (`formula.compiler_requirement_unsatisfied`, `formula.contract_deprecated`, …) without canonical reference docs.
- **[Minor] `bd` shell-out compatibility precondition for Phase 6 is unanchored.** §"Compatibility With `contract`" line "Native compiler path | Uses `CompileWithResult` and canonical diagnostics" / "`GC_NATIVE_FORMULA=false` or `bd` shell-out path | Must either preserve dual `contract` declarations or be proven to parse `[requires]` identically before source conversion" correctly identifies the gating dependency, but the doc never says (a) who owns the `bd` parity proof, (b) what artifact records that proof, or (c) which formula-migration phase removes the `GC_NATIVE_FORMULA` toggle. Without that handoff named, Phase 6 has a hidden dependency on the parallel `formula-migration.md` Phase 4 that a release captain could miss.
- **[Minor] Phase 1 scope is ambiguous; the validation matrix may force a larger first PR than it appears.** The Required validation matrix is 11 rows and the diagnostic ordering is six numbered stages including "Legacy `contract` normalization and `contract`/`requires` conflicts" (stage 4) and "Missing compiler requirement for v2-only constructs" (stage 5). Phase 1 lists "add `Requires`, `NormalizedRequirements`, diagnostics, and validation tests." If Phase 1 must include the v2-only construct registry scan (§"Parser And Validation Contract" registry table) and the conflict diagnostic, Phase 1 is closer to the size of formula-migration's Phase 1 + Phase 2 combined. If only some stages land in Phase 1, the doc should name which. As written this is hard to estimate and easy to under-scope.

**Missing evidence:**
- No defined sub-phase order for Phase 3 caller migration. Sling vs. orders vs. convoy vs. convergence vs. dashboard have different rollback profiles (a sling regression is user-facing immediately; a dashboard regression is cosmetic until a workflow root with mixed metadata appears). The doc treats them as one bucket.
- No statement of when dual-stamping of `gc.formula_contract` is removed from metadata writers, nor what test asserts that removal. Phase 7 ("Alias removal") removes legacy `contract` parsing but the doc does not say whether the metadata writer keeps stamping `gc.formula_contract` for read-side compatibility on existing roots.
- No explicit minimum-binary-floor mechanism. Is it a string in `version.go`? A constant compared against a `gc.binary_version` metadata key? An entry in `pack.toml` `requires_gc`? §"Compatibility With `contract`" mentions `Pack 'requires' / 'requires_gc'` in the glossary but never wires it into Phase 6's gate.
- No tested mixed-version-controllers scenario in §"In-Flight And Convergence Behavior". The doc says "Mixed-version controllers sharing a bead store | New roots are dual-stamped until all supported readers use canonical keys" but the §"Required tests cover" list at the end of that section does not include a mixed-version scenario. Without that test, the dual-stamping invariant is unverified.
- No statement on what happens to in-flight wisps and active convergence loops when Phase 3 lands and the shared predicate replaces direct `gc.formula_contract` reads. The In-Flight section covers `formula_v2` flag flips but not metadata-key fallbacks during Phase 3 deployment.

**Required changes:**
- Split Phase 3 into named, ordered sub-phases (e.g., `3a` shared workflow-root predicate behind an interface; `3b` sling and CLI; `3c` orders; `3d` API/handlers and dashboard projection; `3e` convoy + convergence; `3f` add the static no-raw-consumer guard last) and state explicitly that the static guard is the closing PR of Phase 3. Each sub-phase must independently leave `main` green.
- Either (a) split Phase 6 into per-pack sub-phases (built-in workflows pack, examples, tutorials, test fixtures) so a regression rolls back one pack at a time, or (b) ship a `gc formula migrate --revert` / equivalent script with Phase 6 so first-party reverts are one command. Document the rollback procedure step-by-step in the Rollout Plan section, not as a side note.
- Promote the "minimum binary floor" from a release-checklist note to an enforceable gate. At minimum: define the version key (e.g., `gc version --requires-floor` output), declare which pack metadata field encodes it, and add a CI test that fails if Phase 6 PRs land without the floor field set. Without this, "before any first-party requires-only conversion" depends on a human reading the checklist.
- Name the phase that ends `gc.formula_contract` dual-stamping (likely Phase 7) and add a test that asserts post-Phase-7 roots carry only canonical keys. Add a test that asserts during the alias window roots are dual-stamped (currently implied, not enforced).
- Resolve the docs-timing contradiction: either fold §"Terminology And Documentation Contract" updates into Phase 3 (same release as user-visible diagnostics), or update the Rollout Plan Phase 5 to explicitly accept and call out the one-release gap and require diagnostic codes' canonical messages to be self-explanatory enough for that gap.
- Anchor the `bd` shell-out parity dependency: name the formula-migration phase that proves `bd` parity or removes the `GC_NATIVE_FORMULA` fallback, and make Phase 6 here cite that phase as a precondition. Add a row to the §"Additional compatibility gates" table: "First-party requires-only conversion | requires `bd` parity proof OR `GC_NATIVE_FORMULA` fallback removed (formula-migration Phase 4)".
- Tighten Phase 1 scope. List which validation matrix rows and which diagnostic-order stages must land in Phase 1 vs. deferred; specifically state whether the v2-only construct registry scan ships in Phase 1 (it must, otherwise `formula.compiler_requirement_missing` cannot fire and Phase 1's exit criterion is "no behavior change" not "validation tests pass").
- Add the missing required tests: mixed-version controllers reading dual-stamped roots; alias-window dual-stamp assertion; post-alias-removal canonical-only assertion; Phase 3 deployment with one process on canonical keys and another still on legacy.

**Questions:**
- Is Phase 3 intended as a single PR or as a sequence of sub-PRs? The Rollout Plan implies the latter ("each phase has a narrow rollback") but the static guard, as specified, requires the former. Which is it?
- What is the planned interval between Phase 4 (compatibility bridge stable) and Phase 6 (first-party requires-only)? "Two minor releases" is the documented exit criterion for the alias window writ large, but Phase 4 → Phase 6 is the more interesting boundary because that is when first-party formula source actually changes.
- Who owns the `bd` shell-out parity question, and where is the proof artifact stored? If `formula-migration.md` Phase 4 removes `GC_NATIVE_FORMULA`, that proof becomes moot, but that ordering is not stated here.
- For Phase 7 (alias removal), what is the rollback if external SHA-pinned packs surface that still rely on `contract = "graph.v2"`? The doc says external pinned packs "remain valid through the alias window" but Phase 7 ends the alias. Does the design assume zero external graph-v2 packs at that point, or is there a documented support extension path?
- Does the §"Required validation matrix" row for `contract = "graph.v2"` + `[requires] = ">=1"` (which lists `formula.compiler_requirement_conflict`) interact with Phase 4's "first-party graph formulas dual-declared"? Dual-declared first-party formulas use `>=2`, so this conflict only applies to malformed external packs — but the doc could call that out so reviewers don't read it as breaking the Phase 4 invariant.
