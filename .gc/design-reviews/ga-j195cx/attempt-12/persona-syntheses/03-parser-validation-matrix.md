# Priya Zielinski

**Persona verdict:** approve-with-risks

**Sources:** Claude, Codex

**Consensus findings:**
- [Major] The v2-only construct registry is still not mechanically implementable. Both reviews flag that the table duplicates "Graph workflow metadata keys" and mixes locations, metadata keys, and `gc.kind` values without a single encodable predicate. An implementer cannot generate exhaustive positive and negative tests without deciding which TOML paths are scanned, whether key presence or value matching is the trigger, and which locations each row covers.
- [Major] The validation plan is still a grouped checklist, not a normative matrix. Both reviews ask for explicit coverage across requirement expression, TOML value shape, `[requires]` table state, legacy `contract`, source location, v2-only construct presence, inherited or aggregated requirements, and host capability. Summary words such as "any", "no or yes", and "such as" leave too much room for hand-written case selection.
- [Major] Requirement expression validation needs a closed byte-exact rule. Claude sees the accepted/rejected string contract as example-driven; Codex says the intended v0 grammar is fail-closed. The synthesis is that the direction is correct, but the design must state the canonical rule directly: only `">=1"` and `">=2"` are accepted expression strings, with no trimming, whitespace normalization, case folding, Unicode normalization, or future-range parsing.
- [Major] Empty and default requirement semantics are under-specified. The design must distinguish omitted `[requires]`, an explicit empty `[requires]` table, absent `formula_compiler`, explicit `formula_compiler = ">=1"`, explicit `formula_compiler = ">=2"`, and `formula_compiler = ""`. Empty string must not be allowed to collapse into omission unless the design explicitly chooses and tests that behavior.
- [Major] Diagnostic precedence is only partially enforceable. Both reviews ask for combined-defect tests. Codex additionally notes that ordering within a precedence tier is undefined, while Claude notes that `formula.version_deprecated` is absent from the ordering model. The design must say whether validation aggregates all diagnostics or stops at the first fatal, and how warnings such as `formula.contract_deprecated` and `formula.version_deprecated` appear when fatal diagnostics also exist.
- [Major] Inheritance, aggregation, and attempted lowering need precise parser-contract coverage. Both reviews call out parent/child requirements, expansion and aspect maximum requirement aggregation, and attempted lowering below a parent requirement. Claude also asks whether `extends` chains inherit v2-only construct requirements and whether aspect-injected constructs are diagnosed at the aspect source.
- [Major] Construct-presence semantics are undefined. The design says v2-only fields require `>=2`, but not whether presence of a TOML key is enough or whether only non-default values count. Cases such as empty `check`, zero-valued `retry`, empty metadata values, and each `gc.kind` value must have positive and negative fixture expectations.
- [Major] The "regardless of caller path" guarantee needs a durable enforcement mechanism. Codex treats the current centralization and caller inventory as a strength; Claude correctly identifies that this does not prevent a future formula consumer from writing durable beads without `CompileWithResult` preflight. This is a risk rather than a block, but it needs a concrete guard.

**Disagreements:**
- There is no verdict disagreement. Both required sources return `approve-with-risks`, so this persona should approve the direction while requiring the parser-contract gaps to be closed before implementation is considered settled.
- Codex says the v0 grammar is already fail-closed and byte-exact enough; Claude says the prose still relies on examples and "such as" language. Assessment: the intended behavior is clear, but the document is not yet normative enough. Add the byte-exact accepted set and make all other strings invalid.
- Codex views centralization in `internal/formula` and the caller inventory as sufficient direction; Claude requires positive enforcement for future callers. Assessment: keep Codex's current-state confidence, but add a static, typed, generated, or integration-test guard so the guarantee survives new call sites.
- Claude emphasizes edge lexical cases, `extends`, and empty-string handling; Codex emphasizes matrix structure, TOML shape coverage, provenance, and diagnostic sorting. Assessment: these are complementary and should be merged into one fixture schema rather than treated as competing scopes.
- Codex asks whether an explicit empty `[requires]` table should be observable as user-authored provenance or normalized like omission. Claude focuses on making `formula_compiler = ""` invalid. Assessment: no conflict; the design needs two separate decisions, one for empty table provenance and one for empty string rejection.

**Missing evidence:**
- An encodable v2-only construct registry with columns for construct id, exact TOML path or path pattern, trigger predicate, scanned locations, positive fixture, and negative fixture.
- A formal requirement-expression grammar or equivalent closed accepted set, including an explicit rejection rule for empty strings and all whitespace variants.
- A normative table-driven test matrix that names every required dimension and identifies which Cartesian combinations are required versus derivable by smaller sub-matrices.
- A TOML-shape matrix covering absent table, empty table, inline table, dotted table, nested table, duplicate declaration, unknown sibling key, multiple unknown keys, invalid value type, unsupported string, and accepted string.
- A canonical raw TOML parsing strategy that preserves keys, raw value types, and source positions before typed decoding can erase the evidence needed for structured diagnostics.
- Combined-defect fixtures for invalid type plus unknown axis, unsupported expression plus legacy `contract`, missing v2 requirement plus disabled host capability, `contract` conflict plus disabled host capability, and multiple unknown `[requires]` keys.
- Deterministic diagnostic sorting within a tier, plus rules for warning emission when fatal diagnostics are present.
- Normalized provenance expectations for omitted requirements, empty `[requires]`, explicit `>=1`, explicit `>=2`, inherited parent requirements, expansion/aspect maximum requirements, and attempted lowering.
- A worked fixture for `extends` chains and for v2-only constructs contributed only through expansion or aspect formulas.
- A CI invariant that keeps the registry, parser behavior, and documented fixtures synchronized as new graph compiler constructs are added.

**Required changes:**
- Rewrite the v2-only construct registry as a test-generating table with `construct_id`, `toml_path`, `predicate`, `requires_graph_v2`, `locations_scanned`, `positive_fixture`, and `negative_fixture`. Resolve the duplicate "Graph workflow metadata keys" rows by making metadata key presence and metadata value matching separate, explicit predicates.
- Replace the validation checklist with normative sub-matrices for requirement expression and TOML shape, legacy `contract` compatibility, v2-only construct location, inheritance and aggregation, host capability, and combined-defect precedence. Require a table-driven harness whose fixture struct names every dimension explicitly.
- State the requirement expression rule as byte-exact matching against `{">=1", ">=2"}`. Document that no normalization is performed and that `formula_compiler = ""` is rejected with `formula.compiler_requirement_unsupported`.
- Define raw TOML handling before typed decoding: how unknown `[requires]` keys, non-string `formula_compiler` values, inline tables, dotted tables, nested tables, duplicate declarations, and malformed shapes map to structured diagnostics.
- Define diagnostic aggregation and sorting. State whether validation returns all diagnostics or first fatal diagnostic for each caller surface, how ties are sorted, and where `formula.contract_deprecated` and `formula.version_deprecated` appear relative to fatal requirement diagnostics.
- Define default, provenance, and inheritance semantics for absent `[requires]`, empty `[requires]`, explicit `>=1`, explicit `>=2`, legacy `contract`, parent requirement inheritance, child attempted lowering, expansion/aspect maximum aggregation, and `extends` chains.
- Define construct-presence predicates for every v2-only registry row. For each row and each scanned location, require at least one positive fixture that fails without `>=2` and one negative fixture that does not require `>=2`.
- Add a durable caller-path enforcement mechanism. Acceptable options include a typed wrapper that only `CompileWithResult` can construct, a generated registry of formula consumers checked in CI, or integration tests that force every durable formula write path through normalized compile results.
- Add or explicitly reuse a diagnostic code for "child requirement lower than parent" and include it in the matrix.
- State when `formula.compiler_requirement_missing` is evaluated relative to expansion/aspect aggregation, and ensure diagnostics point at the contributing formula or aspect source rather than only the compiled root.
