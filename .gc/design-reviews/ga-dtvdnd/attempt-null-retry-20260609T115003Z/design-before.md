# Requirements: Core and Gastown Pack Split

## Problem Statement

Gas City still carries SDK-required pack behavior, generic maintenance behavior,
and Gas Town-specific orchestration behavior across legacy source-tree pack
locations:

- `internal/bootstrap/packs/core`
- `examples/gastown/packs/maintenance`
- `examples/gastown/packs/gastown`

That layout makes ownership ambiguous. A non-Gastown Gas City deployment can
appear to depend on Maintenance or Gastown assets, while Gastown behavior can
appear to be supplied by in-tree examples instead of by the public Gastown pack.
It also obscures which behavior is required SDK infrastructure and which
behavior is user-supplied orchestration configuration.

The desired product outcome is a role-neutral Gas City runtime in which Core is
the only Gas City-owned pack required for normal `gc` operation, Maintenance is
retired as a standalone active pack, and Gastown behavior is loaded explicitly
from the public `gascity-packs/gastown` pack. Existing operators must receive
actionable, non-interactive diagnostics and repair guidance for legacy pack
references. Gastown operators must not lose supported behavior when Core assets
are generalized or when Gastown behavior moves to the public pack authority.

The migration is not ready for implementation approval until the named support
artifacts for the asset migration ledger, behavior-preservation proof, public
Gastown validation, rollout/version-skew policy, diagnostic schema, and
acceptance proof matrix exist and pass. Those artifacts are acceptance
evidence, not inline implementation design.

## W6H

| Dimension | Requirement |
| --- | --- |
| Who | Gas City maintainers need an auditable Core pack boundary; Gas City operators need stable non-Gastown runtime behavior; Gastown operators need explicit public Gastown imports; pack authors need clear ownership rules; existing-city upgraders need safe migration diagnostics. |
| What | Split the current mixed pack behavior into required Core SDK behavior, retired Maintenance behavior, and explicit external Gastown behavior. Core must stay role-neutral; Gastown roles, workflows, prompts, commands, overlays, and role-specific recovery behavior belong to the public Gastown pack. |
| When | Fresh `gc init` output must use the new model immediately after the migration ships. Existing cities must be diagnosable before startup or operator workflows fail from retired pack paths. The public Gastown pack and Gas City release must be version-compatible before the migration is accepted. |
| Where | The migration applies to Gas City source-tree packs, generated/materialized system packs, public pack imports, city and rig config, docs, examples, doctor/import-state output, runtime state, lock/cache state, and tests. It also applies to external `gascity-packs/gastown` validation because that pack becomes the Gastown authority. |
| Why | Gas City's SDK must not encode Gastown roles or Maintenance as hidden framework behavior. Role behavior must be user-supplied pack configuration, while SDK infrastructure remains self-sufficient with only the controller and required Core runtime support. |
| How | Core is a required system-pack identity in resolved config for real cities, with trusted provenance that user imports cannot shadow. Gastown is an explicit import using `https://github.com/gastownhall/gascity-packs.git//gastown` with the pinned immutable `sha:` version used by fresh Gastown init. Maintenance is neither auto-included nor silently materialized. Diagnostics run in a bootstrap-only mode when normal pack resolution is broken, report missing Core or retired imports with exact source attribution, and use `gc doctor --fix --non-interactive` as the canonical mutating repair surface. |
| How much / scale | The migration covers every active pack asset under the three legacy pack roots, every generated/materialized system-pack source that can load those assets, every supported fresh-init template, all existing-city legacy import shapes, every supported public-pack lock/cache state, all support artifacts named in the acceptance criteria, and all docs/tests that present retired paths as authoritative. |

| --- | --- | --- | --- |

| Happy path | The asset ledger classifies a split asset such as a shared architecture template. | `plans/core-gastown-pack-migration/support/asset-migration-ledger.yaml` records stable asset and behavior IDs, the original path, Core-neutral successor, Gastown-specific successor if any, every output path or retirement action, stripped behavior, retained behavior, fallback classification, and proof commands. The behavior-preservation proof covers any moved Gastown-specific triggers, scripts, prompt variables, routes, notification paths, and recovery flows. | Ledger validator plus behavior-preservation proof row for the split asset. |

| Negative path | A user or third-party import tries to shadow required Core with another pack named `core`. | Runtime behavior fails closed with a collision diagnostic before behavior-changing operations execute. | Negative path | An existing city imports `packs/maintenance`, `packs/gastown`, `examples/gastown/packs/gastown`, or `.gc/system/packs/maintenance`. | Diagnostics classify the reference as retired or legacy, name the source that introduced it, distinguish required Core from optional Gastown, and do not silently redirect to a hidden fallback. `gc doctor --fix --non-interactive` rewrites only supported local mutable configs after preflight, and reports read-only or transitive import remediation without guessing policy. | Import-state and doctor tests covering each legacy path shape. |
| Negative path | A Core asset or generated Core metadata routes work to a literal Gastown role name such as Mayor, Deacon, Polecat, Refinery, Witness, Boot, Crew, or Gastown. | Role-neutrality checks fail unless the occurrence is in allowed migration documentation, generated review artifacts, or a negative/absence test fixture. | Absence-scan test over source plus generated/materialized outputs with positive and negative controls. |
| Edge case | A user renamed or disabled the default Core maintenance executor. | Deterministic SDK infrastructure still operates with only the controller. Optional agent-executed Core maintenance work is bound through configurable pack data; if no executor is available, the work remains visible or diagnosable rather than causing Go-side role special-casing. | Unit/integration test using a renamed executor and a no-executor city. |
| Edge case | The operator is offline while a Gastown city uses a lockfile for the public Gastown pack. | If the exact pinned external pack is already present in the repo cache with matching provenance and digests, resolution uses that cache and reports provenance. If not, resolution fails with an actionable missing-cache diagnostic and never falls back to in-tree examples, system packs, or bundled synthetic aliases. | Network-disabled offline cache-hit, cache-miss, digest-mismatch, missing-subpath, and stale-alias command tests. |
| Edge case | An existing city has in-flight sessions that were started from retired Maintenance or in-tree Gastown materialized content. | Diagnostic mode reports the in-flight dependency. Mutating repair refuses while live affected sessions or controllers can still consume the retired content. Operators must drain or restart those sessions before repair; after repair no new work may load retired paths. | Upgrade-state test with live-session detection and post-repair no-retired-load verification. |
| Edge case | Gas City and the public Gastown pack are out of the supported pin window. | New binaries with old locked packs and old binaries with activation-pinned packs fail closed before behavior-changing operations, with version-skew diagnostics, downgrade guidance, and rollback expectations from the public pin and version-skew support artifacts. Compatibility-pin windows are explicit and temporary. | Release matrix tests for old/new binary crossed with compatibility/activation pins. |
| Edge case | Stale `.gc/system/packs/maintenance` or old synthetic cache state exists after upgrade. | The stale state is ignored, pruned by an explicit repair action, or reported as retired state. It is never loaded as an active Maintenance pack. | Upgrade-state integration test or command verification. |

## Out Of Scope

- Redesigning MEOW primitives or adding new SDK primitives.
- Reworking the Gastown role model beyond moving Gastown behavior to the
  explicit public pack authority.
- Changing unrelated third-party packs except where their import metadata,
  tests, or documentation point at retired Maintenance or in-tree Gastown
  authority paths.
- Renaming unrelated Go packages, config fields, docs, or operational concepts
  that use "maintenance" for supervisor, store, cleanup, or reliability work
  rather than the retired Maintenance pack.
- Implementing file moves, code changes, or bead creation during the
  requirements phase.
- Replacing the public pack registry/import mechanism.
- Keeping source-code comments solely to preserve the history of removed pack
  locations.
- Treating Core as a new role system or adding framework-side decision logic
  about role behavior.
- Approving the implementation plan before the AC6, AC7, AC14, AC15, AC16, and
  AC17 proof artifacts exist and pass.
