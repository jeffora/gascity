# Mira Acharya - Codex

**Verdict:** block

**Top strengths:**
- AC3, AC7, AC13, AC14, AC15, AC16, and AC17 name the right evidence families: resolution matrix, behavior manifest, coverage transfer, public-pack validation, pin coherence, cache behavior, and an acceptance proof matrix.
- The examples already include the highest-risk regression classes: hidden in-tree Gastown fallback, implicit Maintenance loading, stale system packs, offline cache behavior, duplicate Core, and role-name leakage.
- The requirements correctly treat public Gastown validation as release evidence, not a local implementation detail.

**Critical risks:**
- [Blocker] The requirements artifact does not conform to its own `gc.mayor.requirements.v1` schema, so AC-to-test traceability is not enforceable. The schema requires top-level `User Stories`, `Technical Stories`, and EARS-style `Behavior Requirements`, but the file jumps from `W6H` to `Example Mapping` at `plans/core-gastown-pack-migration/requirements.md:70-82`. AC1 says the requirements-review gate must enforce that schema at `plans/core-gastown-pack-migration/requirements.md:107`, yet the current artifact omits the sections that would bind each acceptance criterion to concrete actors, invariants, and regression witnesses. Without those sections, AC17 can become a checklist of artifact names rather than a proof that every public-pack, Core-loader, and retired-source behavior has an executable witness.
- [Major] The two-repository regression relationship is implied but not yet stated as a versioned contract. AC4, AC14, and AC15 require an immutable public Gastown pin and public-pack proof, but the requirements do not explicitly say which tests live in Gas City, which tests live in `gascity-packs/gastown`, and which Gas City/Gastown version pairs each tier must run against. The local `/data/projects/gascity-packs` checkout currently has no `gastown/` directory, so the requirements need to make the external prerequisite concrete: `gascity-packs` owns Gastown behavior tests against a declared minimum/current Gas City compatibility target; Gas City owns loader/provenance/diagnostic tests against pinned pack fixtures; the release gate owns the candidate Gas City binary crossed with the exact activation SHA and digest.
- [Major] The nightly/inference lane is absent from the requirements, even though the migration is moving prompt/formula/order behavior out of process-local source. `TESTING.md` already defines `acceptance_c` live inference and worker profile mechanics, but the requirements do not require a Gastown-pack inference catalog, artifact output, or non-gating release-week signal. Pack-local unit tests and deterministic packcompat will prove configuration loads; they will not prove that moved prompts, role instructions, nudges, review checks, and recovery flows still behave when a real provider session receives them.

**Missing evidence:**
- No explicit `gascity-packs/gastown` source tree, behavior manifest, pin ledger, or packcompat transcript is available in the checked local public-pack repo yet.
- No requirements-level test ownership table exists for: Gas City PR CI, `gascity-packs` PR CI, deterministic cross-repo CI, live public-network release validation, and nightly/live-inference validation.
- No EARS-style behavior requirements bind AC13/AC14/AC17 to active execution evidence such as `go test -json`, pack-local test artifacts, packcompat transcripts, intentionally broken negative controls, and uploaded inference reports.

**Required changes:**
- Add the missing schema sections. Include user stories for Gas City maintainers, Gastown pack maintainers, Gastown operators, and existing-city upgraders; technical stories for cross-repo packcompat, pin/version-skew, coverage transfer, role-neutral absence scans, and live inference smoke coverage; and EARS-style behavior requirements with stable IDs.
- Add a requirements-level regression ownership contract. Recommended split: Gas City owns Core loader, required-pack participation, role-neutral absence scans, retired-source diagnostics, doctor/import-state repair, pin/cache provenance, and deterministic packcompat invocation; `gascity-packs/gastown` owns moved Gastown roles, prompts, formulas, orders, commands, scripts, checks, docs, and behavior-preservation witnesses; cross-repo release gates own compatibility and activation matrices using immutable SHAs, pack digests, and behavior-manifest digests.
- Add the version matrix to requirements, not only the implementation plan: Gas City PR CI runs against checked fixtures or an approved pinned public-pack cache; `gascity-packs` PR CI runs against the current supported Gas City compatibility target and the next Gas City candidate when available; release validation runs candidate Gas City x public Gastown activation SHA with in-tree fallback disabled; nightly inference runs current Gas City main/release-candidate x public Gastown main/activation pin and uploads transcripts without becoming a normal PR gate.
- Add a concrete test catalog by tier: pack-local Python/shell/unit tests for scripts and generated manifests; Gas City unit/scanner tests for resolver, cache, diagnostics, and role neutrality; testscript/command tests for `gc init --template gastown`, doctor/import-state output, and fallback-disabled failures; integration/packcompat tests for external pack loading and moved formula/order/script triggerability; nightly inference tests for a small set of real provider workflows covering work pickup, review/fix loop, status/reporting, and recovery/diagnostic behavior.

**Questions:**
- What is the minimum supported Gas City version that `gascity-packs/gastown` must test against before the activation release?
- Which live provider profiles are mandatory for release-week Gastown inference smoke: Claude only, Codex only, or the existing `claude/tmux-cli`, `codex/tmux-cli`, `gemini/tmux-cli`, and `opencode/tmux-cli` matrix?
