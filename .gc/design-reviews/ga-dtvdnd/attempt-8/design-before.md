# Requirements: Core and Gastown Pack Split

## Problem Statement

Gas City still carries SDK-required pack behavior, generic maintenance behavior,
and Gas Town-specific orchestration behavior across legacy source-tree pack
locations:

- `internal/bootstrap/packs/core`
- `examples/gastown/packs/maintenance`
- `examples/gastown/packs/gastown`

That layout makes ownership ambiguous. A non-Gastown Gas City deployment can
appear to depend on Maintenance or Gastown assets, while Gastown behavior can
appear to be supplied by in-tree examples instead of by the public Gastown pack.
It also obscures which behavior is required SDK infrastructure and which
behavior is user-supplied orchestration configuration.

For this migration, `internal/bootstrap/packs/core` is the sole canonical
Gas City source root for the release-bundled Core pack. Any generated,
materialized, cached, public, synthetic, fixture, or example path that exposes
Core-like content must either trace back to that source authority with
provenance and digest evidence, or be classified as non-runtime stale state,
test fixture, or compatibility shim that cannot satisfy the required Core
identity.

The desired product outcome is a role-neutral Gas City runtime in which Core is
the only Gas City-owned pack required for normal `gc` operation, Maintenance is
retired as a standalone active pack, and Gastown behavior is loaded explicitly
from the public `gascity-packs/gastown` pack. Existing operators must receive
actionable, non-interactive diagnostics and repair guidance for legacy pack
references. Gastown operators must not lose supported behavior when Core assets
are generalized or when Gastown behavior moves to the public pack authority.

The migration is not ready for implementation approval until the named support
artifacts for the asset migration ledger, source-consumer closure, behavior
preservation proof, public Gastown validation, rollout/version-skew policy,
diagnostic schema, role-neutrality scan, and acceptance proof matrix exist,
are wired into executable gates, and pass. Those artifacts are acceptance
evidence, not inline implementation design.

The required pack-resolution product contract is that Core has exactly one
canonical Gas City-owned identity and provenance: the release-bundled Core pack
payload from `internal/bootstrap/packs/core`, represented in resolved config as
a required system layer with source, version or digest, materialized path,
lock/cache provenance, and collision state. User, city, rig, root, public,
cache, overlay, and transitive imports may add optional packs but may not
replace, alias, shadow, or silently satisfy the required Core identity. Missing
Core blocks normal behavior-changing runtime operations while leaving bootstrap
diagnostics and explicit repair available. Stale materialized Core copies,
synthetic public aliases, duplicate pack names, same-named assets,
provider-conditioned support packs, and transitive diamond conflicts must have
deterministic matrix rows that either name the winning source with full
attribution or fail closed before any hidden fallback can run.

## W6H

| Dimension | Requirement |
| --- | --- |
| Who | Gas City maintainers need an auditable Core pack boundary; Gas City operators need stable non-Gastown runtime behavior; Gastown operators need explicit public Gastown imports; pack authors need clear ownership rules; existing-city upgraders need safe migration diagnostics. |
| What | Split the current mixed pack behavior into required Core SDK behavior, retired Maintenance behavior, and explicit external Gastown behavior. Core must stay role-neutral; Gastown roles, workflows, prompts, commands, overlays, and role-specific recovery behavior belong to the public Gastown pack. |
| When | Fresh `gc init` output must use the new model immediately after the migration ships. Existing cities must be diagnosable before startup or operator workflows fail from retired pack paths. The public Gastown pack and Gas City release must be version-compatible before the migration is accepted. |
| Where | The migration applies to Gas City source-tree packs, generated/materialized system packs, public pack imports, city and rig config, docs, examples, doctor/import-state output, runtime state, lock/cache state, and tests. It also applies to external `gascity-packs/gastown` validation because that pack becomes the Gastown authority. |
| Why | Gas City's SDK must not encode Gastown roles or Maintenance as hidden framework behavior. Role behavior must be user-supplied pack configuration, while SDK infrastructure remains self-sufficient with only the controller and required Core runtime support. |
| How | Core is a required system-pack identity in resolved config for real cities, with `internal/bootstrap/packs/core` as the canonical source root, trusted provenance, a required system-layer representation, and collision state that user imports cannot shadow. Gastown is an explicit import using `https://github.com/gastownhall/gascity-packs.git//gastown` with the pinned immutable `sha:` version used by fresh Gastown init. Maintenance is neither auto-included nor silently materialized. Pack resolution uses the explicit matrix contract for missing Core, stale Core, same-named assets, public synthetic aliases, duplicate identities, root/city/rig imports, overlays, provider-conditioned `bd`/`dolt` support packs, locks, caches, and transitive diamond conflicts. Diagnostics run in a bootstrap-only mode when normal pack resolution is broken, report missing Core or retired imports with exact source attribution, and use `gc doctor --fix --non-interactive` as the canonical mutating repair surface. |
| How much / scale | The migration covers every active pack asset under the canonical Core source root and the legacy Maintenance/Gastown roots, every Go embed package, builtin registry function, materialization command, direct hook/import path, generated or synthetic layout/hash, test fixture, supported fresh-init template, existing-city legacy import shape, public-pack lock/cache state, support artifact named in the acceptance criteria, and doc/test path that presents retired sources as authoritative. |

## Out Of Scope

- Redesigning MEOW primitives or adding new SDK primitives.
- Reworking the Gastown role model beyond moving Gastown behavior to the
  explicit public pack authority.
- Changing unrelated third-party packs except where their import metadata,
  tests, or documentation point at retired Maintenance or in-tree Gastown
  authority paths.
- Renaming unrelated Go packages, config fields, docs, or operational concepts
  that use "maintenance" for supervisor, store, cleanup, or reliability work
  rather than the retired Maintenance pack.
- Implementing file moves, code changes, or bead creation during the
  requirements phase.
- Replacing the public pack registry/import mechanism.
- Keeping source-code comments solely to preserve the history of removed pack
  locations.
- Treating Core as a new role system or adding framework-side decision logic
  about role behavior.
- Approving the implementation plan before the AC6, AC7, AC14, AC15, AC16, and
  AC17 proof artifacts exist and pass.
