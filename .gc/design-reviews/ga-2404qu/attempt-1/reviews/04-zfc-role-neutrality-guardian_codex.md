# Ingrid Kovac

**Verdict:** block

**Top strengths:**
- The design has moved beyond naive token scanning. The role-surface inventory covers production Go, Core/provider/public-pack assets, generated OpenAPI/dashboard output, generated docs, comments, scripts, prompt prose, metadata keys, and sub-identifiers such as `DogTheme`.
- The host-Core binding model is directionally right: Core owns `core.maintenance_worker`, public Gastown patches symbolically, and active resolution must use binding references instead of `dog` literals.
- Public Gastown preservation is tied to exact pinned commits, packcompat witnesses, and generated ledgers, which is the right evidence model for avoiding silent role-behavior loss.

**Critical risks:**
- [Blocker] The scanner contract conflicts with the production Gastown adapter the design still keeps. The design says production Go under `cmd/gc` and `internal/` is in scanner scope and that hardcoded `dog`, `mayor`, `deacon`, `witness`, `refinery`, `polecat`, `boot`, `crew`, and `gastown` surfaces must move to configured resolution or narrow compatibility rows; those rows are limited to historical fixtures, docs examples, old-pack diagnostics, or Core `maintenance_worker` compatibility data (design.md:1910-1927). It also says Go identifiers/string literals for Gastown have no allowed mechanism in production Core/SDK logic (design.md:2158-2168), and specifically rejects `gastown` template wiring and public-pack constants used by Core unless a row allows them (design.md:2486-2494). But the migration still depends on production Go constants and template wiring: `internal/config/public_packs.go` defines `PublicGastownPackSource` and `PublicGastownPackVersion`; `internal/config/config.go` builds `GastownCity` with hardcoded `"gastown"` imports; `cmd/gc/cmd_init.go` offers `gc init` template `"gastown"`; and `cmd/gc/import_state_doctor_check.go` rewrites legacy imports to hardcoded public Gastown. The design does not name an allowed role-surface class for this adapter path. As written, the scanner either fails the intended migration or the allowlist has to grow by convenience, which is exactly the failure mode this lane is meant to block.
- [Major] SDK self-sufficiency is asserted but not proved per Core maintenance operation. The design lists generic infrastructure orders/scripts moving into Core, including gate sweep, route nudge, blocker-close cascade, orphan sweep, order tracking, spawn-storm detection, wisp compaction, JSONL export, stale cleanup/reaper, and binary doctor checks (design.md:2790-2803). It then allows orders to target the Core maintenance agent while saying controller-owned SDK operations must not depend on that agent (design.md:2838-2845), and requires broad renamed/omitted-worker tests (design.md:2847-2852). That is not enough to prove self-sufficiency. Each moved Core order/formula/script needs an ownership row that classifies it as either controller-owned SDK infrastructure that must work with no worker, or optional worker maintenance that may be disabled with a diagnostic. Without that per-operation split, essential infrastructure can accidentally become `dog`-bound pack behavior.
- [Minor] The older prose still describes `dog` as "now the Core maintenance agent" and allows `mol-dog-*` names for compatibility (design.md:2815-2822). Later sections correctly move resolution to `core.maintenance_worker`, but implementation readers could still treat the formula/order name as semantic. The design should state that retained `mol-dog-*` and `DOG_DONE` strings are compatibility identifiers only, never target-resolution inputs, and each retained active string needs a role-surface row.

**Missing evidence:**
- A sample `role-surface.generated.yaml` row set for production Gastown adapter surfaces: `PublicGastownPackSource`, `PublicGastownPackVersion`, `GastownCity`, `gc init --template gastown`, legacy import rewrites, and the current default `mayor` minimal-city scaffolding.
- A per-Core-maintenance asset table showing no-worker behavior, renamed-worker behavior, proof test, and whether the operation is SDK-required or optional pack maintenance.
- Negative scanner fixtures proving production `cmd/gc` or `internal/` cannot add a new role literal by hiding it in sub-identifiers, generated CLI examples, generated OpenAPI/dashboard strings, formula metadata, or Core prompt fragments.

**Required changes:**
- Add an explicit disposition for the production public-Gastown adapter. Either move it out of production SDK logic into user-supplied config/catalog data, or create narrow, owned, expiring role-surface rows for every remaining production `gastown`/`PublicGastown*`/template/init/doctor surface with proof tests. Do the same for current non-Gastown default `mayor` scaffolding and fallback prompt surfaces.
- Add a Core maintenance self-sufficiency matrix for every Core-owned order/formula/script. Any operation required for beads, events, gates, health patrol, order dispatch, routing, or session/controller correctness must have no-worker controller-backed proof. Optional worker-bound maintenance must be explicitly disabled or skipped when `core.maintenance_worker` is omitted, with a named diagnostic and no hidden dependency.
- Normalize `dog` wording so the binding is the only semantic target. Retained names such as `mol-dog-jsonl`, `mol-dog-reaper`, and `DOG_DONE` must be compatibility rows with tests proving dispatch, routing, mail, and nudge behavior resolve through configured bindings/recipients, not through string inference.

**Questions:**
- Is `gc init --template gastown` intended to be a permanent first-party product affordance, or a time-bounded migration adapter? The scanner and row-expiry model need that decision.
- When `core.maintenance_worker` is omitted, are worker-bound Core orders unscheduled, skipped at trigger evaluation, or allowed to create failed dispatch attempts? The design should pick one observable behavior per order class.
