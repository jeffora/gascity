# Felix Moreau

**Persona verdict:** block

**Sources:** Claude, Codex, DeepSeek; supplemental Gemini artifact present and reviewed

**Consensus findings:**
- [Blocker] The docs/DX inventory command is narrower than the release-gate contract. All sources flag or depend on this gap: the proposed `rg` command misses `.mdx`, JSON, TXT, docs navigation, generated schema/reference artifacts, and some generated CLI/help or dashboard text while the design claims an extension-agnostic wording lint. This turns known stale operator surfaces into late CI failures or missed docs regressions.
- [Blocker] The canonical public reference contract is not concrete enough. Claude and Codex require a named public Gastown companion reference and validator; DeepSeek/Gemini also flag ambiguous handling of `docs/reference/system-packs.md` and `docs/docs.json` navigation. The design must state whether the system-packs page is created or updated in the target tree, register it in navigation, and name the exact public Gastown companion path verified from the pinned checkout.
- [Major] Operator terminology is still inconsistent around the most sensitive concepts. The reviews flag collisions among `maintenance_worker`, `core.maintenance_worker.*`, "Core maintenance worker", "maintenance agent", `dog`, and `dog` as a pool. Operator-facing text should use one concept phrase that matches the config key, describe `dog` as a Core-provided worker pool where appropriate, and reserve historical/compatibility terms for explicit matrix-allowed contexts.
- [Major] Public Gastown source strings and import examples are not unified. Claude flags the mismatch between bare `github.com/gastownhall/gascity-packs/gastown`, `.git//gastown`, and `/tree/main/gastown`; DeepSeek/Gemini add stale `includes = ["packs/gastown"]`, `[imports.maintenance]`, and dead in-tree Gastown links. The design needs one copyable canonical source form and a named worklist for replacing non-install browser URLs and retired local paths.
- [Major] Known operator-facing docs and examples remain outside the explicit update scope. The raw reviews name stale or fragile surfaces including `docs/getting-started/troubleshooting.md`, `docs/troubleshooting/gc-start-walkthrough.mdx`, tutorials 01/05/07, `docs/getting-started/coming-from-gastown.md`, `docs/guides/shareable-packs.md`, `docs/guides/migrating-to-pack-vnext.md`, `docs/reference/cli.md`, `docs/schema/pack-schema.{json,txt}`, docs navigation, `examples/swarm/city.toml`, `examples/dolt/pack.toml`, `examples/dolt/assets/scripts/port_resolve.sh`, and Gastown scripts such as `status-line.sh`.
- [Major] The design's seeded doctor wording conflicts with its case-aware rule. The proposed phrase "maintenance is retired..." lowercases the retired pack name while the matrix reserves capital "Maintenance" for the retired pack and lower-case "maintenance" for ordinary English. The doctor seed and goldens should use a matrix-approved phrase such as "Maintenance is retired; Core supplies generic maintenance..."
- [Major] Generated docs, schema, and help artifacts are operator-facing contracts and must be regenerated and linted with the same terminology rules. Reviewers specifically call out schema examples such as `"../maintenance"` and ask that generated-doc freshness gates run in the same slice as source wording changes.
- [Major] Tutorial and troubleshooting integrity is not proven strongly enough. Later tutorials and walkthroughs still reference moved or template-specific assets such as `mol-dog-*`, `prune-branches`, local `packs/gastown`, and `.gc/system/packs/gastown`. The docs gate must prove a new operator can complete affected Core-only and public-Gastown flows without retired local pack paths or contradictory Maintenance guidance.
- [Major] Some visible docs decisions depend on unresolved cross-lane mechanics. DeepSeek identifies runtime-state path handling, `[[patches.agent]] name = "dog"` resolution against required Core, and moved-order qualification/pool resolution as semantic questions. Those can be owned by other lanes, but Felix's lane cannot approve the docs plan until the resulting operator-visible behavior is named.

**Disagreements:**
- Verdicts split: Claude and Codex both say `approve-with-risks`, while DeepSeek and the supplemental Gemini artifact say `block`. Assessment: the persona verdict is `block` because the shared inventory, terminology, canonical-reference, and tutorial/troubleshooting gaps are concrete operator-facing hazards, not merely editorial polish.
- Claude reports `docs/reference/system-packs.md` exists in the design's own `rig_root` docs tree, while DeepSeek/Gemini report that the page does not exist in the active checkout they inspected. Assessment: the discrepancy is itself release-risk signal. The design must name the intended target tree, create-versus-update action, and docs-nav assertion.
- Codex treats the current design as approvable if the promised generated wording matrix and release gates are made real. DeepSeek/Gemini treat the known blind spots as blockers now. Assessment: keep the matrix/gate architecture, but require the known high-risk files and generated artifacts to be seeded into the gate before approval.
- DeepSeek treats runtime-state migration, cross-pack `dog` patching, and moved-order qualification as blockers; Claude and Codex mostly keep this lane centered on documentation wording. Assessment: those are cross-lane implementation decisions, but docs, troubleshooting commands, and tutorial output depend on them, so the docs lane requires the visible behavior to be specified.
- Store/Dolt maintenance terminology is not disputed as valid, but reviewers differ on how much is already protected. Assessment: retain `gc maintenance`, `[maintenance.dolt]`, Dolt maintenance docs, store-maintenance terms, and generated schema identifiers only through classified allowlist rows.

**Required changes:**
- Replace the narrow inventory command with the generated wording-linter invocation, or explicitly label the `rg` command as a non-authoritative sanity check and expand it to include all enforced formats and generated artifacts.
- Make the system-pack reference anchor concrete: create `docs/reference/system-packs.md` and register it in `docs/docs.json`, or designate an existing registered page; add nav/link freshness tests before behavior-changing slices rely on it.
- Name the exact public Gastown companion reference path and require Gas City CI to lint it from the pinned public Gastown checkout.
- Standardize operator terminology around the Core maintenance-worker concept. Describe `dog` in tutorials as a Core-provided background utility worker pool, and reserve `dog`/`maintenance_worker`/historical Maintenance references for matrix-approved contexts.
- Pin one canonical public Gastown source string and update shareable-pack, migration, CLI, troubleshooting, and coming-from-Gastown examples away from retired local paths and GitHub browser URLs.
- Correct the seeded import-state doctor phrase to use capital "Maintenance" for the retired pack and register it as a golden phrase shared by doctor output and docs.
- Add the known high-risk docs, examples, generated artifacts, and scripts to the update scope or structured inventory gate, including the troubleshooting, tutorial, shareable-pack, migration, schema, CLI, docs-nav, swarm, dolt, and Gastown script files named by the reviews.
- Define tutorial transcript tests as executable flows, not only prose matching, for the affected onboarding and troubleshooting paths; assert that retired local pack paths and contradictory Maintenance guidance do not appear.
- Regenerate schema/help/reference docs from source wording changes in the same slice and classify every retained `maintenance`, `system/packs`, `runtime/packs`, `gastown`, `PublicGastown`, `dog`, and `Core` hit.
- Resolve runtime-state path handling, public Gastown `dog` patching, and moved-order qualifiers/aliases in their owning lanes, then reflect the chosen behavior in tutorials, troubleshooting, and public-pack comments.
