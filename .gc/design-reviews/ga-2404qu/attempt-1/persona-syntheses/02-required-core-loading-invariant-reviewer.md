# Elias Sato

**Persona verdict:** block

**Sources:** Claude, Codex, DeepSeek V4 Flash

**Consensus findings:**
- [Blocker] The required-Core invariant still has contradictory doctor behavior. Codex and DeepSeek both identify that the Core Presence Doctor section tells plain `gc doctor` to use `internal/systempacks.LoadRuntimeCity`, while the read-only doctor contract forbids materialization, repair, cache promotion, quarantine, or TOML rewrites in report-only diagnostics. This would let diagnostics hide the corruption they are meant to report.
- [Blocker] The resolver participation proof is asserted more strongly than it is designed. Claude shows that existing `internal/config` provenance is binding-name/source-string based, not a resolver-produced, digest-keyed participation record with stable layer/import-edge identity. Codex asks for the corrected plain-doctor and provider proof flow; DeepSeek accepts the typed-participation direction but still relies on the design specifying how that proof is emitted and validated. The design must make the resolver API, owning package, digest source, and post-resolution validation contract concrete.
- [Blocker] Loader bypass containment is not yet implementable as written. All reviewers require a generated default-deny inventory, but the design still mixes a substring-test model with whole-production reachability claims, misses or contradicts real loader symbols, and lacks a named generator/test entrypoint. Known risky paths remain insufficiently classified, including `internal/dispatch/control.go:loadAttemptRouteConfig`, controller reload, no-refresh completion/stop paths, API/raw config readers, doctor/configedit readers, provider prepasses, and helper wrappers.
- [Blocker] The scanner symbol contract is stale and incomplete. The design still references non-existent `config.LoadCity`, while real production bypass surfaces include `config.Load`, `config.LoadWithIncludes`, `config.LoadWithIncludesOptions`, raw TOML pre-reads, package aliases, wrapper helpers, and exported config-returning helpers. A stale symbol list undermines the default-deny guarantee.
- [Major] Provider host-pack selection is still underspecified. Codex requires an exact `RequiredPackPlan` algorithm covering layered config, fragments/includes, env overrides, malformed partial config, and prepass/final mismatch. DeepSeek instead recommends validating the full embedded host-pack set (`core`, `bd`, `dolt`) before resolution to avoid the bootstrap paradox. The design must choose one concrete algorithm.
- [Major] Loud-failure semantics are not fully uniform across loader classes. Claude calls out a live behavior-driving path that loads without Core and swallows errors; Codex flags plain doctor materializing by accident; DeepSeek flags runtime repair races and read-only filesystem behavior. The design needs one matrix for missing, corrupt, partial, stale, extra-file, materialized-but-unresolved, no-refresh, diagnostic, repairable, and read-only cases.
- [Minor] Several operational details remain unanchored: the second production `GC_BOOTSTRAP` consumer in `internal/doctor/implicit_import_cache_check.go`, path normalization for participation comparison, empty-pack participation, stale inventory rows, and newly exported config loaders.

**Disagreements:**
- There is no verdict disagreement: Claude, Codex, and the current DeepSeek V4 Flash artifact all block.
- DeepSeek treats the typed participation direction as mostly solved by the Attempt 17 contract, while Claude says the resolver that exists cannot yet emit the proof the design demands. Assessment: the design can keep the typed proof direction, but it must add the concrete `internal/config` API and tests before this lane can approve.
- Codex frames provider selection as a missing `RequiredPackPlan` prepass algorithm; DeepSeek argues for validating all embedded host packs up front to avoid a prepass. Assessment: both can satisfy the invariant if specified rigorously, but leaving both implied is unsafe.
- The reviewers vary on scanner implementation depth. Claude wants AST/call-graph reachability or a narrowed guarantee; Codex wants a named generator, schema test, freshness test, and bypass fixtures; DeepSeek stresses symbol-list correction. Assessment: the design should specify a generated AST-based inventory for direct calls and wrapper/function classifications, with any remaining call-graph limitation made explicit.
- DeepSeek asks for runtime self-heal locking and an embedded VFS/read-only fallback. Claude and Codex mainly demand fail-closed/no-refresh behavior and read-only diagnostics. Assessment: the design must choose whether ordinary runtime loaders repair or fail closed; either choice needs explicit concurrency and immutable-filesystem behavior.

**Missing evidence:**
- A concrete `RequiredSystemPackParticipation` contract: package ownership, fields, producer, carrier in or beside `config.Provenance`, digest source, import-edge/layer identity, implicit include handling, and behavior for shadowed, overridden, empty, corrupt, partial, and materialized-but-unresolved Core.
- A checked-in/generated `loader-inventory.generated.yaml` equivalent with every current production config-read call site classified as runtime wrapper, no-refresh runtime wrapper, edit-only partial read, doctor diagnostic, provider prepass, raw display/API read, or deleted.
- The scanner generator command/package/test names and fixture suite for aliases, wrapper methods, function values, package aliases, raw TOML decoders, hand-built include lists, non-OS filesystem loads, `LoadWithIncludesOptions`, and newly exported resolver helpers.
- A corrected plain-doctor flow proving `gc doctor` report-only mode uses no-refresh/read-only validators and reserves materialization, quarantine, prune, promotion, repair, and runtime `LoadRuntimeCity` validation for explicit fix paths.
- A provider-pack selection algorithm with tests for root config, rig config, fragments/includes, env overrides, malformed partial config, `bd`, `dolt`, exec/no-provider cases, non-bundled providers, and prepass/final mismatch.
- A loader-class by failure-class matrix covering normal runtime loads, no-refresh loads, controller reloads, API/helpers, dispatch/control, completion, stop, configedit, doctor diagnostics, and provider prepasses.
- Tests for path normalization, empty effective contribution, extra files, stale rows, future allowlisted partial-helper reuse on behavior paths, read-only filesystem behavior, and concurrent runtime repair if runtime repair remains allowed.

**Required changes:**
- Rewrite the Core Presence Doctor section so plain `gc doctor` uses read-only/no-refresh validation APIs. `LoadRuntimeCity`, repair, cache promotion, quarantine, prune, import rewrites, and final post-fix validation must be limited to explicit fix/mutation paths.
- Add a resolver participation API subsection to System Pack Loading. Specify how `internal/config` emits digest-linked required-pack participation without importing `internal/systempacks`, how `internal/systempacks` validates it against pre-resolution file-set integrity, and how tests cover shadowed, overridden, empty, corrupt, partial, and materialized-but-unresolved Core.
- Make the generated loader inventory the source of truth for production config reads across non-test `cmd/gc` and behavior-driving `internal/` paths. It must reject unclassified `config.Load`, `config.LoadWithIncludes`, `config.LoadWithIncludesOptions`, raw TOML pre-reads, aliases, wrappers, function values, and exported config-returning helpers.
- Correct the scanner symbol set: remove phantom `config.LoadCity`, include `LoadWithIncludesOptions`, and add freshness tests that fail on stale rows or newly exported resolver functions.
- Classify the known risky paths explicitly, including `internal/dispatch/control.go:loadAttemptRouteConfig`, `tryReloadConfig`, `cityConfigIncludesWithBuiltinPacks`, `loadCityConfig*`, no-refresh completion/stop, `effectiveCityName`, API/raw config helpers, doctor/configedit reads, and provider prepass helpers. Each row needs an enforcement class, reason, and focused test.
- Choose and document the provider host-pack algorithm. Either validate the full embedded host-pack set before resolution, or define a constrained prepass plus post-resolution mismatch handling; then test the selected path exhaustively.
- Define uniform required-pack failure semantics for every loader class. Missing, corrupt, partial, stale-unusable, extra-file if in scope, and materialized-but-unresolved required Core must not degrade to warnings on behavior-driving production paths.
- Specify runtime repair policy. If ordinary runtime loaders self-heal, add lock/atomic publish/read-only fallback behavior and failure-injection tests; if they do not, make them fail closed and delegate mutation to `gc doctor --fix`.
- Name and retire all production `GC_BOOTSTRAP` consumers, including `internal/doctor/implicit_import_cache_check.go`, in the cleanup plan.
