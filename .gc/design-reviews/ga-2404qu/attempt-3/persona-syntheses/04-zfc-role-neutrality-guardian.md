# Ingrid Kovac

**Persona verdict:** block

**Sources:** Claude, Codex, DeepSeek V4 Flash, Gemini-labeled re-review

**Consensus findings:**
- [Blocker] The Go role surface is unguarded and unresolved. Reviewers converged on production Go role literals and role-derived control paths, including tmux role emoji/theme APIs, `MayorTheme`/`DeaconTheme`/`DogTheme`, default city scaffolding that names `mayor`, warmup mail defaults, prompt fallback to `prompts/mayor.md`, `internal/sling` heuristics for `mol-polecat-*` and `mol-refinery-patrol`, and API agent-kind classification for `crew`. The design adds a Core asset scanner, but does not define a Go scanner, migration table, replacement mechanism, rollout slice, or exception policy for these hits.
- [Blocker] The design gates source deletion on resolving role-themed/tmux Go APIs, but no rollout slice performs that work. This creates a deadlock: source deletion and public Gastown pin movement are declared blocked on a decision the plan never makes.
- [Blocker] The migration removes the only existing role-name guard without an equivalent replacement. `TestTmuxThemeScriptHasNoHardcodedRoleNames` leaves with the in-tree Gastown assets, while the proposed Core scanner covers pack assets only. That is a net regression in role-neutrality enforcement.
- [Blocker] `dog` containment contradicts the stated invariant. The design says `dog` is pack configuration, not a Go special case or SDK dependency, but live Go contains `DogTheme()`, Core/Maintenance orders assume a literal `dog` pool, and the current `dog` agent definition lives outside Core. The design does not decide whether Core ships a configurable maintenance agent, Core orders consume a maintenance-pool variable, or `dog` remains purely user-supplied.
- [Blocker] Core notification and maintenance target generalization is not specified per asset. Core-bound formulas, orders, prompts, skills, and scripts still contain or inherit role-specific targets such as mayor, witness, deacon, refinery, and polecat. The design states these must be removed or renamed by configuration, but does not choose the replacement pattern for each asset: formula variable, config-resolved target, user-supplied indirection, or Gastown overlay.
- [Major] The Core role-token scanner is described as an outcome, not a contract. It needs an exact root, file-type set, structured TOML field coverage, prohibited token list, path/token/field allowlist policy, `dog` handling, comment and metadata coverage, generated asset coverage, and negative fixtures. A broad whole-file scan or broad allowlist would miss the contamination patterns reviewers cited.
- [Major] Formula-name heuristics in Go violate ZFC even if the formulas move to Gastown. `internal/sling` deriving behavior from Gastown formula names is framework cognition; the replacement should be role-neutral formula metadata, explicit variables, or pack configuration consumed through generic APIs.
- [Major] Core self-sufficiency tests are not concrete enough. The design requires Core to work with the maintenance agent renamed or absent, but does not list which controller-owned operations must still work, which agent-executed maintenance operations may be disabled or retargeted, or how orders resolve without a literal `dog` pool.
- [Major] Rollout and registry changes need explicit gates. Reviewers called out the transition from required builtin packs `["core", "maintenance"]` to `["core"]`, stale `.gc/system/packs/maintenance` directories, `All()` pack-list removal, public Gastown pin timing, embedded registry changes, and source-tree deletion ordering as production-safety requirements.

**Disagreements:**
- There is no verdict disagreement: every available raw review blocks.
- The main scope disagreement is how to treat Go de-roling. Claude, Codex, and DeepSeek require a Go role-token guard or migration table in this effort because the design gates source deletion on it. The Gemini-labeled re-review allows Go-source cleanup to be deferred only if the design explicitly removes it from the blocking migration scope and states that an asset-only scanner does not prove end-to-end role neutrality. The synthesis assessment is that either path can be valid, but the current design chooses neither.
- Reviewers differ on the cleanest `dog` model. Some accept Core shipping a minimal configurable maintenance-agent definition with rename/disable tests; others prefer Core orders taking a configurable maintenance pool and leaving `dog` as a Gastown convention. The required decision is the same: Core must not silently depend on an implicit literal `dog` pool.
- Reviewers emphasize different role-reference surfaces. Claude centers tmux/theme/config Go APIs; Codex emphasizes scanner breadth, maintenance scripts, and ownership audits; DeepSeek and the Gemini-labeled re-review add warmup/prompt fallbacks, `classifyAgentKind`, structured TOML scanning, stale Maintenance-pack transitions, and public-pack rollout ordering. These findings are additive, not contradictory.
- There is some naming ambiguity in the raw sources because the `gemini` artifact is titled as a DeepSeek V4 Flash independent re-review. Its content is still treated as an available raw review artifact because it is present in the persona review directory.

**Missing evidence:**
- A scope decision for Go de-roling: in scope for this migration or explicitly deferred.
- A non-test Go role-token inventory with each hit classified as control logic, user-facing example/help text, migration compatibility, generated docs, fixture, or allowed exception.
- A Go role-name migration table listing the replacement mechanism, rollout slice, and test for tmux themes/icons, `DogTheme()`, default city scaffolding, warmup defaults, prompt fallback, `sling` formula heuristics, `classifyAgentKind`, and required builtin pack names.
- A concrete `dog` ownership contract: Core-shipped configurable maintenance agent, formula-variable maintenance pool, config-resolved maintenance-agent indirection, or fully user-supplied Gastown convention.
- Tests proving a Core-only city loads, resolves Core orders/formulas, and runs controller-owned SDK operations when the maintenance agent is renamed or omitted.
- A scanner contract covering Core assets with path, field, and token precision; structured TOML fields; prohibited token set; `dog` allowlist format; comment/description/metadata coverage; generated asset coverage; and negative fixtures.
- A replacement for the deleted tmux role-name guard that covers relocated pack assets and the remaining Gas City Go surface, or an explicit statement that Go neutrality is tracked separately.
- A per-asset notification/maintenance target plan for every Core formula, order, prompt, skill, and script that currently names a role.
- Migration tests for removing Maintenance from required builtin packs while ignoring stale on-disk Maintenance state.
- Explicit ordering for public Gastown publication, `PublicGastownPackVersion` updates, embedded registry changes, `All()` removal, and source-tree deletion.

**Required changes:**
- Decide whether Go de-roling is in scope for this migration. If in scope, add a rollout slice, Go role-token scanner, migration table, replacement mechanisms, and tests. If deferred, remove the Go role APIs from the blocking migration table and state clearly that the asset guard does not prove end-to-end Gas City role neutrality.
- Add a Go role-name migration table covering at least tmux role icons/themes, `DogTheme()`, `DefaultCity`/`WizardCity`/`GastownCity`, warmup mail defaults, prompt fallback, `internal/sling` formula-name heuristics, `classifyAgentKind`, and required builtin pack names.
- Specify the role-neutral replacement for formula-name heuristics in `internal/sling`, such as formula metadata or explicit formula variables supplied through generic APIs.
- Resolve `dog` ownership. Either Core ships a minimal configurable maintenance-agent definition with rename/disable tests, or Core orders stop referencing literal `dog` and take a configured maintenance-pool target. Do not leave Core infrastructure dependent on a removed Maintenance-pack agent definition.
- Define the Core notification target contract for every Core formula, order, script, prompt, and skill that currently names a role. For each target, choose formula variable, config-resolved target, user-supplied indirection, or Gastown overlay, and prove it works when defaults are renamed or absent.
- Convert the Core role-name guard into a precise scanner contract: root, file types, TOML fields, embedded command text, prohibited tokens, path/token/field allowlists, `dog` treatment, comment/description/metadata coverage, generated asset coverage, and negative fixtures. Cite `mol-review-quorum.toml` as the reference pattern for role-neutral Core dispatch.
- Replace `TestTmuxThemeScriptHasNoHardcodedRoleNames` with relocated pack-asset coverage plus a Gas City Go-source guard, or document Go-source coverage as a separate tracked migration with no end-to-end neutrality claim in this design.
- Add a `requiredBuiltinPackNames` transition gate from `["core", "maintenance"]` to `["core"]`, with an upgrade test for cities that still have `.gc/system/packs/maintenance` on disk.
- Clarify rollout ordering so the public Gastown replacement is published and pinned before embedded registry changes, and the registry stops referencing Maintenance/Gastown before their source trees are deleted.
