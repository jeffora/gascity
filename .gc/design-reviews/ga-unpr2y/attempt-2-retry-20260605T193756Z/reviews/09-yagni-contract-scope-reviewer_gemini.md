# Kwame Asante - DeepSeek V4 Flash

**Verdict:** block

Lane: minimal vocabulary, facade creep, event-log deferral, backlog scope control. Assessed the design against code-level vocabulary mapping, cross-document consistency with REQUIREMENTS.md, per-slice vocabulary landing order, operability contract versus vocabulary scoping, and TR-006/TR-007 shaping of current API contracts.

**Top strengths:**

- The Vocabulary And Existing Contracts table (`DESIGN.md:64-86`) is a real YAGNI guard: every proposed term maps to a concrete existing type with a slice rule, and the "Avoid one broad `SessionFacts` type" rule plus rule-of-two shared-type guard provide structural checks the original draft lacked.
- The mutation landscape table (`DESIGN.md:113-142`) is unusually specific. Every production writer site, field family, target path, and exception status is enumerated. This gives the static guard a concrete allowlist to freeze.
- Backlog scope control is honest: slices 5 and 6 are explicitly blocked on missing proof files (`cmd/gc/scale_from_zero_test.go`, `cmd/gc/provider_health_gate_test.go`, `cmd/gc/session_progress_test.go` all verified absent), and the traceability matrix says so instead of hand-waving.

**Critical risks:**

- [Major] **`forbidden-kind` and `requires-materialization` are policy outcomes, not classification outcomes — they violate TR-006's own separation principle.** TR-006 says the classifier "separates what an input token is from what an operation is allowed to do with it" (`DESIGN.md:163`). But `negative_kind` includes `forbidden-kind` and `requires-materialization` (`DESIGN.md:193-194`), which are operation-policy decisions, not token-classification facts. The current code confirms this separation already exists: `internal/api/session_resolution.go` has a `materialize bool` option that controls whether a `configured-named-identity` token resolves to creation or `ErrSessionNotFound` (line 58 option struct), and `cmd/gc/cmd_mail.go` takes a completely different resolution path that never materializes. Whether a token is `forbidden` or `requires-materialization` depends on which surface is asking. If the classifier returns these values, it has already made the operation-policy decision that TR-006 says belongs to the adapter. This is not a naming issue; it is a domain-boundary violation that creates vocabulary (two negative kinds, plus the policy matrix that goes with them) before any slice needs it. Required change: remove `forbidden-kind` and `requires-materialization` from classifier `negative_kind`. The classifier returns what the token is; adapters decide whether each kind is allowed for their operation. Negative classifier results should be `not-found` and `ambiguous` only.

- [Major] **The classifier result type is a de facto god-struct that violates the anti-facade rule at the type level.** The typed result fields list (`DESIGN.md:189-195`) defines 11+ mostly-optional fields shared across all 11 result kinds with no per-kind population rules. The existing code already uses operation-specific result patterns: `ErrSessionNotFound` wraps the input string, `ErrAmbiguous` carries match details, and `IllegalTransitionError` carries from/command fields — each result type carries only the fields its operation needs (`internal/session/resolve.go:17-18`, `internal/session/state_machine.go`). If implementers follow the field list literally, every result kind carries `materializable`, `materialization_reason`, `template`, `config_target`, and `candidates[]` even though `direct-session-id` and `live-session-name` never need them. The vocabulary section says "each slice introduces only the fact family and result vocabulary needed for that operation" (`DESIGN.md:64`), but the result type is the opposite of that principle. Required change: add per-kind population rules or define the result type as a tagged union with kind-specific payloads. At minimum, add an explicit statement: "Result fields are populated only for kinds where the semantics require them; a flat struct with optional fields is the implementation choice, not a mandate to carry unused fields across all kinds."

- [Major] **The operability contract's "required diagnostic fields" list is a universal requirement that contradicts the per-slice vocabulary scoping rule.** The vocabulary section says "each slice introduces only the fact family and result vocabulary needed for that operation" (`DESIGN.md:64`) and a shared type is allowed only after two exact-use deciders. But the operability contract says "Every decision, command, and conflict result must carry" a list including generation, instance token, runtime session key, event emission result, subscriber/recovery path, work-query completeness, and five fact-freshness fields (`DESIGN.md:367-376`). For slice 1 (read-only target classification), generation, instance token, runtime session key, event emission result, and work-query completeness are meaningless — no mutation occurs, no runtime identity is involved, no events are emitted, and work-query completeness applies to reconciler paths only. If this contract is enforced literally on every result type, it forces a shared mega-diagnostics struct that violates the vocabulary section's own anti-facade rule. The two sections need explicit scoping reconciliation. Required change: add: "Operability fields are per-command-cluster: each decision/result type carries only the evidence fields its adopted surfaces consume today. The full diagnostic field list is the eventual union across clusters, not a day-one requirement on every result."

- [Major] **`path-alias` is promoted into core classifier vocabulary without preserving how narrow and surface-specific it is today.** The classifier result table names `path-alias` as a classifier kind at `DESIGN.md:177`. But the current implementation at `internal/api/session_resolution.go:365-427` describes an API-specific pool-Title fallback that is active/awake/legacy-empty-state only, excludes named-session beads (`apiIsNamedSessionBead`), and uses a most-recently-created tiebreaker for duplicate Titles. No other surface (CLI, mail, extmsg) uses path-alias resolution — `cmd/gc` callers resolve through `session.ResolveSessionID` and never fall through to Title matching. If this becomes a session-owned result kind without those constraints, a temporary API compatibility shim becomes durable session vocabulary. Required change: either keep path-alias fact gathering and acceptance entirely adapter-owned (the classifier receives an optional adapter-supplied candidate for path-alias ranking after live session identity), or spell out the exact current constraints in the slice 1 contract and state that no other surface may opt in without a requirement row.

- [Minor] **The static guard requirement applies uniformly to all slices but should be scoped by slice type.** The guard says "Before a slice is considered implemented" (`DESIGN.md:143`), requiring an AST-based scanner for `SetMetadata`, `SetMetadataBatch`, `Update`, `Create`, `Close`, and `MetadataPatch` usage. This is correct for mutation-owning slices. But slice 1 is read-only target classification — its explicit implementation is classifier result types and operation policy input only (`DESIGN.md:560-571`). If implementers follow the guard literally, slice 1 balloons into an AST scanner plus mutation inventory before any production mutation path changes. Required change: scope the guard by field family and slice type. For slice 1, require a classification-adoption guard (no new resolver fallbacks, no new direct classification re-implementations). Require the mutation-bypass scanner only when a slice introduces or delegates session-owned commands for that metadata family.

- [Minor] **TR-007 acceptance criteria apply to all slices by default, not just event-emitting ones.** TR-007 says "Design event names and payloads so they can later become durable event-log facts" (`DESIGN.md:536-541`). The acceptance criteria require event payloads to carry stable identifiers and enough context for idempotent reaction (`DESIGN.md:540-541`). These criteria are stated as permanent design constraints, not scoped to slices that emit events. If a read-only classifier or command that does not emit events must still satisfy these criteria, the design tax applies to work that never produces events. The recovery section (`DESIGN.md:265-288`) already says critical reactions must converge from durable facts, making rich event payloads redundant for critical paths. Required change: scope TR-007 to event-emitting slices: "When a slice emits or changes session events, event payloads must satisfy TR-007 criteria. Slices that do not emit events are not required to design event payloads."

- [Minor] **The operation policy matrix introduces 6 surface × 11 kind = 66 policy cells for slice 1.** The matrix at `DESIGN.md:198-209` is useful as documentation but risks becoming a required implementation artifact. Slice 1 only needs to classify tokens; adapters decide permissions. If the matrix becomes a required test matrix for the classifier, slice 1 carries 66 policy combinations before any adapter adopts it. Required change: add a statement that the operation policy matrix is adapter documentation, not a classifier test requirement. The classifier's acceptance criteria should be: correct precedence, correct kind assignment, correct candidate details for ambiguous/not-found cases.

**Missing evidence:**

- There is no per-kind population rule for the classifier result fields. Without it, reviewers cannot tell whether `materializable`/`materialization_reason`/`candidates[]` are populated for `direct-session-id` and `live-session-name`, which would be a YAGNI violation at the instance level.
- `closeFailedCreateBead` at `cmd/gc/session_beads.go:1736-1750` writes `pending_create_claim`, `pending_create_started_at` (create/start family, slice 2) and `sleep_intent` (wake/hold/drain family, slice 4) in a single call. The mutation landscape table does not list this function as a cross-family writer. Without it, the static guard's allowlist is incomplete.
- There is no explicit statement that slice 1 can be completed without adding the mutation static guard. The document implies the opposite by putting the guard under the global mutation boundary with "Before a slice is considered implemented."
- TR-007 does not name a concrete current slice that emits a new session event, so reviewers cannot tell whether it is an active acceptance criterion or only a design constraint for future event-emitting slices.

**Required changes:**

1. Remove `forbidden-kind` and `requires-materialization` from classifier `negative_kind`. The classifier returns what the token is; adapters decide policy. Add: "Negative classifier results are `not-found` and `ambiguous` only; policy outcomes like `forbidden-kind` and `requires-materialization` are adapter-level decisions."
2. Add per-kind population rules to the classifier result spec, or state that the result type is a tagged union with kind-specific payloads, not a flat struct with 11+ optional fields.
3. Reconcile the operability contract with the vocabulary scoping rule. Add: "Operability fields are per-command-cluster: each decision/result type carries only the evidence fields its adopted surfaces consume today. The full diagnostic field list is the eventual union across clusters, not a day-one requirement on every result."
4. Keep path-alias as adapter-supplied candidate, or add its current API-only constraints to the classifier contract with a statement that no other surface may opt in without a requirement row.
5. Scope the static guard by field family and slice type. Mutation-owning slices require the bypass scanner; read-only extraction slices require an adoption guard.
6. Scope TR-007 to event-emitting slices only.
7. Add `closeFailedCreateBead` to the mutation landscape table as a cross-family identity/metadata writer, noting which slice owns converting it.

**Questions:**

- Is the operability contract intended as a universal requirement on every result type, or as the eventual union of per-cluster diagnostic fields? The current text reads as universal.
- Is `path-alias` intended to be a first-class session-domain target kind, or only an API compatibility candidate that the classifier can rank after live session identity?
- Which planned slice is expected to emit the first new session event, if any? Without that, TR-007's acceptance criteria float without anchor.
- Should the operation policy matrix be a classifier test requirement, or adapter documentation that each surface maintains independently?
