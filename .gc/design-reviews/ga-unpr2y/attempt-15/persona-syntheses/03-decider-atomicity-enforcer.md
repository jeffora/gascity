# Takeshi Yamamoto

**Persona verdict:** block

**Sources:** Claude, Codex, DeepSeek V4 Flash

**Consensus findings:**
- [Blocker] The atomic command contract is not yet tied to an enforceable fact snapshot and commit fence. All reviewers identify the same core gap: the design lists immutable facts, stale-fact rules, write primitive/fence/token, and recovery fields, but it does not require the commit to revalidate the exact snapshot the decider consumed or to fail on changed facts.
- [Blocker] Store write semantics are under-specified for the current persistence surface. Claude and Codex both note that the design presents "conditional fence" as an option without showing a store-level compare-and-swap or revision primitive. Codex and DeepSeek further flag partial or sequential metadata writes, while Claude frames the viable current model as value-embedded `instance_token`/`generation` fencing plus idempotent repair rather than a bare store-enforced conditional update.
- [Blocker] Runtime side-effect ordering is still a checklist instead of an operation contract. Claude and Codex both require per-operation ordering for wake/start, close, drain, stop/interrupt, and repair flows. The design must preserve existing differences: runtime-start and wake need durable intent before provider work, while close currently has runtime-first failure behavior that must either be preserved or replaced by a documented close-intent protocol.
- [Blocker] Pure decider determinism is not enforceable while decider/projection paths can read ambient time or config. Claude and DeepSeek cite `ProjectLifecycle` zero-`Now` fallbacks to `time.Now().UTC()`, and Claude also notes that target identity classification depends on a config-name set that must be supplied as an explicit fact. Codex asks for a static guard that pure deciders cannot import or call stores, runtime providers, config loading, event buses, CLI/API rendering, mail/work surfaces, or direct wall-clock reads.
- [Major] Current concurrency mechanisms and compatibility paths are not mapped into the new contracts. DeepSeek flags that `WithSessionMutationLock` is process-local, no-token `instance_token` backfills are unfenced, and close behavior is split between richer manager close and narrower bead close paths. Codex similarly requires each command row to state which reducer, lifecycle patch helper, or reconciler path it preserves so advisory state-machine transitions are not mistaken for durable command semantics.
- [Major] Slice 0 artifact and CI-guard evidence is incomplete. Codex requires a pure-decider import/I/O guard, while DeepSeek calls out that the promised boundary artifacts and automated checks must exist in the repository before later mutating slices rely on them.

**Disagreements:**
- Claude returns `approve-with-risks`; Codex and DeepSeek return `block`. My assessment is `block` because the risks Claude names are the load-bearing atomicity rules for every later mutation-moving slice, not optional implementation polish.
- The reviewers differ on the acceptable store strategy. DeepSeek leans toward a global cross-process synchronization or atomic batch capability, Codex requires command rows to bind to concrete store semantics and forbid commands whose stale-safety cannot be proven, and Claude accepts token fencing plus idempotent reconciliation if the design says so honestly. My assessment: either native conditional writes or tokened blind writes with deterministic repair can be acceptable, but the design must choose and prove one per operation.
- The clock-purity remediation needs nuance. DeepSeek asks to remove the `ProjectLifecycle` wall-clock fallback entirely; Claude allows that it may be acceptable for read-only rendering but not for mutation-feeding deciders. My assessment: mutation-command deciders must receive non-zero `now` as an explicit fact; any legacy rendering fallback must be quarantined and excluded from command validation.
- DeepSeek uniquely emphasizes dual close paths and unfenced token backfills as blockers. The other reviews do not independently prove those exact implementation claims, but they align with the shared requirement that every mutating operation map current behavior, concurrency assumptions, stale checks, and repair ownership before decomposition.
- Codex asks for broader `COMMAND_APPLIERS.yaml` rows and validators, while Claude prefers avoiding a false-uniform command bus. These are compatible if the artifact records operation-specific contracts rather than forcing one generic command pipeline.

**Missing evidence:**
- A definition of an immutable operation fact snapshot, including whether facts are read from one coherent bead snapshot, whether `now` and relevant config are mandatory facts, and what token or expected metadata preconditions identify the snapshot at commit time.
- Per-operation command-applier rows for wake/start, close, drain, stop-turn/interrupt, identity retirement, runtime-missing cleanup, and repair/backfill paths.
- A store capability matrix naming which methods are atomic, sequential, partial-failure-prone, conditional, or blind for each backend used by session lifecycle commands.
- Tests that simulate stale snapshots, raced wake/start/close/drain operations, partial metadata writes, duplicate command attempts, provider success followed by commit failure, provider stop followed by close failure, event failure after durable commit, and crash recovery by durable scan.
- A pure-decider file set and automated guard that rejects direct store, runtime, config, event, API/CLI rendering, mail/work, and wall-clock access in classifier and decider code.
- A mapping from current lifecycle reducer/table behavior, lifecycle patch helpers, manager close, bead close, reconciler paths, and token backfills to the future command contracts.
- Evidence that Slice 0 boundary inventory, allowlist, scenario parity, and CI validators are materialized before mutating slices depend on them.

**Required changes:**
- Restate decider purity as an enforceable contract: pure deciders consume explicit immutable facts, including `now` and relevant config snapshots, and perform no store, runtime, config, event, rendering, work-query, or wall-clock reads during evaluation.
- Add a rule to the Atomic Command Contract that the commit fence must revalidate the exact immutable fact snapshot consumed by the decider, or a documented superset, and must reject or repair-converge any changed fact.
- State plainly that the current bead-store surfaces do not provide a bare CAS-style conditional fence unless a specific backend method is added or cited. Remove "conditional fence" as an unqualified menu item, or define it in terms of concrete store support, value-embedded `generation`/`instance_token` checks, and repair ownership.
- Require command-applier rows to bind every mutating operation to concrete store semantics: method used, atomicity/partial-write behavior, pre-commit reread point, fence token or revision checked, stale outcome, post-write verifier, repair owner, and failure-injection tests.
- Add operation-specific durable/runtime/event ordering rows for wake/start, close, drain, stop-turn/interrupt, identity retirement, runtime-missing cleanup, and token backfills before creating implementation beads for those slices.
- Add a global ordering invariant for runtime-creating operations: durable intent carrying the generation/instance token is committed before irreversible provider side effects, post-commit events follow durable commit, and runtime failure after durable commit converges by durable scan rather than event delivery. Name and test any runtime-first exceptions such as close.
- Map state-machine transitions to lifecycle patch helpers and reconciler behavior so advisory transitions cannot replace prepare/commit, runtime confirmation, close cleanup, identity retirement, or repair semantics.
- Materialize the Slice 0 boundary artifacts and CI validators, including the pure-decider import/I/O guard and scenario parity checks, before approving any mutation-moving slice.
