# Liam Okonkwo

**Persona verdict:** approve-with-risks

**Sources:** Claude, Codex, DeepSeek V4 Flash

**Consensus findings:**
- [Major] The session/reconciler/runtime ownership split is now directionally correct. All sources agree that `internal/session` may own lifecycle eligibility, terminal-state projection, wake blockers, and identity classification over immutable facts, while work demand, pool sizing, dispatch scheduling, provider health, progress policy, restart budgets, alert dedupe, idle-sleep policy, and runtime provider observations must stay in the controller/reconciler/runtime-adapter side.
- [Major] The boundary matrix must become a mechanical gate, not just documentation. Before behavior-moving slices proceed, `BOUNDARY_MATRIX.yaml` rows need to exist for provider health, progress and idle thresholds, runtime observations, wake-cause production, partial snapshots, drain/drain-ack, runtime-missing cleanup, adapter provider actions, pool demand, and work-demand compilation. Rows must include owners, allowed session inputs/outputs, forbidden imports/facts/policies, freshness and partial-state behavior, destructive-action safety, recovery owner, proof selector, and negative fixtures.
- [Major] Pure session deciders need enforceable impurity guards. Codex and DeepSeek both require tests or AST/import guards for the pure decider file set. Those guards should reject store queries, runtime provider calls, config traversal, event emission, reconciler/work-demand compilation, alert/budget/progress/provider-health policy, subprocess loops, and direct wall-clock fallback in pure decider code.
- [Major] Runtime-missing cleanup is the highest-risk runtime fact conversion. A missing runtime observation can drive durable state such as `sleep_reason="runtime-missing"` only under an explicit, tested convergence rule. The row should anchor to false-negative tolerance and resume-identity preservation, and require a multi-pass corroboration count or grace/quarantine window so one transient process/probe miss cannot become durable closed, released, or cleanup truth.
- [Major] RuntimeIntent must be operation-specific and provider-neutral. All sources accept bounded intent fields in principle, but each runtime-start, wake, drain, stop/interrupt, and cleanup contract must name exactly which fields are allowed and prove provider health, progress, budget, scheduling, and alert policy remain outside `internal/session`. `provider family` may cross only for routing or identity; session deciders must not branch lifecycle policy on it.
- [Major] Partial, stale, unknown, and provider-error facts need first-class fixtures. Current reconciler behavior suppresses destructive actions when store/session queries are partial and requirements preserve fail-open/fail-closed distinctions for provider health and progress. Boundary rows and command contracts must preserve that behavior so incomplete work/runtime evidence cannot trigger close, drain, rollback, release, cleanup, or restart unless a current requirement explicitly says so.
- [Minor] Wake/hold/wait ownership still needs row-level refinement. The matrix inventory includes wake, hold, and wait, but required-before-moving rows omit hold and wait. Hold projection is session-owned, while hold production such as restart-budget backoff is reconciler-owned; that distinction should be explicit before slices touching wake eligibility and reconciler facts move code.
- [Minor] Shared helper chains across wake eligibility and health gates need a recorded disposition. If Slice 0 finds helper code shared across session-owned wake eligibility and reconciler-owned health/provider gates, the matrix should record whether each helper is split, duplicated, or moved to a neutral package before slices 3 and 6 both modify `cmd/gc/session_reconciler.go`.

**Disagreements:**
- Verdict severity differs. Claude and Codex both concluded `approve-with-risks` because the revised design text names the right ownership boundaries and rejection rules. DeepSeek concluded `block`, mainly because current code still contains `time.Now().UTC()` fallbacks in `ProjectLifecycle`/`creatingStateIsStale`, Slice 0 artifacts are not yet materialized, and runtime-missing/session-key details are not physically enforced. My assessment: these are real required gates before Slice 0 closure or behavior-moving slices, but not enough to mark this persona synthesis as a design-text block now that the design explicitly requires mandatory `now`, operation contracts, boundary rows, and destructive-action rejection.
- Slice 0 treatment of `SESSION-RECON-*` evidence is disputed. Claude sees the requirement to repair or owner-retire `SESSION-RECON-002`, `SESSION-RECON-003`, `SESSION-RECON-006`, and `SESSION-RECON-007` as a necessary stale-evidence guard before later slices cite those rows. DeepSeek views it as coupling session Slice 0 to reconciler-owned behavior too early. My assessment: keep the stale-evidence reconciliation, but require owner disposition rather than forcing session-owned implementation; reconciler-owned rows can be retired, amended, or delegated to later reconciler slices.
- Runtime session key in intent is disputed. Claude and Codex treat `runtime session key` as an acceptable provider-neutral field when an operation contract proves it is needed. DeepSeek says including `session_key` in prepare-time `RuntimeStartIntent` can smuggle an observation token into an intent. My assessment: require each operation to classify the field as deterministic pre-derived identity or post-start observation; do not allow a post-start observed key in a prepare-time decider input.
- Decider reads are not defined sharply enough. Claude allows session bead reads via session resolvers if runtime/progress/demand facts are immutable snapshots. Codex and DeepSeek push toward zero store queries in pure deciders. My assessment: define the guarded pure set explicitly. Classification/resolution adapters may read indexed session beads, but pure lifecycle command deciders should consume immutable fact snapshots and avoid direct store/runtime/config/event access.

**Missing evidence:**
- Physical `BOUNDARY_MATRIX.yaml` rows and schemas for the required runtime/reconciler/session boundaries listed in the design.
- The exact pure-decider package/file set and CI guard that enforces no store/runtime/config/event/work-query/provider-policy imports or direct wall-clock fallback inside that set.
- Operation-specific RuntimeIntent schemas for runtime start, wake, drain, stop/interrupt, and runtime-missing cleanup.
- A stated owner and field lifecycle for runtime observation freshness timestamps and runtime session keys.
- Explicit anti-flap evidence for runtime-missing cleanup: corroboration count, grace window, quarantine rule, or equivalent tested convergence guard.
- Slice 6 inventory of narrow lifecycle eligibility facts that may be extracted from reconciler code, with negative examples for work counts, pool size, provider health, progress, and demand facts that must remain outside session deciders.
- Counting-store, benchmark, or fixture evidence for large-city reconciler fact materialization budgets.
- Repaired, amended, or owner-retired evidence rows for stale `SESSION-RECON-*` requirement proofs.

**Required changes:**
- Materialize `BOUNDARY_MATRIX.yaml` before behavior-moving slices and require rows for provider health, progress/idle thresholds, runtime observations, wake-cause production, partial snapshots, drain/drain-ack, runtime-missing cleanup, adapter provider actions, pool demand, and work-demand compilation.
- Add a mechanical guard for the declared pure session decider file set. It must reject store/runtime/config/event/work-query imports, provider-health/progress/budget/alert policy fields, direct subprocess/provider calls, and direct `time.Now` fallback in pure deciders.
- In the runtime-missing cleanup row and Atomic Command Contract, name the false-negative tolerance and resume-identity preservation anchors, then require a tested multi-pass or time-bounded anti-flap rule before any missing observation becomes durable cleanup/release/closed truth.
- Make RuntimeIntent operation-specific. Each operation contract must list allowed fields, define whether runtime session key is pre-derived identity or post-start observation, and state that provider family is routing/identity-only, not lifecycle policy input.
- State the decider read boundary explicitly: pure lifecycle command deciders consume immutable snapshots; any session bead reads must happen through declared resolvers/adapters and must not import reconciler helpers or demand/schedule/health/progress queries.
- Promote `hold` and `wait` to required boundary rows before behavior-moving slices, and add a hold-production-owner distinction so reconciler backoff/budget holds are not confused with session-owned blocker projection.
- Add a required helper-chain disposition in Slice 0 for code shared across session-owned wake eligibility and reconciler-owned health/provider gates: split, duplicate, or move to a neutral package.
- Add partial-fact fixtures proving destructive actions are rejected or deferred for unknown, stale, partial, or provider-error work/session/runtime facts, except where a current requirement explicitly preserves fail-open behavior.
