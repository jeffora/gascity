# Ingrid Holm

**Persona verdict:** approve-with-risks

**Sources:** Claude, Codex, DeepSeek V4 Flash

**Consensus findings:**
- [Major] The design is directionally sound for non-mutating Slice 0 only. It now requires structured diagnostics, trace mappings, renderer proof, event relationships, redaction metadata, hot-path budgets, subprocess budgets, and large-city evidence before behavior moves. None of the sources approve behavior-moving slices until those artifacts exist and are validated.
- [Major] `DIAGNOSTICS_MANIFEST.yaml` must be an executable contract, not a checked-in placeholder. Rows need stable operation/check IDs, reason/outcome codes, renderer surfaces, trace mappings, doctor/session-inspect/API/CLI proof, redaction keys, event relationships, query budgets, subprocess budgets, hot-loop constraints, proof commands, and negative fixtures proving machine-readable data is not hidden only in message strings.
- [Major] Lifecycle decider diagnostics must preserve the structured operator "why." The current projection already has typed wake causes and blockers; a generic reason string is not enough for wake, close, drain, identity-retirement, or repair decisions. Diagnostics must carry canonical typed wake cause/blocker/result data with parity tests against existing display and trace behavior.
- [Major] Every diagnostic row that affects session operations must map to `gc trace` or explicitly state why it is not trace-rendered. Coverage must include accepted, rejected, blocked, deferred, stale-fact, partial-fact, repair-needed, event-emission-failed, event-skipped, duplicate-event, scan-recovered, and scan-failed outcomes, because the incident workflow depends on `gc trace`, `.gc/events.jsonl`, doctor/inspect output, and controller logs.
- [Major] The path-alias lookup is a real hot-path risk. `resolveLiveSessionByPathAlias` currently performs an all-session scan on an API query path. The design correctly names the hazard, but before delegation it must choose index, removal, or a measured bounded strategy that is safe on the production backend. A process-forking all-session scan cannot be waved through as "budgeted" without large-city proof and a hard maximum.
- [Major] Budgets must count production `bdstore` calls as subprocess cost. Ordinary `Get`, `SetMetadata`, and list operations can fork `bd`, so per-session store calls in reconciler loops are subprocess loops in production. Budget rows must record store-call count, subprocess count, scanned rows, runtime probes, partial-snapshot behavior, fixture size, threshold, owner, allowed delta, and proof command.
- [Major] Reconciler fact compilation needs a concrete large-city strategy. The design may satisfy this with an indexed/bulk/snapshot reader, a cache/incremental adapter, or counting-store proof that the current approach stays under threshold; it cannot leave repeated per-session subprocess fan-out as an implementation detail for behavior-moving slices.
- [Major] `repair-needed` is the right read/write separation for empty-type session beads, but the execution lifecycle is undefined. The design must state who schedules or runs the audited repair, whether reads return a typed degraded result or retry after repair, how before/after evidence is surfaced, and what concurrency fence protects the bead while repair is pending.
- [Minor] Budget rows currently name future thresholds but lack current baselines. Slice 0 should capture measured current scan row counts, store-call counts per reconcile pass, subprocess counts, event fan-out size, durable scan size, and large-city fixture results so regressions can be detected from day one.
- [Minor] Subscriber fan-out and durable recovery scans need backpressure and idempotency tests. Event emission must not synchronously amplify by session count, and missed-event convergence must not rely on unbounded scans.

**Disagreements:**
- Verdict severity differs. Claude and Codex approve with risks for Slice 0 only. DeepSeek blocks because path-alias indexing/removal, repair-needed execution, and reconciler fact caching are still deferred. My assessment: this lane can approve the design for non-mutating Slice 0, but behavior-moving or operability-sensitive slices are blocked until the manifest, validators, numeric budgets, and concrete performance decisions exist.
- Path-alias resolution has a sharper disagreement. Claude and Codex accept the design's "index, remove, or keep with explicit scan budget" formulation if Slice 0 proves it. DeepSeek rejects any budget-only all-session scan on the API hot path. My assessment: on the production `bdstore` backend, keeping the current full scan is acceptable only if the row proves a hard maximum, subprocess budget, newest-created tiebreaker behavior, and large-city safety; absent that proof, the design should require index or removal before delegation.
- Reconciler hot-loop mitigation differs in specificity. Codex and Claude require budgets, counting-store evidence, and no subprocess loops. DeepSeek wants a mandated caching or bulk-read mechanism. My assessment: the design should not mandate one mechanism prematurely, but Slice 0 must either choose a concrete mechanism or prove the existing path meets strict store-call/subprocess budgets.
- `SESSION-RECON-*` Slice 0 evidence is disputed again in this lane. DeepSeek says reconciler-owned pool/health/progress requirements should not block session Slice 0. Claude and Codex accept stale-evidence reconciliation as part of safe preflight. My assessment: keep the requirement to repair, amend, or owner-retire stale evidence, but make ownership explicit so session Slice 0 does not inherit reconciler implementation responsibility.
- `RepairEmptyType` quarantine is both a strength and a risk. All sources support removing silent read-side writes; DeepSeek emphasizes that availability can regress if the repair lifecycle is not defined. The synthesis treats lifecycle definition as a required close gate before the classifier is delegated.

**Missing evidence:**
- A materialized, validated `DIAGNOSTICS_MANIFEST.yaml` and `TestSessionDiagnosticsManifest` or equivalent validator.
- Centralized diagnostic operation/reason/outcome constants and proof that every manifest row maps to trace, doctor/session-inspect, API/CLI, dashboard/SSE, or an explicit non-rendered decision.
- Typed lifecycle diagnostic output for `WakeCause`, blocker, drain/close reason, repair-needed state, and parity tests against existing display/trace rendering.
- Numeric budget rows for API target resolution, `resolveLiveSessionByPathAlias`, reconciler fact compilation, event recovery scans, subscriber fan-out, and large-city baselines.
- Current measured baselines for scan row counts, store-call counts, subprocess counts, event fan-out size, durable recovery scan size, and reconciler pass cost.
- A path-alias decision with proof: index, removal, or hard bounded large-city scan budget that counts `bdstore` subprocess cost.
- A concrete reconciler fact compilation strategy or counting proof that prevents per-session subprocess loops.
- The execution, trace, repair, and concurrency model for `repair-needed` empty-type session handling.
- Backpressure/idempotency tests for subscriber fan-out, missed events, duplicate events, durable scan recovery, and event-emission failure.
- Route-specific large-city proof for Huma and legacy API target resolution, plus controller-loop proof for reconciler fact compilation with representative session/work counts and partial snapshots.

**Required changes:**
- Add `DIAGNOSTICS_MANIFEST.yaml` with stable operation IDs, reason/outcome codes, trace site mappings, renderer rows, event relationships, redaction keys, query/subprocess budgets, hot-loop constraints, and negative fixtures; add a validator that fails when any named row, renderer, proof selector, or centralized constant is missing.
- Require lifecycle deciders to emit canonical typed wake cause, blocker, drain/close reason, retryability, source facts, and stale/missing/partial indicators. Add parity tests against existing lifecycle display and `gc trace` sites such as wake/sleep, drain advance, and start execution.
- Make trace coverage include rejected, blocked, deferred, stale-fact, partial-fact, repair-needed, event failed/skipped/duplicated, scan-recovered, and scan-failed cases for wake, drain, close, runtime start, provider health, repair, and work release.
- Resolve `resolveLiveSessionByPathAlias` before delegation: index it, remove it from the first-adopter hot path, or prove a hard maximum row/subprocess budget and large-city fixture behavior. Do not allow an unbounded process-forking all-session scan on an API query path.
- Record current measured baselines and numeric thresholds for every budget row: max store calls, subprocesses, scanned rows, runtime probes, event fan-out, durable scan size, fixture size, benchmark or counting-store command, owner, and allowed delta.
- Explicitly state that production `bdstore` store operations count as subprocesses in hot-path budgets, so per-session `Get`/`SetMetadata` loops are treated as subprocess loops.
- Choose or prove the reconciler fact compilation strategy: indexed/bulk reader, snapshot/cache/incremental adapter, or counting-store proof that the current path stays inside strict large-city thresholds.
- Define the `repair-needed` lifecycle for empty-type sessions, including scheduler/owner, non-blocking behavior, typed API/CLI result or retry semantics, trace/doctor before-and-after evidence, persistence-error propagation, and concurrency fencing.
- Add subscriber fan-out and durable recovery scan tests proving bounded work, idempotency on duplicate events, convergence after missed events, and no synchronous session-count amplification.
- Require every behavior-moving slice to reference the exact diagnostics, trace, repair, event, and budget row IDs it consumes; citing the design section generally is not enough.
