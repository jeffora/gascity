# Ravi Krishnamurthy

**Persona verdict:** block

**Sources:** Claude, Codex, DeepSeek V4 Flash

**Consensus findings:**
- [Blocker] The migration/coexistence section is materially improved, but it is not yet safe to decompose behavior-moving slices. All sources agree that Slice 0 is the only safe non-mutating step; mutation-moving slices still lack source-complete inventories, exact exception rows, field/key-level ownership transfer, rollback proof, and cross-process coexistence mechanics.
- [Blocker] Slice 1 collides with the worker-boundary migration through a shared resolver entry point. `resolveSessionTargetIDWithContext` currently owns read-only lookup, allow-closed lookup, and `materialize:true` named-session creation. The terminal materialize path reaches `materializeNamedSessionWithContext` and then `CreateAliasedNamedWithTransportAndMetadata`, which is a live worker-boundary exception. Extracting read-only precedence without sequencing the mutating mode creates either a hidden worker-boundary collision or two drifting precedence implementations.
- [Blocker] The worker/API close inventory is source-inaccurate. The design says production API/CLI close remains aligned with `worker.Handle.CloseDetailed`, including Huma routes, but the live Huma command handler still constructs a session manager and calls `mgr.CloseDetailed(id)`. That path must be routed through `worker.Handle` or recorded as an exact expiring exception with route, handler, fields, response compatibility, event/cleanup behavior, owner, expiry, and retirement proof.
- [Blocker] The migration trigger is too narrow. The design names `internal/api/session_resolution.go`, `cmd/gc/session_reconciler.go`, and `cmd/gc/session_beads.go`, but the source-visible mutation/boundary surface also includes Huma command handlers, legacy API handlers, session create rollback, explicit wake/nudge, lifecycle parallel start, sleep, wait, circuit breaker, soft reload, and reconciler helpers. Migration rows must be inventory-driven from `SESSION_BOUNDARY_SYMBOLS.yaml`, `API_CLI_ROUTE_INVENTORY.yaml`, and `WORKER_BOUNDARY_EXCEPTIONS.yaml`, not triggered only by three filenames.
- [Major] Close is already split across multiple production writers, and Slice 4 would add another close writer unless the design broadens the scope. Reconciler drained close, failed-create/sweep close, and parallel-start failed-create close all write terminal state through raw `ClosePatch`/store calls. Close-alignment must cover all production close writers, not only API and CLI callers.
- [Major] Field-family ownership is too coarse for coexistence. Writers can collide on exact metadata keys such as `wake_request`, `pending_create_*`, `session_key`, close status, wake/hold/drain keys, and dynamic key sources while appearing to be in the same field family. Migration rows need exact key or dynamic-key-source ownership before/during/after each slice, with old writer, new command owner, fence, validation parity, rollback behavior, and retirement condition.
- [Major] Cross-process fence requirements need a standard, enforceable shape. The design correctly forbids unfenced legacy/new writers, but it still lets slices pick their own fence. Because CLI/API and controller/reconciler processes do not share memory, in-process mutexes are insufficient. Rows should restrict coexistence to approved cross-process mechanisms such as store-level compare-and-swap/token preconditions or another explicitly approved durable fence, with raced-writer tests.
- [Major] Rollback proof is under-specified. Rows say whether rollback preserves, clears, or repairs fields, but each behavior-moving slice must prove it can ship alone and revert alone without requiring a successor slice. That proof must include old-reader/new-field tolerance, new-reader/old-field tolerance, no dangling provisional type, no hidden flag-day metadata, and a repair/backfill path when rollback leaves durable metadata behind.
- [Major] Validation split-brain is not closed by race tests alone. A repair-converged blind write can converge under timing races while still accepting a value the new session command rejects. Validation-critical fields need CAS/token fencing or proof that legacy raw writers call the same validation as the new command.
- [Major] `RepairEmptyType` quarantine fixes the read-side write hole but leaves an availability gap. Returning `repair-needed` keeps the classifier read-only, but the design must define when and how the audited repair command runs so read paths do not silently write, block indefinitely, or regress healthy session lookups to 404/500 behavior.

**Disagreements:**
- DeepSeek asks for a new global state-version or schema-epoch tag on session beads as the rollback safety mechanism. Claude and Codex ask for per-slice rollback proof, reader tolerance, repair/backfill, and key-level ownership, but do not require an epoch specifically. My assessment: the design must require an epoch or an equivalent explicit version/tolerance proof per key family. If no equivalent proof exists, the epoch requirement is the cleanest way to avoid mixed-version rollback ambiguity.
- DeepSeek wants store-level mutation exceptions for wake/drain eliminated entirely. Claude and Codex allow exceptions only if they are exact, expiring, owner-approved, and parity-tested. My assessment: root policy currently allows documented exceptions, so the blocker is not the mere existence of an exception path; the blocker is any broad or mutation-bearing exception without exact route/symbol ownership, expiry, and retirement proof.
- Claude emphasizes the shared resolver sequencing and single-source precedence fork as the first executable Slice 1 blocker. Codex emphasizes missing exact Huma/API exceptions and inventory coverage. DeepSeek emphasizes standardized fences and rollback epochs. These are complementary rather than contradictory: all point to the same unsafe coexistence window.
- DeepSeek states only Slice 1 is independently shippable and revertible. Claude blocks Slice 1 as currently written because the read-only classifier and materializing create path share the same function. My assessment: Slice 1 can become shippable only after the shared-resolver sequencing rule and anti-drift parity are added.

**Missing evidence:**
- Exact `WORKER_BOUNDARY_EXCEPTIONS.yaml` rows for API/Huma manager-use mutation paths, especially Huma close and API named-session materialization.
- Source-complete `API_CLI_ROUTE_INVENTORY.yaml` and `SESSION_BOUNDARY_SYMBOLS.yaml` rows covering Huma routes, legacy mux routes, Cobra commands, `apiClient()` fallbacks, generic bead routes, dashboard/generated-client obligations, wake/nudge/sleep/wait/drain/close paths, create rollback, lifecycle parallel start, and reconciler repair writers.
- Worker-boundary end-state and sequencing for `internal/api/session_resolution.go`: whether direct materialization is retired before, during, or after target-classifier extraction.
- Anti-drift proof for resolution precedence while read-only classifier and materializing create coexist: shared precedence implementation or parity tests on match vectors/result kinds.
- Migration rows for Huma close, API materializing named-session create, explicit wake, close/delete cleanup, reconciler wake/drain/repair writers, and session create rollback.
- Exact key-level before/during/after ownership and dynamic-key-source mapping for each migrated metadata key.
- Approved cross-process fence choices and tests proving old-writer/new-command coexistence is physically safe across separate OS processes.
- Validation-parity tests for any legacy raw writer that coexists with a new session command on a validation-critical field.
- Per-slice ship-alone and revert-alone proof, including old/new reader tolerance and repair/backfill handling for durable metadata left behind.
- A defined execution lifecycle for `repair-needed` empty-type session repairs.

**Required changes:**
- Add a shared-resolver sequencing subsection for `resolveSessionTargetIDWithContext`. It must state the ordering relative to the worker-boundary migration and require either a single source of resolution precedence used by both read-only and materializing modes, or parity tests proving the inline materialize path and new classifier produce identical match vectors/result kinds until materialization migrates.
- Replace the three-file migration trigger with an inventory-driven trigger. Any slice touching a path or symbol in `SESSION_BOUNDARY_SYMBOLS.yaml`, `API_CLI_ROUTE_INVENTORY.yaml`, or `WORKER_BOUNDARY_EXCEPTIONS.yaml` that has both worker-boundary and session-mutation relevance must add a migration row.
- Add exact expiring exception rows for current API/Huma manager-use mutation paths, especially Huma close and API materializing named-session create. Each row must name route, handler function, manager method, fields/top-level status touched, response compatibility proof, event/cleanup behavior, owner, expiry, and retirement condition.
- Expand close-alignment from "production API and CLI callers" to "all production close writers." Enumerate reconciler drained-close, failed-create/sweep close, and parallel-start failed-create close paths as inputs to the Slice 4 close command contract.
- Require migration rows at exact key or dynamic-key-source granularity. A row may group keys for readability, but validators must fail if a touched key lacks before/during/after owner, old writer, new command owner, fence, validation parity, rollback behavior, and retirement condition.
- Standardize allowed cross-process fence mechanisms for writer coexistence. Disallow in-process-only locks as coexistence proof; require durable CAS/token/precondition or an explicitly approved equivalent plus raced-writer tests.
- Constrain repair-converged blind writes to fields where legacy and new writers share validation or where the legacy writer is demoted to use the new validation. Otherwise require CAS/token fencing.
- Strengthen rollback gates so every behavior-moving slice proves it can ship alone and revert alone. Add old-reader/new-field tolerance, new-reader/old-field tolerance, no required successor slice, no dangling provisional types, and repair/backfill proof for retained durable metadata.
- Add a per-field window-closes rule: once a field moves behind a session-owned command, production raw writers for that field must be retired or listed as exact expiring exceptions with the same validation and fence proof.
- Define the `repair-needed` lifecycle for empty-type sessions: who schedules or runs the audited repair command, whether reads return a typed degraded result or retry after repair, and how the path avoids read-side writes and availability regression.
