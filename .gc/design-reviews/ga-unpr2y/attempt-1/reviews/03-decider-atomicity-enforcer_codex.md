# Takeshi Yamamoto - Codex

**Verdict:** approve-with-risks

**Top strengths:**
- The revision no longer pretends that the atomic command model is already approved. It explicitly keeps decomposition blocked except for a self-validating, non-mutating Slice 0, and says future command, event, runtime, and diagnostic contracts become normative only in the slice that first delegates a caller to them.
- The Atomic Command Contract now names the load-bearing fields I expect before a mutating slice moves: immutable facts with mandatory `now`, config facts, token/revision/value preconditions, exact pre-commit revalidation, store capability rows, post-write verification, provider/event ordering, partial-state handling, and failure-injection tests.
- The design now acknowledges the current store reality instead of assuming a hidden CAS. The general bead store interface has no revision or compare-and-swap primitive, and external stores can apply metadata batches sequentially, so the per-operation requirement to choose a real write primitive, token fence, or repair-converged blind write is the right direction.

**Critical risks:**
- [Major] Pure decider enforcement is still mostly contractual prose. `internal/session/DESIGN.md` says deciders consume immutable facts, and `BOUNDARY_MATRIX.yaml` rows must list forbidden imports/fields, but Slice 0 does not explicitly require a pure-decider file set or static guard that rejects store, runtime, config loading, event bus, work/mail/extmsg, API/CLI rendering, or wall-clock access in future decider files. The current code still has valid non-decider wall-clock fallbacks such as `ProjectLifecycle` zero-`Now` handling; the future pure set needs a guard that quarantines those paths instead of letting them drift into command deciders.
- [Major] The "repair-converged blind write" option is necessary for compatibility, but it is also the easiest escape hatch. The design requires a row to name the rule and crash/race tests, yet it does not define the minimum criteria for when a blind write is allowed versus forbidden. For commands that create runtime identity, clear pending-create metadata, close a bead, release identity, or complete drain, the row must prove why token/revision/value fencing is impossible or unnecessary before accepting blind repair convergence.
- [Major] Runtime side-effect ordering is required per operation, but the design still lacks a reusable ordering template for runtime-start, close, drain, and stop/interrupt. This matters because the safe order differs by operation: runtime-start normally needs durable intent before provider work, close currently has provider-stop failure behavior that can leave the bead open, and event emission must remain post-commit and non-authoritative. Per-operation rows can solve this, but only if the schema forces exact commit, event, provider-action, failure, and scan-recovery ordering.
- [Minor] Slice 0 creates `COMMAND_APPLIERS.yaml` as a schema plus universal inventory while operation-specific command rows are deferred. That is acceptable only because Slice 0 is non-mutating. The workflow close gate must make it impossible for a later bead to cite the existence of `COMMAND_APPLIERS.yaml` as approval for wake, close, drain, runtime-start, or repair behavior without a slice-specific row.

**Missing evidence:**
- Materialized Slice 0 artifacts and validators: `SLICE0_CONTRACT.yaml`, `SESSION_BOUNDARY_SYMBOLS.yaml`, `COMMAND_APPLIERS.yaml`, `BOUNDARY_MATRIX.yaml`, and the proof command are still design promises, not present evidence in this review.
- A pure-decider guard and fixture set proving future decider packages cannot perform store/runtime/config/event/work/query/rendering I/O or read ambient time.
- Operation-specific command rows for wake/start, close, drain, stop/interrupt, runtime-start prepare/commit/rollback, runtime-missing cleanup, identity retirement, and repair/backfill.
- Store capability proof for every persistence surface a session command may use, including sequential external metadata batches, logical transactions without observer-visible atomicity guarantees, close/reopen behavior, and any store-specific conditional-write support.
- Failure-injection tests for stale snapshots, raced writers, partial metadata batches, duplicate commands, provider success followed by commit failure, commit success followed by event failure, skipped events, and durable scan convergence.
- Evidence that existing clock fallbacks in lifecycle projection are outside the future pure decider file set or are replaced with explicit `now` facts before any mutation-feeding decider uses them.

**Required changes:**
- Add an explicit pure-decider enforcement row to Slice 0 or to the first behavior-moving preflight schema. It should require a named file/package set, an import/call guard, positive and negative fixtures, and proof that `time.Now`, stores, runtime providers, config loaders, event buses, work/mail/extmsg packages, API/CLI renderers, and subprocess helpers are unavailable to decider code.
- Tighten the command-applier schema so a mutating row cannot pass unless it chooses exactly one commit fence strategy: store-native conditional write, value-embedded token/revision/value revalidation immediately before commit, or repair-converged blind write with stated eligibility criteria and crash/race tests. Do not allow a row to list these as alternatives without choosing one.
- Define minimum eligibility for "repair-converged blind write": which facts may be stale, which durable fields may be overwritten, what supersession key prevents older attempts from winning, what scanner owns convergence, and which partial states are safe. For destructive or runtime-identity operations, require the row to justify why a token or revision fence is not required.
- Add operation-order templates for runtime-start, close, drain, and stop/interrupt rows. Each template should force the row to state durable mutation order, provider action order, post-commit event order, provider-success/commit-failure behavior, commit-success/event-failure behavior, skipped-event behavior, and durable scan recovery.
- Explicitly state that Slice 0 approval does not approve any mutation-moving command row. Later beads must carry metadata for the exact operation preflight row they consume, and the finalizer must fail if the row is missing, schema-invalid, or not exercised by stale/race/partial-write tests.

**Questions:**
- Will the pure decider guard live in Slice 0 as a universal validator, or in each mutation-moving slice as an operation-specific preflight? Either is workable, but the close gate must know where to find it.
- Which store surfaces are in scope for session lifecycle commands during this migration: HQStore, BdStore, exec store, FileStore/MemStore, or all of them?
- For runtime-start, is `instance_token` intended to be the primary attempt fence for normal paths, with no-token writes treated only as repair/adoption compatibility paths?
