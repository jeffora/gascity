# Kwame Asante - DeepSeek V4 Flash

**Verdict:** approve-with-risks

Lane: minimal vocabulary, facade creep, event-log deferral, backlog scope control. This is an independent review of the attempt-4 revision of `internal/session/DESIGN.md` against `REQUIREMENTS.md`, the source code, and prior-review findings. I focused on whether vocabulary introduced matches what the first implementing slice actually needs, whether TR-007's future-compat language pulls more structure into today's contracts than the in-process event model requires, and whether anything is on a path to becoming a broad facade.

**Top strengths:**

- The vocabulary checkpoint table (`DESIGN.md:102-108`) is the strongest YAGNI control in the document. It binds every proposed shared type to a first-allowed-caller and lists required fields at introduction. Combined with the rule-of-two promotion gate ("A shared type is allowed only after two implemented deciders consume the same exact field set", `DESIGN.md:76-78`) and the flat-envelope prohibition ("Flat optional envelopes are not acceptable for new shared types", `DESIGN.md:108-109`), this is concrete, auditable YAGNI enforcement — not a slogan.
- The mutation landscape table (`DESIGN.md:157-173`) is complete and actionable. It enumerates every production writer, field family, target path, owner slice, and exception status with inventory IDs. The prior iteration flagged `closeFailedCreateBead` as a missing cross-family writer; I verified that this function writes `pending_create_claim`, `pending_create_started_at` (create/start family, slice 3), and `sleep_intent` (wake/hold/drain family, slice 5) in a single call (`cmd/gc/session_beads.go:1736-1750`). The table lists W-008 for `cmd/gc/session_beads.go` as "close metadata, bead close, identity retirement, assigned-work release" (lifecycle/identity, slice 4) but does not flag this cross-family write. This is a gap, but it is a gap in inventory completeness, not a design-structural problem.
- The event recovery ordering table (`DESIGN.md:365-380`) is now explicit that critical reactions converge from durable facts and controller scans, and that `SessionDrainAckedWithAssignedWork` is diagnostic/acceleration rather than recovery authority. This is the correct recovery model for in-process delivery and directly addresses the lane's prior concern about events becoming commands.
- Slices 5 and 6 are still blocked on missing proof files. The traceability matrix says this honestly. This is the right scope gate: no implementation until proof exists.

**Critical risks:**

- **[Major] The per-operation policy input is enumerated but its type and construction are undefined, making the minimum slice-1 vocabulary unverifiable.** The vocabulary section says slice 1 introduces "a per-operation policy input" (`DESIGN.md:91`), and the target-classification contract lists nine policy input flags: `surface`, `allow_closed`, `materialize_named`, `allow_path_alias`, `allow_historical_alias`, `read_only`, `allow_template_factory`, `allow_ordinary_config_target`, and `reject_config_missing_named` (`DESIGN.md:270-273`). But the design does not state whether: (a) the classifier receives all nine flags at once as a single struct; (b) the classifier receives only the subset needed by the current caller; or (c) each adapter constructs its own policy and the classifier applies it. This is not a naming concern — it directly determines slice-1 vocabulary size. Current code confirms two distinct entry points: `ResolveSessionID` takes no flags, while `ResolveSessionIDAllowClosed` takes a single `allowClosed` boolean (`internal/session/resolve.go:33-37`). API resolution adds `materialize` (`internal/api/session_resolution.go:57-58`). If the classifier consumes all nine flags on day one, slice-1 vocabulary includes a policy struct spanning six surfaces before a single surface delegates. If adapters construct their own policy, the classifier is smaller but the design must state that. **Required change**: define the policy input type, list which fields slice 1 requires, and state whether the classifier consumes it or adapters apply it to the raw classification result. At minimum, specify whether `allow_closed` and `materialize_named` are classifier inputs or adapter post-filters, and align the result's `forbidden-kind`, `requires-materialization`, and `closed-not-allowed` with that decision.

- **[Major] `forbidden-kind`, `requires-materialization`, and `closed-not-allowed` are operation-policy outcomes in classifier clothing, violating TR-006's separation of "what a token is" from "what an operation may do with it."** The design states the classifier "separates what an input token is from what an operation is allowed to do with it" (`DESIGN.md:163`), yet the result includes `forbidden-kind`, `requires-materialization`, and `closed-not-allowed` as negative kinds (`DESIGN.md:280-281`). Current code confirms the policy split: `internal/api/session_resolution.go:57-58` shows `materialize` and `allowClosed` as separate options, and `cmd/gc/cmd_mail.go` takes a completely different resolution path that never materializes. Whether a `configured-named-reserved` target is `requires-materialization` or `forbidden` depends on which surface is asking. The classifier can return what the token names (a configured identity with no canonical bead) and the operation's policy context (allow-materialize = false) can determine it is forbidden — but this requires the classifier to receive the policy input, which loops back to the undefined policy type. If the classifier does not receive policy input, these negative kinds cannot be produced. Prior iteration-4 reviewers (Claude and Codex) both flagged this; the design has not resolved the contradiction. **Required change**: either remove `forbidden-kind`, `requires-materialization`, and `closed-not-allowed` from classifier negative kinds and let adapters apply policy, or make the policy input an explicit classifier parameter and list the required fields in the slice-1 vocabulary. One or the other — not both.

- **[Major] The operability contract's required diagnostic fields are a universal result envelope that contradicts the per-slice vocabulary rule.** `DESIGN.md:547-554` says "Every decision, command, and conflict result must carry" sixteen categories of evidence: operation/site/reason/outcome, three identity fields, lifecycle projection, blocker, wake cause, target kind, generation, instance token, runtime session key, five fact-freshness fields, conflict details, and event emission result. For target classification (slice 1), generation, instance token, runtime session key, and event emission result are meaningless — the operation is read-only and emits no events. The design explicitly scopes vocabulary per slice ("Each slice introduces only the fact family and result vocabulary needed for that operation"), but the operability contract applies uniformly. If implementers read this as requiring every field for every result, slice 1 sprouts a diagnostics facade. **Required change**: state that the operability field list is the eventual union across command clusters, and each result type carries only the fields its adopted surfaces consume today. Add a per-operation column or a clear scoping note: "Classification results carry identity, target kind, candidates, and conflict details; generation, instance token, and event emission are command-cluster fields."

**Minor risks:**

- **[Minor] `SessionFactEvent` is listed in the vocabulary checkpoint but no slice owns its introduction.** The checkpoints table (`DESIGN.md:108`) gives `SessionFactEvent` a first-allowed-caller of "First post-commit command subscriber" — but no slice in the backlog introduces a new session event. Slice 1 is read-only; slices 2-4 are the earliest event-emitting slices. TR-007's acceptance criteria (`DESIGN.md:725-733`) apply to any event-emitting slice, but nothing in the document says which slice emits the first new event. Prior reviewers flagged this across iterations 1-4; it remains unaddressed. **Required change**: assign `SessionFactEvent` introduction to a specific slice (earliest 2 or 3) and state that slice 1 introduces no event vocabulary.

- **[Minor] `historical-alias` as a universal classifier kind conflicts with `REQUIREMENTS.md` SESSION-ID-003.** `REQUIREMENTS.md:102-103` states that normal resolution "does not fall through to template names, agent names, template basenames, or historical aliases." The classifier precedence table includes `historical-alias` at position 11 (`DESIGN.md:248`), and the compatibility chain for mail query says it "includes historical alias recipients where `apiSessionMailboxAddresses` currently does" (`DESIGN.md:263`). This is fine as an adapter-supplied candidate, but listing it as a universal classifier kind invites implementers to include it in the default resolution path, violating SESSION-ID-003. The design should state that `historical-alias` is only produced when a surface explicitly opts in via `allow_historical_alias=true`. **Required change**: add a note to the `historical-alias` row that it is not part of the default resolution path (SESSION-ID-003) and is only produced when the surface policy explicitly enables it.

- **[Minor] `path-alias` is API-specific in current code but promoted to a universal classifier kind.** `resolveLiveSessionByPathAlias` (`internal/api/session_resolution.go:365-402`) is an API-only function with state filters, named-session exclusion, and most-recent tiebreaker that no other surface uses. The classifier precedence table lists `path-alias` at position 8. If the classifier produces `path-alias` as a session-domain kind, other surfaces (CLI, mail, extmsg, assignee normalization) may be expected to handle it, but the design correctly limits it to "API path alias/title match" in the kind table (`DESIGN.md:245`). This is acceptable if the design states that `path-alias` candidates are adapter-supplied rather than classifier-discovered — the classifier should not scan all session titles. **Required change**: state that `path-alias` candidates are supplied by the API adapter's title-matching step, not discovered by the classifier from a general store scan.

- **[Minor] `closeFailedCreateBead` writes cross-family metadata but is not flagged in the mutation landscape table.** `cmd/gc/session_beads.go:1736-1750` writes `pending_create_claim`, `pending_create_started_at` (create/start family, slice 3) and `sleep_intent` (wake/hold/drain family, slice 5) in addition to lifecycle fields. W-008 lists `cmd/gc/session_beads.go` as "close metadata, bead close, identity retirement, assigned-work release" but does not flag this cross-family write. **Required change**: add a note to W-008 or add a separate inventory entry that marks `closeFailedCreateBead` as a cross-family writer touching create/start and wake/hold/drain fields, with owner slices 3 and 5 noted.

- **[Minor] TR-007 acceptance criteria should be scoped to event-emitting slices.** The criteria require event payloads to carry "stable identifiers and enough context for subscribers to react idempotently" and "idempotency keys for subscriber scans" (`DESIGN.md:391`, `DESIGN.md:398-407`). For slices 2-4 that emit events, this is appropriate. But the criteria apply globally, and the per-event matrix includes a "Duplicate key" column that reads like a wire-format requirement for payloads that are currently thin (`events.SessionStranded` is `events.NoPayload{}`). The design's own recovery model says critical recovery scans durable facts, not event payloads. **Required change**: add a note to TR-007 stating that in-process event payloads for the initial implementation remain thin diagnostic accelerators (envelope plus minimal subject identity), and that idempotency/dedup keys are computed by the durable scan from bead facts. Move the per-event "Duplicate key" specification into a clearly-labeled "if and when a durable outbox is designed" section.

- **[Minor] Slices 3-7 contracts are front-loaded with detailed field enumerations before slices 1-2 prove the pattern.** `RuntimeStartIntent` (slice 3) and `SessionFactEvent` (no assigned slice) have required-field enumerations in the checkpoints table (`DESIGN.md:107-108`) and the event matrix. The document does not mark which contracts are frozen gates versus provisional scaffolding. Implementation-bead reviewers may treat the entire 886-line surface as equally binding. **Required change**: add a note that contracts for slices beyond the current implementation frontier are provisional and will be finalized when their slice starts, distinguishing them from the frozen slice-1/slice-2 gates.

**Missing evidence:**

- No definition of the per-operation policy input type, its fields, or which surfaces supply which fields on day one. Reviewers cannot verify whether slice-1 vocabulary is minimal without this.
- No per-kind population rules for the classifier result type (`TargetSelection`). Reviewers cannot determine whether `materializable`, `materialization_reason`, or `candidates[]` are populated for `direct-session-id` and `live-session-name`.
- No mapping from `live-session-name` / `live-alias` result kinds to the canonical projection vocabulary in `REQUIREMENTS.md`. `REQUIREMENTS.md:53-54` warns against parallel terms; the design uses "live" but the projection vocabulary uses "canonical" for open exact matches.
- `closeFailedCreateBead` is missing from the mutation landscape table as a cross-family writer.
- No statement of which planned slice emits the first new session event, leaving TR-007's payload design criteria unanchored.
- No shrink rule for the vocabulary checkpoints: the enumerations of "Required fields at introduction" read as commitments rather than upper bounds.

**Required changes:**

1. Define the per-operation policy input type. State whether the classifier receives all nine policy flags, a subset, or whether adapters apply policy to a raw classification result. List the minimum fields slice 1 requires and align `forbidden-kind`, `requires-materialization`, and `closed-not-allowed` with the decision.
2. Resolve the classifier/policy split: either the classifier returns only token identity and classification conflicts (removing `forbidden-kind`, `requires-materialization`, `closed-not-allowed` from negative kinds), or the classifier explicitly consumes a policy input parameter and the policy input fields are counted in slice-1 vocabulary.
3. Scope the operability contract per command cluster: each result type carries only the diagnostic fields its adopted surfaces consume today. The full field list is the eventual union, not a day-one requirement on every result.
4. Assign `SessionFactEvent` introduction to a specific slice and state that slice 1 emits no events and introduces no event vocabulary.
5. Add per-kind population rules to the classifier result spec, or state that the result is a tagged union with kind-specific payloads.
6. Note that `historical-alias` is not in the default resolution path (SESSION-ID-003) and is only produced when the surface policy explicitly enables `allow_historical_alias`.
7. Note that `path-alias` candidates are API-adapter-supplied, not classifier-discovered from a general store scan.
8. Add `closeFailedCreateBead` to the mutation landscape table as a cross-family writer with owner slices 3 and 5 noted.
9. Scope TR-007's payload and idempotency requirements to event-emitting slices and note that in-process payloads remain thin for the initial implementation. Move the per-event "Duplicate key" column to a "future durable outbox" section.
10. Mark slices 3-7 contracts (atomicity rows, operability mappings, `RuntimeStartIntent`/`SessionFactEvent` field lists) as provisional scaffolding to be finalized at slice start.
11. Add a shrink rule to vocabulary checkpoints: a slice may introduce fewer fields than enumerated when its first caller needs fewer, and the enumerations are upper bounds, not commitments.
12. Map `live-session-name` and `live-alias` to the canonical projection vocabulary, or rename them to `canonical-session-name` / `canonical-alias` to avoid parallel terms.

**Questions:**

- Does the per-operation policy input include just `allow_closed` and `materialize_named` (the two current code paths), or the full nine-flag set? Slice-1 vocabulary size depends on this answer.
- Should `closed-session-name` and `closed-alias` be universal classifier result kinds or adapter-dependent kinds that only appear when the caller supplies `allow_closed=true`? Current code has separate `ResolveSessionID` and `ResolveSessionIDAllowClosed` entry points.
- Is `historical-alias` intended to be produced by the classifier's default resolution path, or only when a surface explicitly enables it? `REQUIREMENTS.md` SESSION-ID-003 says normal resolution excludes historical aliases.
- Which planned slice is expected to emit the first new session event? Without that, TR-007's payload design criteria are unanchored for implementation.
- Are `live-session-name` and `live-alias` acceptable design terms, or should they use the canonical projection vocabulary from `REQUIREMENTS.md` (`canonical` for open exact matches)?
